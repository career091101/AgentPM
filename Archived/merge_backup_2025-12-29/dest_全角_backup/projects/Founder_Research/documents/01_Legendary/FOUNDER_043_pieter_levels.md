## 2. 創業ストーリー

### 2.1 課題発見（需要発見）

**PandaMixからTubelyticsへ**:
- オランダ・アムステルダム生まれ育ち、Conservatorium van Amsterdamで音楽制作を学ぶ
- テクノ音楽プロデューサー・DJとして活動しながら、YouTubeでPandaMixチャンネルを運営
- 複数のYouTubeチャンネルの解析を一元化したい課題を感じ、最初のスタートアップ「Tubelytics」を開発
- 1年かけて開発したが収益化できず失敗（重要な学び：小さく始めて素早くテストすべき）

**デジタルノマドとしての課題発見**:
- 2013年、全ての持ち物を売却しアジア各地（バリ、香港、バンコク、ソウル、チェンマイ、シンガポール）をバックパック1つで旅しながらリモートワーク
- デジタルノマドとして各都市の生活コスト、インターネット速度、安全性などの情報が散在しており、一元化された情報源がなかった
- 自身の経験から「どの都市がリモートワークに最適か」という課題を痛感

**うつ病との戦いと12 Startups in 12 Months**:
- 2014年、資金が底をつきかけ、うつ状態に陥る（アムステルダムの両親の家に戻る）
- Jennifer Dewaltの「180 Websites in 180 Days」に触発され「12 startups in 12 months」チャレンジを開始
- 長年悩んでいた2つの問題を克服するため：
  1. プロジェクトを完成させられない（完璧主義）
  2. 失敗を恐れてリリースできない
- 創造することがセラピーになると確信し、毎月1つのスタートアップをリリースする自己課題を設定

**需要検証方法（Googleスプレッドシート戦略）**:
- 2014年6月24日、Googleスプレッドシートをツイートで公開
- 「デジタルノマドに最適な都市のデータを集めよう」と呼びかけ
- **24時間以内に約100人がデータを入力** → 強い需要シグナル
- ユーザーは依頼以上の情報（安全性、LGBTQ+フレンドリー度、カフェ密度など）を自発的に追加
- Pieterの言葉：「スプレッドシートだけでTwitterが盛り上がっている。これで仮説が検証できた」

### 2.2 CPF検証（Customer Problem Fit）

**クラウドソースによる検証**:
- 実施数: 正式なインタビューではなく、Twitterでのクラウドソース検証
- 手法: Googleスプレッドシートによる公開データ収集
- 発見した課題の共通点: デジタルノマドは都市選択に必要な情報が断片的で困っている

**12 Startups in 12 Monthsの成果**:
- 12個のうちほとんどは失敗（メディオクリティ＝凡庸なもの）
- Nomad Listは4番目のプロジェクトとして誕生
- Product Huntでローンチし急成長
- 最初はGoogleスプレッドシートとして公開 → 1ヶ月でPHPサイト化

**3U検証**:
- **Unworkable**: 既存の都市情報サイトは旅行者向けで、長期滞在者の視点がない。複数サイトを横断して情報収集する必要があり非効率
- **Unavoidable**: リモートワーク人口の増加により需要拡大は必然
- **Urgent**: 次の移動先を決める際に即座に情報が必要

**支払い意思（WTP）**:
- 確認方法: MVP公開後すぐに有料会員制を導入
- 結果: 初期から有料登録者を獲得
- 価格推移: $5 → $10 → $25 → $50 → $65（ライフタイム会員制）
- ライフタイム課金を選んだ理由：Pieter自身がサブスク疲れを感じており、自動更新で使っていないサービスに課金され続けることへの嫌悪感

### 2.3 PSF検証（Problem Solution Fit）

**10倍優位性**:

| 軸 | 従来の解決策 | Pieterソリューション | 倍率 |
|---|------------|-----------------|------|
| 情報収集時間 | 複数サイトで個別調査（数時間） | ワンクリックで都市比較（数分） | 10倍 |
| 開発スピード | 数ヶ月〜数年 | 数日〜数週間 | 10倍+ |
| 運営コスト | チーム+オフィス | ソロ+ラップトップ | 100倍 |
| 意思決定速度 | 会議・承認プロセス | 即決・即実装 | 50倍 |
| 自動化率 | 50%以下 | 98%（2,000以上の自動化スクリプト） | 2倍 |

**MVP**:
- タイプ: Googleスプレッドシート → PHPサイト（1ヶ月で変換）
- 偶発的ローンチ: サーバー再起動時にnginx.confを誤ってアップロードし、意図せずサイトが公開
- 初期反応: **Product HuntとHacker Newsで同時に1位を獲得**
- 技術スタック: PHP + jQuery + SQLite（Product Huntのレイアウトを参考に）

**UVP（独自の価値提案）**:
- Nomad List: 「デジタルノマドのための都市データベース＆コミュニティ」
- Remote OK: 「世界最大のリモートワーク求人サイト」
- Photo AI: 「AIで自分の写真から様々なプロフェッショナルポートレートを生成」

**競合との差別化**:
- ユーザー参加型のデータ収集（クラウドソース）
- リアルタイム更新
- コミュニティ機能（同じ都市にいるノマド同士のマッチング）
- 完全ソロ運営による超高速意思決定
- Build in Publicによる透明性と信頼構築

## 3. ピボット/失敗経験

### 3.1 初期の失敗

- **Tubelytics（2013年頃）**: 自身のYouTube音楽チャンネル（PandaMix）の解析ツール。1年かけて開発したが収益化できず
- **12 Startups in 12 Months以前**: 複数のプロジェクトが未完成のまま頓挫。完成恐怖症と失敗恐怖症に悩む
- **70以上のプロジェクト**: 生涯で70以上のプロダクトを開発。大半は失敗したが、それぞれから学んだ

### 3.2 ピボット経験

**Avatar AI → Photo AI（成功したピボット）**:
- 元のアイデア: Avatar AI（漫画風アバター生成）
- 初期成果: 最初の週で約$150K収益（当時最大の成功）
- 課題発見: ユーザーがカートゥーンではなく**リアルなプロフェッショナル写真**を求めていることを発見
- ピボット後: Photo AI（リアルな写真生成）
- 結果: 18ヶ月で$0から$132K MRR達成、現在は収益の70%を占める最大のプロダクト
- 学び: ユーザーの実際の使用パターンを観察し、需要に応じて迅速にピボット

**Rebase（終了したプロジェクト）**:
- 概要: リモートワーカー向け移住支援サービス（Immigration as a Service）
- 対象国: ポルトガル、メキシコ
- ビジネスモデル: 申込み、支払い、詳細入力で移住手続きを代行
- 終了理由: ポルトガルNHRプログラムが2024年1月で終了。新規移民の処理時間が不足
- 運営期間: 約2年
- 学び: 外部要因（政策変更）への依存リスク

### 3.3 失敗から学んだ教訓

1. **小さく始めて素早く**: Tubelyticsの1年開発失敗から、「作る前にテスト」を学ぶ
2. **失敗を恐れない**: 「たくさん試して失敗し、改善していく」というメンタルモデル
3. **トラクションがなければ次へ**: 数ヶ月で反応がなければ次のアイデアに移る
4. **完璧より出荷**: 不完全でもリリースし、実際のフィードバックで改善

## 4. 技術スタック・開発哲学

### 4.1 シンプルなスタック

| カテゴリ | 技術 |
|---------|------|
| バックエンド | Vanilla PHP（1ファイル構成） |
| フロントエンド | jQuery（Reactなし） |
| データベース | SQLite |
| ホスティング | Linode VPS, Ubuntu, NGINX |
| コード管理 | ローカル管理 |
| 決済 | Stripe |
| AI（Photo AI用） | GPU servers |

### 4.2 なぜシンプルなのか

- 「最初に学んだから」（トレンドではなく生産性重視）
- 1つのスタックを深くマスター
- フレームワーク移行の時間コストを回避
- 各アプリは基本的に1つのindex.phpファイル
- 結果：**$3M ARRをPHP/jQuery/SQLiteで達成**

### 4.3 開発哲学

- 計画なし、ロードマップなし
- とにかくシップして適応
- 「準備ができる前にリリースすること」が競争優位
- 反復的・漸進的開発
- 技術の選択は「トレンディさ」より「生産性」

## 6. Twitter/SNS戦略（Build in Public）

### 6.1 フォロワー推移

| 時期 | フォロワー数 |
|------|------------|
| 2014年 | 数千人 |
| 2020年 | 約10万人 |
| 2024年 | 42万人+ |
| 2025年 | 50万人+ |

### 6.2 Build in Public戦略

**オープンスタートアップの先駆者**:
- 各プロジェクトの収益、トラフィック、その他メトリクスを公開ページで共有
- Stripeダッシュボードの収益数字をTwitterで定期的に共有
- 成功も失敗もリアルタイムで公開

**Build in Publicの6要素**:
1. **ジャーニーの共有**: 成功も失敗も全て公開
2. **戦略的透明性**: コードの一部やトリックを共有するが、全ては見せない
3. **ライブフィードバックループ**: Twitterをリアルタイム顧客サポートとして活用
4. **パブリック問題解決**: Stripeのバグ → CTOに公開でメンション → 解決 + 無料PR
5. **コミュニティ構築**: ユーザーの成功をRT、インディーメーカーツールを紹介
6. **ユーモアと親しみやすさ**: ジョークツイートでブランドを人間化

**透明性の効果**:
- 信頼構築：実際の収益と苦労を見せることで、洗練されたマーケティングより信頼される
- アカウンタビリティ：公開コミットメントで目標達成へのプレッシャー
- オーガニックマーケティング：本物のストーリーテリングが従来の広告より高いエンゲージメント
- コミュニティ構築：フォロワーがインサイダー感を得て、強いつながりを形成

## 7. 成功要因分析

### 7.1 主要成功要因

1. **極限のミニマリズム**: 従業員ゼロ、シンプルなスタック、1つのファイルで動くアプリ

4. **透明性（Build in Public）**: 収益・失敗・学びを全て公開
5. **ニッチへのフォーカス**: デジタルノマドという明確なターゲット
6. **自分の課題を解決**: デジタルノマドとしての実体験から生まれたプロダクト

- 2014年: デジタルノマドムーブメントの黎明期
- 2020年: コロナ禍でリモートワークが爆発的に普及

### 7.3 差別化要因

- VCを受けない → 完全なコントロール
- 従業員を雇わない → 意思決定の速さ
- シンプルな技術 → メンテナンスコスト最小
- Build in Public → 無料マーケティング
- 40カ国以上、150以上の都市を旅した経験 → 本物の実践者視点
