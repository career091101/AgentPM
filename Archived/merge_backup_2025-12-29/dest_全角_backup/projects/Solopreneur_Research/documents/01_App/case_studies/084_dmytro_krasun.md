# Case Study: Dmytro Krasun (ScreenshotOne)

**基本情報**

- **人物名**: Dmytro Krasun
- **代表プロダクト**: ScreenshotOne
- **収益**: 月商 約 250 万円 (推定 $16.6k/mo, ARR $200k)
- **X (Twitter)**: https://x.com/dmytrokrasun
- **関連 URL**: https://screenshotone.com/
- **調査日**: 2025-12-27

---

## 1. 概要（3 行まとめ）

- **Boring SaaS**: 「Web サイトのスクリーンショットを API で撮るだけ」という、極めて地味で競合も多い（コモディティ化した）市場にあえて後発で参入。
- **SEO & Reliability**: 派手なマーケティングではなく、開発者が直面する「フォント崩れ」「Cookie バナーの除去」などの細かい技術的課題をどこよりも丁寧に解決し、信頼を積み上げた。
- **Steady Growth**: 派手なバズりはないが、ブログ記事（技術解説）と SEO、そして既存顧客の口コミで、MRR が毎月確実に積み上がる「複利型」の成長を実現。

---

## 2. 収益モデルと推移

- **収益モデル**: B2B API サブスクリプション（従量課金）。
- **推移**:
  - **Launch**: 2022 年 5 月、会社員を辞めて独立。
  - **$1k MRR**: 2023 年 3 月（約 1 年後）。
  - **$10k MRR**: 2024 年後半。
  - **Now**: 2025 年 4 月時点で ARR $200k（年商 3000 万円）を突破。

---

## 3. プロダクト開発の背景

- **「退屈なビジネス（Boring Business）」**: 彼は「不確実なイノベーション」よりも「確実な需要」を選んだ。スクリーンショット API は既に市場があるが、既存ツールは遅かったり、サポートが悪かったりした。
- **Developer First**: 自分が開発者だからこそ分かる「使いやすい API ドキュメント」「SDK の充実」にこだわった。

---

## 4. 成長戦略（Marketing Strategy）

### Programmatic SEO

- **Comparison Pages**: 「Urlbox alternative」「ScreenshotAPI vs ScreenshotOne」などの比較ページを大量に作成し、顕在層（乗り換え検討客）を拾った。
- **Use Case Pages**: 「Google Maps Screenshot API」「Twitter Screenshot API」など、具体的な用途ごとの LP を用意。

### Building in Public

- **Transparent**: 売上や解約率（Churn）だけでなく、サーバーの技術的な障害やその解決策まで X で赤裸々に公開し、エンジニアからの信頼を獲得した。

---

## 5. 成功の要因（Success Factors）

1. **Reliability > Features**: インフラ系 API に求められるのは「多機能」ではなく「落ちないこと」「正確であること」。そこにリソースを集中した。
2. **Customer Support**: Dmytro 本人が即レスでサポート対応することで、大手の競合から顧客を奪った。

---

## 6. 失敗と教訓（Failure & Lessons）

- **焦りは禁物**: 最初の 1 年は成長が遅く、不安な時期もあった。しかし、SaaS は「J カーブ」ではなく「階段状」に伸びることを理解し、淡々と機能改善を続けたことが実を結んだ。

---

## 7. 日本市場への適用可能性

- **適用性スコア**: ◯ (4.0/5.0)
- **理由**: 日本語フォント（Web フォント）の描画崩れは、海外製スクショ API の最大の弱点。ここを完璧にカバーした「日本特化スクショ API」は地味に需要がある。
- **具体的なアイデア**:
  - 「魚拓」サービスの API 版（法的証拠用）。
  - 「スマホ表示」特化の LP 検証用スクショ API。

---

## 8. 次のアクション（Action Plan）

1. **Long Tail キーワード**: 「PDF 変換 API」「QR コード 生成 API」など、地味だが企業が必ず使う「ユーティリティ API」を開発する。
