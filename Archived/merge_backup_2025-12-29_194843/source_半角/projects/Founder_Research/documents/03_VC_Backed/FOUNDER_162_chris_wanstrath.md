## 2. 創業ストーリー

### 2.1 課題発見（需要発見）

**着想源**:
- **個人的課題**: Chris WanstrathとP.J. HyettはCNETで働きながらRuby on Railsのオープンソースプロジェクトに貢献していたが、既存のバージョン管理ツールに不満
  - SourceForge: プロジェクト公開には優れているが、コラボレーション機能が貧弱
  - Git自体: 強力だが、習得曲線が険しく、ホスティング・共有が困難
  - プルリクエストの統合: 信頼された開発者に変更を依頼し、説得する必要があり、煩雑
- **市場観察**: 2007年、Gitが急速に普及（Linuxカーネル開発で採用）。しかし、使いやすいホスティングサービスが存在しない
- **競合の弱点**:
  - SourceForge: プロジェクト公開には良いが、コラボレーションに最適化されていない
  - Bitbucket（2008年ローンチ）: Mercurialのみサポート（Gitは未対応）
  - GoogleCode、CodePlex: 企業主導で開発者体験が劣る
- **ソーシャルコーディングの着想**: Tom Preston-Werner（共同創業者）が「Gitにソーシャル機能を加えたホスティングサイト」を着想。2007年10月のRuby on Rails開発者ミートアップでChrisに提案

**需要検証方法**:
- **プライベートベータ（2008年1月）**: Bay Areaのスタートアップ・友人開発者にメール送信
  - 反応: 「即座に強く肯定的」（Chris談）
  - Ruby on Railsコミュニティが初期アダプター（創業者がRuby開発者のため口コミ拡散）
- **パブリックローンチ（2008年4月）**: 2ヶ月で2,000ユーザー獲得
- **初期の反応**: 2009年7月（約1年）で10万ユーザー突破 → 明確な需要を確認

### 2.2 CPF検証（Customer Problem Fit）

**インタビュー/顧客検証**:
- 実施数: 25件（推定）
  - プライベートベータ期間中、Bay Areaの友人・スタートアップ開発者にヒアリング
  - 非公式インタビュー: Ruby on Railsミートアップ、カンファレンスでの対話
  - 自己課題: Chrisと共同創業者自身が「理想のツール」を求めていた
- 手法: 定性インタビュー、プロトタイプフィードバック、ベータ利用観察
- 発見した課題の共通点:
  1. **コラボレーション困難**: 既存ツールは「プロジェクト公開」に最適化、「共同開発」には不向き
  2. **Gitの学習曲線**: Gitは強力だが初心者に難解、GUIツール不足
  3. **ソーシャル機能欠如**: コードレビュー、フォーク、マージの管理が煩雑
  4. **分散ワークフロー未対応**: 従来ツール（SVN等）は中央集権型、Gitの分散型に最適化されていない

**3U検証**:
- Unworkable（現状では解決不可能）:
  - SourceForge等既存ツール: コラボレーション機能が貧弱、UIが古い
  - Git自体: コマンドラインのみ、学習コスト高、ホスティング・共有が困難
  - メール添付でのパッチ送信: 非効率、追跡困難、スケールしない
- Unavoidable（避けられない）:
  - オープンソース開発の増加（2000年代後半）: 避けられないトレンド
  - 複数開発者でのコラボレーション: ソフトウェア開発の本質
  - バージョン管理: 全ての開発プロジェクトで必須
- Urgent（緊急性が高い）:
  - 中程度（スコア: 7/10）
  - 理由: 開発者は既存ツールで「なんとか」やれているが、非効率を感じている
  - Gitの普及加速（Linuxカーネル採用）→ 早期参入が重要

**支払い意思（WTP）**:
- 確認方法: フリーミアムモデル（無料公開リポジトリ + 有料プライベートリポジトリ）
- 結果:
  - 2012年時点（Series A前）で既に黒字化（ローンチから4年）
  - 収益成長率: 前年比300%（2012年）
  - Series A（$100M）時点で既に「VC資金不要」レベルの収益性
  - Microsoft買収時（2018年）の年間収益: 約$300M（推定）
  - 2025年時点の年間収益: 約$2B（推定）
- WTP確認: Day 1から「プライベートリポジトリは有料」モデル → 即座に支払いユーザー獲得

### 2.3 PSF検証（Problem Solution Fit）

**10倍優位性**:

| 軸 | 従来の解決策 | GitHubソリューション | 倍率 |
|---|------------|-----------------|------|
| コラボレーション容易性 | SourceForge: メーリングリスト・パッチ添付 | プルリクエスト・フォーク・イシュー統合UI | 50x |
| 使いやすさ | Git CLI: 習得に数週間-数ヶ月 | WebベースUI + Git連携（初心者でも即利用可能）| 20x |
| ソーシャル機能 | なし（SourceForge等はプロジェクト公開のみ）| フォロー、スター、アクティビティフィード | 100x（ゼロ→あり）|
| 発見性 | 検索エンジン頼み | GitHub内検索、トレンドリポジトリ、レコメンド | 10x |
| 統合性 | 個別ツール（バグトラッカー、Wiki等）を別途管理 | イシュー、Wiki、プロジェクト管理を統合 | 15x |

**MVP**:
- タイプ: Prototype（フル機能プロトタイプ）
  - プライベートベータ（2008年1月）: Git hosting + Web UI + フォーク機能
  - パブリックローンチ（2008年4月）: 上記 + プルリクエスト + イシュートラッカー
  - 技術スタック: Ruby on Rails（ChrisとTomの得意分野）
- 初期反応:
  - 2008年2月: Ruby on RailsプロジェクトがGitHubに移行 → Rubyコミュニティ全体が追随
  - 2008年4月（パブリックローンチ）: 2,000ユーザー（2ヶ月）
  - 2009年7月: 10万ユーザー（14ヶ月）
  - 2010年: 100万ユーザー、SourceForge・Google Codeを上回る
- CVR: 15%（推定、無料→有料転換率）
  - ローンチ初日から収益化（フリーミアムモデル）
  - 2012年時点で黒字、年率300%成長

**UVP（独自の価値提案）**:
- 「ソーシャルコーディング」: コードを「公開」するだけでなく、「協力して構築」するプラットフォーム
- Chris Wanstrath: 「最高のプログラマーは他者と執拗に反復する。一緒に物を作る。これがソフトウェア開発の未来」
- 「開発者ファースト」: PM、会計士、CEOではなく、開発者自身が開発者のために作ったツール
- 「Gitを簡単に」: 強力だが難解なGitを、誰でも使えるようにする

**競合との差別化**:
1. **開発者体験の徹底追求**: SourceForge等は「プロジェクトホスティング」、GitHubは「コラボレーション」に最適化
2. **ソーシャル機能**: フォロー、スター、アクティビティフィードでコミュニティ形成
3. **Git最適化**: Gitの分散型ワークフローに完璧に対応（Bitbucketは当初Mercurialのみ）
4. **フリーミアム**: 公開リポジトリ無料 → オープンソースコミュニティが大量流入
5. **開発者主導**: 創業者全員が現役開発者 → プロダクトの「味」が良い
6. **コミュニティ重視**: オープンソースプロジェクトを優遇 → ネットワーク効果

## 3. ピボット/失敗経験

### 3.1 初期の失敗

- **FamSpam（2007年初頭）**:
  - ChrisとP.J. Hyettが2007年初頭にCNETを退職し、コンサルティング会社Err Freeを設立
  - 同時に、一般消費者向けファイル共有サイト「FamSpam」をローンチ
  - 狙い: 家族・グループが写真やファイルをオンラインで共有
  - 結果: 失敗（トラクション不足、差別化困難）
  - 学び: 「自分が解決したい課題」に注力すべき。一般消費者向けは競合多数で差別化困難

### 3.2 ピボット（該当する場合）

- **元のアイデア**: FamSpam（一般消費者向けファイル共有）
- **ピボット後**: GitHub（開発者向けGitホスティング・コラボレーション）
- **きっかけ**:
  - FamSpam失敗（2007年半ば）
  - 2007年10月、Tom Preston-WernerがSan FranciscoのスポーツバーでChrisに「Gitホスティングサイト」のアイデアを提案
  - ChrisとTomは週末を使って3ヶ月間コーディング（2007年10月-2008年1月）
  - 2008年1月、プライベートベータローンチ → 強い肯定的反応
- **学び**:
  1. **「自分が欲しいものを作る」**: FamSpamは「市場機会」狙い、GitHubは「自分の課題解決」
  2. **ニッチ市場の深掘り**: 一般消費者（広い・浅い）→ 開発者（狭い・深い）
  3. **コミュニティ活用**: Ruby on Railsコミュニティの一員として、口コミで拡散
  4. **段階的検証**: 週末プロジェクト（リスク低）→ プライベートベータ → パブリックローンチ
  5. **ブートストラップの利点**: VC資金なし → 自由な意思決定、開発者体験最優先

## 4. 成長戦略

### 4.1 初期トラクション獲得

- **Ruby on Railsコミュニティ（2008年2月-4月）**:
  - 創業者全員がRuby開発者 → 同コミュニティで信頼獲得
  - 2008年2月、Ruby on RailsプロジェクトがGitHubに移行
  - Rubyコミュニティ全体が急速に追随 → 初期ユーザーベース形成
- **プライベートベータ（2008年1月-3月）**:
  - Bay Areaの友人・スタートアップ開発者にメール
  - 口コミで拡散（開発者コミュニティは狭い世界）
- **フリーミアムモデル**:
  - 公開リポジトリ無料 → オープンソースプロジェクトが大量流入
  - 2009年7月: 10万ユーザー
  - 2010年: 100万ユーザー、SourceForge・Google Codeを上回る

  - VC資金なし（2012年まで） → 黒字化を早期達成

### 4.2 フライホイール

   - 開発者が個人でGitHub利用 → 企業にも持ち込む
   - プライベートリポジトリ（有料）→ 収益化
4. **エコシステム拡大**
   - サードパーティツール連携（CI/CD、プロジェクト管理）

5. **データ・機能改善**
   - ユーザー増 → データ蓄積 → 機能改善（検索、レコメンド等）
   - トレンドリポジトリ、Explore機能 → 発見性向上
6. **ブランド確立**
   - 「ソーシャルコーディング」= GitHub → 業界標準化

### 4.3 スケール戦略

## 6. 成功要因分析

### 6.1 主要成功要因

1. **「自分が欲しいものを作る」哲学**:
   - FamSpam（市場機会狙い）失敗 → GitHub（自己課題解決）成功
   - 創業者全員が現役開発者 → プロダクトの「味」が良い
   - Chris: 「PM、会計士、CEOではなく、開発者自身が開発者のために作った唯一のツール」
2. **開発者体験の徹底追求**:
   - 使いやすさ、デザイン、パフォーマンス全てに妥協なし
   - 「コラボレーション」に最適化（SourceForgeは「公開」に最適化）
   - ソーシャル機能（フォロー、スター）でコミュニティ形成
3. **フリーミアムモデルの完璧な実行**:
   - 公開リポジトリ無料 → オープンソースコミュニティが大量流入
   - Day 1から収益化（プライベートリポジトリ有料）
   - ネットワーク効果 → プラットフォーム価値向上
4. **ブートストラップ戦略**:
   - 2012年までVC資金ゼロ → 黒字化達成
   - 自由な意思決定、開発者体験最優先
   - VCの短期収益圧力を回避
5. **コミュニティ重視**:
   - Ruby on Railsコミュニティから開始 → 口コミで拡散
   - オープンソースプロジェクトを優遇 → エコシステム形成
   - 開発者カンファレンスでのプレゼンス
6. **タイミング**:
   - Git普及期（2007-2008年）に参入
   - SourceForge等既存ツールの限界が明確
   - オープンソース・分散開発の台頭

### 6.2 タイミング要因

- **Git普及期**（2007-2008年）:
  - Linuxカーネル開発でGit採用 → 開発者コミュニティに拡散
  - SVN等中央集権型から分散型への移行期
- **Web 2.0・ソーシャル時代**（2008年前後）:
  - Facebook、Twitter等ソーシャルプラットフォーム台頭
  - 「ソーシャルコーディング」というコンセプトが時代にフィット
- **オープンソース活性化**（2000年代後半）:
  - Linux、Apache、MySQL等の成功
  - 企業もオープンソース活用を加速
- **クラウド時代**（2008年～）:
  - AWS登場（2006年）→ インフラコスト低下
  - SaaS型ツールが一般化

### 6.3 差別化要因

- **開発者ファースト**:
  - 創業者全員が現役開発者（PM主導ではない）
  - 開発者体験への妥協なし
- **ソーシャル機能**:
  - フォロー、スター、アクティビティフィード
  - コード = ソーシャルオブジェクト
- **Git最適化**:
  - 分散型ワークフローに完璧対応
  - Bitbucket（当初Mercurialのみ）に先行
- **フリーミアム**:
  - オープンソース優遇 → コミュニティ形成
  - エンタープライズ収益化
- **エコシステム**:
  - API公開 → サードパーティツール連携
  - GitHub Marketplace
- **ブランディング**:
  - 「ソーシャルコーディング」= GitHub
  - 業界標準としての地位確立

## 8. orchestrate-phase1への示唆

### 8.1 需要発見（/discover-demand）

- **「自分の課題」から始める**:
  - FamSpam（市場機会）失敗 → GitHub（自己課題）成功
  - Chris: 「自分が毎日使いたいツール」を作った
  - → 需要発見の最良の方法は「自分が顧客」になること
- **コミュニティ観察**:
  - Ruby on Railsミートアップ、カンファレンスで開発者の課題を直接聞く
  - オープンソースプロジェクトへの貢献で「現場の痛み」を体験
- **既存ツールの限界を特定**:
  - SourceForge: 「公開」はできるが「コラボレーション」が弱い
  - Git: 強力だが難解
  - → 既存ツールの「穴」を埋める

### 8.2 CPF検証（/validate-cpf）

- **プライベートベータの威力**:
  - 3ヶ月の週末開発 → プライベートベータ → 強い肯定的反応
  - 友人・コミュニティの信頼を活用（Bay Areaスタートアップ、Ruby開発者）
  - → 低コストで高速検証
- **3U検証**:
  - Unworkable: 既存ツール（SourceForge等）ではコラボレーション困難
  - Unavoidable: バージョン管理は全開発プロジェクトで必須
  - Urgent: 中程度（7/10）。既存ツールで「なんとか」やれているが非効率
- **WTP確認**:
  - Day 1から「プライベートリポジトリ有料」→ 即座に支払いユーザー獲得
  - 4年で黒字化 → WTP極めて高い

### 8.3 PSF検証（/validate-10x）

- **10倍優位性の明確化**:
  - コラボレーション: 50x（メール添付パッチ → プルリクエスト）
  - ソーシャル機能: 100x（ゼロ → あり）
  - 使いやすさ: 20x（Git CLI → Web UI）
- **MVP戦略**:
  - Ruby on Railsで高速開発（創業者の得意分野）
  - プライベートベータ（3ヶ月）→ パブリックローンチ（フル機能）
  - → 段階的リスク管理
- **「開発者による開発者のため」**:
  - 創業者全員が現役開発者 → プロダクトの「味」が良い
  - PM、マーケター主導ではない → 本質的課題に集中

## 9. 事業アイデア候補

この事例から着想を得られる日本向けビジネスアイデア:

1. **デザイナー向け「GitHub」（Figma競合）**:
   - 課題: Figmaは高い、日本企業はセキュリティ懸念でクラウド利用躊躇
   - ソリューション: デザインファイルのバージョン管理 + コラボレーション + 日本企業向けオンプレミス版
   - 10倍軸: コラボレーション（メール添付 → リアルタイム共同編集）、コスト（Figmaの1/3）
   - 差別化: 日本企業のセキュリティ要件対応、デザインシステム管理特化
2. **データサイエンティスト向けコラボレーションプラットフォーム**:
   - 課題: Jupyter Notebook等はバージョン管理・共有が困難
   - ソリューション: データ分析コードのGitHub（データセット + コード + 結果の統合管理）
   - 10倍軸: コラボレーション（個人作業 → チーム共有）、再現性（実験結果の追跡）
   - 差別化: データセットのバージョン管理、実験結果の可視化、社内データガバナンス対応
3. **法務・契約書のバージョン管理プラットフォーム**:
   - 課題: Word + メール添付で契約書をやり取り → バージョン混乱、変更履歴不明
   - ソリューション: 契約書のGitHub（変更履歴、コメント、承認フロー統合）
   - 10倍軸: 変更追跡（Word比較機能 → Git差分）、コラボレーション（メール → プラットフォーム）
   - 差別化: 法務特化UI、電子署名統合、日本の商習慣対応
