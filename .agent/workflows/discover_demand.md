# 需要発見ワークフロー

「困りごとの生ログ」を起点に、新規ビジネス / SaaS / AIツールの有望なアイデアを発見するワークフロー。

## 入力・出力

| 項目 | 内容 |
|------|------|
| **入力** | 探索分野キーワード（オプション） |
| **出力** | `documents/1_initiating/demand_discovery.md` |
| **次のWF** | `/discover_idea` または直接 `/create_lean_canvas` |

> [!TIP]
> **生ログ重視**: 市場調査レポートや推測ではなく、実際のユーザー発言を最優先します。

## 基本スタンス

- 市場調査レポートや推測は使わない
- 実際のユーザー発言（生ログ）を最優先する
- 「すでに困っている人がいるか」を唯一の判断基準とする
- 起業家視点でお金の匂いに敏感であること
- 不確実性がある場合は必ず明示する

## 対象言語・分析方針

- **日本語・英語の両方を対象**
- 言語圏は分けて分析する
- 日本語圏と英語圏で以下の違いを意識:
  - 困り方の表現
  - 支払い意識
  - コミュニティ文化

## 主な情報源

以下を優先的に用いる（`search_web` 経由でアクセス）：

| プラットフォーム | 日本語圏 | 英語圏 | 用途 |
|----------------|:-------:|:------:|------|
| Reddit | - | ✅ | コミュニティ議論・不満 |
| Yahoo!知恵袋 | ✅ | - | 質問・困りごと |
| Stack Overflow / teratail | ✅ | ✅ | 技術課題 |
| GitHub Issues / Discussions | ✅ | ✅ | 開発者課題 |
| G2 / Capterra | - | ✅ | SaaSレビュー・不満 |
| Xユーザー投稿 | ✅ | ✅ | リアルタイム不満 |

## STEP 1: 検索クエリ設計【自動実行】

### 使用ツール
`search_web`

### 実行手順
// turbo
探索分野に基づいて、以下のパターンでクエリを設計：

**日本語クエリ（Yahoo!知恵袋・X向け）**:
```
"{分野} おすすめ ツール 比較"
"{分野} 自動化 できない 困っている"
"{分野} 時間かかる 面倒"
"{分野} お金払ってでも 解決したい"
"site:chiebukuro.yahoo.co.jp {分野} 困っている"
```

**英語クエリ（Reddit・Stack Overflow向け）**:
```
"site:reddit.com {topic} frustrating alternative tool"
"site:reddit.com {topic} automation wish there was"
"site:stackoverflow.com {topic} no good solution"
"{topic} software recommendations comparison"
"best {topic} tool 2024 reddit"
```

### 検索実行
- 各クエリで3-5件の結果を取得
- 合計10-15件のソースを収集

### エラー対応
| エラー | 対応 |
|-------|------|
| 検索結果0件 | クエリを言い換えて再検索 |
| API制限 | 30秒待機後リトライ（最大3回） |
| 関連性低い結果 | 別の情報源に切り替え |

## STEP 2: 生ログ収集（日本語圏）【自動実行】

### 使用ツール
`search_web`

### 実行手順
// turbo
日本語圏の情報源から「困りごと」を収集：

```markdown
## 日本語圏 生ログ収集

### 収集対象の発言パターン
- 「〜で困っている」「〜がうまくいかない」
- 「おすすめのツールありますか」
- 「〜を自動化したい」
- 「〜に時間がかかって仕方ない」
- 「〜の代わりになるサービスは？」

### 収集テンプレート
#### 生ログ #JP-X
- **プラットフォーム**: [Yahoo!知恵袋/X/teratail等]
- **URL**: [検索で見つけたURL]
- **原文引用**:
> "[ユーザー発言をそのまま引用]"
- **回答・議論**: [回答があれば要約]
- **いいね/回答数**: [エンゲージメント指標]
```

### 抽出すべき発言の基準
- ✅ 同様の悩みが複数回出現している
- ✅ 「おすすめ」「比較」「ツール」「代替」「自動化」などの語が含まれる
- ✅ 解決に時間・労力・お金を払ってもよいニュアンスがある
- ✅ 回答が長文、または議論が続いている

## STEP 4: 「困りごと」抽出・構造化【自動実行】

### 実行手順
// turbo
収集した生ログから需要候補を構造化：

```markdown
## 需要候補 #X: [タイトル]

### 【1】生ログ引用（最重要）

**日本語圏**:
> "[日本語の原文をそのまま引用]"
> 出典: [プラットフォーム名、可能ならURL]

**英語圏**:
> "[英語の原文をそのまま引用]"
> Source: [Platform, URL if available]

### 【2】構造化要約
| 項目 | 内容 |
|------|------|
| **誰が** | [具体的なターゲット像] |
| **何に困っているか** | [課題の内容] |
| **なぜ困っているか** | [背景・原因] |
| **今どう対処しているか** | [現状の解決策、あれば] |
```

### 抽出基準
- 最低5件の需要候補を抽出
- 同様の声が複数ある場合は1つにまとめ、件数を記録
- デジタルで解決可能なもののみ対象

## STEP 5: 需要スコアリング【自動実行・必須】

### 実行手順
// turbo
各需要候補について、以下の4軸で5段階評価：

```markdown
### 【6】需要スコア

| 評価項目 | スコア | 根拠 |
|---------|:------:|------|
| **切実度** | X/5 | [どれだけ困っているか] |
| **頻度** | X/5 | [同様の声がどれだけあるか] |
| **支払い匂い** | X/5 | [お金を払う可能性] |
| **未解決度** | X/5 | [既存解決策の弱さ] |
| **合計** | **XX/20** | |

**総合判定**: ✅有望 / ⚠️検討余地 / ❌見送り
```

### スコアリング基準

| 項目 | 5点 | 3点 | 1点 |
|------|-----|-----|-----|
| 切実度 | 「今すぐ解決必須」「困り果てている」 | 「できれば解決したい」 | 「別にいい」「あれば便利」 |
| 頻度 | 同様の声10件以上 | 3-9件 | 1-2件 |
| 支払い匂い | 「お金払ってでも」発言あり | 時間・労力コスト言及 | 無料希望が明確 |
| 未解決度 | 既存策なし or 強い不満多数 | 既存策あるが不十分 | 十分解決済み |

### 判定基準
| 合計スコア | 判定 | アクション |
|:---------:|------|-----------|
| 16-20 | ✅ 有望 | 優先的にアイデア化 |
| 12-15 | ⚠️ 検討余地 | 追加調査推奨 |
| 1-11 | ❌ 見送り | 対象外 |

## STEP 7: マネタイズ仮説策定【自動実行】

### 実行手順
// turbo
各解決アイデアに対してマネタイズ仮説を策定：

```markdown
### 【5】マネタイズ仮説

| 項目 | 内容 |
|------|------|
| **誰が払うか** | [BtoB/BtoC、具体的なペルソナ] |
| **何に対して払うか** | [価値提案、解決される課題] |
| **課金形態** | [月額/従量/買い切り/フリーミアム] |
| **想定価格帯** | [月額○円 / 年額○円] |
| **根拠** | [類似サービスの価格帯、生ログからの示唆] |
```

### マネタイズ判断のポイント
- 生ログで「お金を払ってでも」という発言があるか
- 既存の類似サービスの価格帯
- BtoBの場合は経費計上可能か

## STEP 8: 不確実性・注意点の明示【自動実行】

### 実行手順
// turbo
各需要候補に対して、正直にリスクと不確実性を明示：

```markdown
### 【7】不確実性・注意点

**サンプル品質**:
- [ ] サンプル数が少ない（信頼性低）
- [ ] 特定業界に偏っている
- [ ] 一時的トレンドの可能性

**市場リスク**:
- [ ] 大手参入リスク
- [ ] 規制リスク
- [ ] 技術的実現ハードル

**検証推奨事項**:
- [追加で確認すべきこと]
```

### トーン
- 冷静だが、起業家視点
- 無駄に煽らない
- 「これは金になりそう / なりにくい」をはっきり言う
- ただし断定せず、根拠を示す
