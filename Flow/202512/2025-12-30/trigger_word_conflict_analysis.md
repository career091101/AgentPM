# トリガーワード競合分析レポート

**作成日**: 2025-12-30
**対象**: 5フォルダ全体のトリガーワード体系

---

## エグゼクティブサマリー

### 主要発見事項

1. **3つの異なるトリガー体系が混在**
   - **PMBOKトリガー** (.cursor master_rules): フェーズベース、日本語パターンマッチング
   - **Ultra Lightトリガー** (.agent, .claude, .cursor UL): 3-4コマンド固定
   - **スタートアップワークフロートリガー** (.agent workflows): 45種類の専門ワークフロー

2. **競合は検出されず**
   - 各トリガーは明確に分離されている
   - 同じワードが異なるアクションを引き起こす事例なし

3. **問題点**
   - トリガーワードが分散(統一レジストリなし)
   - ユーザーがどのトリガーが使えるか把握困難
   - ツール間でトリガーのサポート範囲が異なる

## 4. 問題点と課題

### 4.1 トリガーワード発見性の問題

**現状**:
- トリガーワードが5フォルダに分散
- ユーザーが「どのトリガーが使えるか」を把握できない
- ツールごとにサポート範囲が異なる

**影響**:
- ユーザーは試行錯誤で探すしかない
- せっかくのワークフローが使われない可能性

### 4.2 トリガー命名規則の不統一

**パターン1**: 日本語のみ
- 例: タスクメモ、次の一手、これやる

**パターン2**: 日本語 + 英語併記
- 例: プロジェクト憲章 | プロジェクトチャーター
- 例: Development | 開発

**パターン3**: 専門用語
- 例: CPF診断、PMF診断、10倍検証

**問題**:
- 一貫性なし
- ユーザーが覚えにくい

### 4.3 ツール間のトリガー同期問題

**例**: 「今日のタスク」
- .antigravity: 簡易実装
- .cursor (従来版): 詳細実装
- .claude: 表示のみ
- .agent: 未実装

**問題**:
- 同じトリガーでもツールによって動作が異なる
- ユーザーが混乱する可能性

### 4.4 スタートアップワークフローの孤立

**現状**:
- .agent/workflows/ に45種類の専門ワークフロー
- 他ツール(.claude, .cursor, .codex)では一部のみサポート

**問題**:
- Antigravity専用機能になっている
- 他ツールユーザーが使えない

## 5. 推奨事項

### 5.1 優先度高: 統一トリガーワードレジストリの作成

#### 目的
全トリガーワードを一元管理し、ユーザーが検索可能にする

#### 実装案

**場所**: `docs/ai/TRIGGER_WORDS_REFERENCE.md` (新規作成)

**構成**:
```markdown
# トリガーワード総合リファレンス

## 1. タスク管理
### Ultra Light版
- **タスクメモ**: inbox追記
  - サポート: .agent, .claude, .cursor

- **次の一手**: A/B/Cパターン生成
  - サポート: .agent, .claude, .cursor

...

## 2. PMBOKフェーズ
### 立ち上げ(Initiating)
- **プロジェクト憲章**: プロジェクト憲章作成
  - サポート: .cursor, .antigravity, .claude

...

## 3. スタートアップワークフロー
### 検証系
- **10倍検証**: 10xポテンシャル検証
  - サポート: .agent, .claude (skill)

- **CPF診断**: カスタマープロブレムフィット診断
  - サポート: .agent

...
```

**メリット**:
- ユーザーが全トリガーを一覧できる
- ツール別サポート状況が明確
- 新規トリガー追加時の重複チェックが容易

### 5.2 優先度中: トリガー命名規則の標準化

#### 提案: 3種類の命名規則を定義

**Type A: 簡易コマンド型** (タスク管理など)
- 形式: ひらがな/カタカナ2-5文字
- 例: タスクメモ、次の一手、これやる

**Type B: PMBOK型** (文書作成など)
- 形式: [名詞] + [動詞(省略可)]
- 例: プロジェクト憲章、WBS作成、ペルソナ作成

**Type C: ワークフロー型** (専門プロセス)
- 形式: [専門用語] + [処理]
- 例: CPF診断、10倍検証、Phase1開始

**ルール**:
- 英語併記は () 内に記載: プロジェクト憲章 (Project Charter)
- 省略形は避ける: ✗ "PRD" → ✓ "PRD作成" or "要求仕様書作成"

### 5.3 優先度中: ツール間トリガー同期

#### 提案: 共通トリガーは統一実装にリダイレクト

**例**: 「今日のタスク」
- .antigravity: 削除 → .cursor従来版を参照
- .cursor (Ultra Light): 削除 → .cursor従来版を参照
- .claude: `.cursor/07_task_management.mdc` を参照

**メリット**:
- 実装が1箇所に集約
- ツール間で動作が統一される
- メンテナンスが容易

### 5.4 優先度低: スタートアップワークフローの展開

#### Option A: .claude/skills/ に移植
- .agent/workflows/ の45ワークフローを.claude/skills/に移植
- Claude Code でも使えるようにする

#### Option B: 参照方式
- .claude/commands/ から .agent/workflows/ を参照
- 既存実装を再利用

#### Option C: 現状維持
- Antigravity専用機能として残す
- ドキュメントで明記

**推奨**: Option B (参照方式) - 重複を避けつつ機能共有

## 6. 実装ロードマップ

### フェーズ1: ドキュメント整備 (2-3時間)

**目標**: ユーザーがトリガーを発見できるようにする

   - ツール別サポート状況を記載

2. ✅ **各フォルダにリンク追加**
   - README.md に「トリガーワード一覧は docs/ai/TRIGGER_WORDS_REFERENCE.md を参照」と記載

### フェーズ2: 命名規則の適用 (3-4時間)

**目標**: 新規トリガーの命名を統一

1. ✅ **命名規則ドキュメント作成**
   - `docs/ai/TRIGGER_NAMING_CONVENTION.md`

2. ⏳ **既存トリガーの見直し**
   - 規則に従っていないトリガーをリストアップ
   - 変更優先度を判定(低/中/高)

3. ⏳ **高優先度トリガーのリネーム**
   - ユーザー影響が大きいものから順次変更

### フェーズ3: 実装の統合 (4-6時間)

**目標**: 重複実装を解消

1. ✅ **タスク管理トリガーの統合**

2. ⏳ **スタートアップWFの展開**
