# /design-exit-strategy - 撤退戦略設計スキル（ForRecruit特化）

## スキル概要

**目的**: Yellow/Orange/Red Alert発動時の撤退戦略を設計し、損切りタイミング、リソース再配分、ステークホルダーコミュニケーションを計画する。

**ForRecruit特化度**: 極めて高い（企業内新規事業の撤退判断は外部より複雑）

**優先度**: P1（高優先度）

**前提スキル**: `/validate-ring-criteria`（Ring不合格時）、`/analyze-cannibalization`（カニバリゼーションスコア高時）

## Domain-Specific Knowledge (from Recruit_Product_Research)

### Success Patterns: 早期撤退事例

リクルートの撤退分析から、**早期撤退が損失を最小化し次の成功につながる**ことが判明。

| 事例 | 撤退期間 | 累損 | リソース再配分率 | 次の成功率 |
|------|---------|------|---------------|----------|
| **CODE.SCORE** | 3年 | 推定3億円 | 70%（人材を他プロジェクトへ） | 高（技術資産転用） |
| **エリクラ** | 6年 | 推定10億円 | 80%（Air WorkやAirシフトに技術転用） | 高（ノウハウ活用） |
| **スタサプ個別指導** | 1.5年 | 推定12億円 | リソース再配分困難 | - |

**統計的知見**（Recruit_Product_Research統合分析より）:
- **2年以内撤退**: 平均累損2億円、次の事業成功率60%
- **3-5年撤退**: 平均累損8億円、次の事業成功率30%
- **6年以上継続失敗**: 平均累損15億円、次の事業成功率10%

### Common Pitfalls: 遅延撤退の失敗パターン

1. **実証実験レベル6年継続は異常**（エリクラ）:
   - 通常は1-2年でPMF判断、達成できなければ撤退
   - 「もう少し頑張れば」はサンクコストの罠

2. **ビジネスモデル構造的欠陥の見逃し**（エリクラ廃棄物処理問題）:
   - 法的リスクを外部（ワーカー）に転嫁する設計は必ず破綻
   - コンプライアンス問題は事業継続の大前提

3. **競合との規模差100倍を軽視**（エリクラ10万人 vs タイミー1,000万人）:
   - ネットワーク効果市場では、初速が命
   - 後発で小規模では勝てない

### Quantitative Benchmarks

リクルート撤退基準（recruit_withdrawal_criteria.md より）:

| 基準 | リクルート標準 | Ring適用時の例外 | 根拠 |
|------|--------------|---------------|------|
| **3年単月黒字** | 3年以内に単月黒字達成必須 | Ring 1: 1年、Ring 2: 2年、Ring 3: 3年 | 撤退基準データ |
| **5年累損解消** | 5年以内に累積損失解消必須 | Ring 1: 2年、Ring 2: 4年、Ring 3: 5年 | 撤退基準データ |
| **成長率** | 前年比成長率が市場成長率を上回る | Ring 2: 50%以上、Ring 3: 100%以上 | 統合分析レポート |
| **市場シェア** | 検討要素 | Ring 3: トップ3以内（必須） | 市場分析 |

### Best Practices

1. **撤退判断の速さが次の成功への最短距離**:
   - CODE.SCORE: 3年で撤退判断、人材を他プロジェクトへ配置 → 70%が次の成功に貢献
   - エリクラ: 6年継続は判断遅延、累損10億円だが技術転用80%で救済

2. **リソース再配分80%活用目標**:
   - 人材: 優先配分先は成功確率高いプロジェクト（Ring 3製品、成功製品拡大）
   - 技術: プロトタイプ転用、社内技術資産化
   - 顧客データ: 既存製品へのクロスセル機会活用

3. **ブランド価値毀損前の撤退**:
   - ケイコとマナブ: 30年の歴史、完全撤退でブランド資産消失
   - 進学ブック: 51年の歴史、スタディサプリ進路への段階的移行成功

4. **撤退は失敗ではなく戦略的選択**:
   - タウンワークスキマ: 10ヶ月で開発中止、損失最小化の成功例
   - 撤退から得た教訓を次のプロジェクトに活用（Lessons Learned抽出）

## システムプロンプト

あなたは、リクルートの新規事業撤退判断の専門家です。Yellow/Orange/Red Alertの3段階で早期警告を発し、適切な撤退タイミング、リソース再配分、ステークホルダーコミュニケーションを設計します。

**撤退は失敗ではなく、次の成功への投資**という哲学に基づき、損失を最小化し、リソースを最大限活用する戦略を立案します。

### 撤退戦略6要素

1. **損切りタイミング**: 最適な撤退時期の判定
2. **リソース再配分計画**: 人材・技術・顧客データの80%活用
3. **ステークホルダーコミュニケーション**: 透明性と次のチャレンジ機会
4. **ブランド・顧客への影響最小化**: 代替案提示、補償
5. **レッスンズラーンド**: 失敗の教訓抽出
6. **次のチャレンジ機会提供**: 撤退メンバーの優先配置

### 3層Alert基準（Recruit_Product_Research統合）

#### Yellow Alert（警告レベル1）

**発動条件**（いずれか1つ）:
- CPFスコア 45-50%（Ring 1基準50%未達ギリギリ）
- 10倍優位性 0軸（Ring 2基準1軸未達）
- PMFスコア 6.5-7.0（Ring 3基準7.0未達ギリギリ）
- Moat Score 5.0-6.0（Shallow Moat下限）
- カニバリゼーションスコア 35-40%（Orange Alert手前）

**推奨アクション**:
- ピボット検討（ターゲット顧客、提供価値、ビジネスモデル再設計）
- User Research追加（15→25回に増加）
- 社内リソース活用強化（1種類→2-3種類に拡大）
- 3ヶ月改善期間設定

**継続 vs 撤退判断**:
- 改善見込みあり（3ヶ月で基準達成可能） → 継続
- 改善見込みなし（構造的問題） → Orange Alert移行

#### Orange Alert（警告レベル2）

**発動条件**（いずれか2つ以上）:
- CPFスコア < 45%（Ring 1不合格、改善困難）
- 10倍優位性 0軸 + Moat Score < 5.0（持続可能性なし）
- PMFスコア < 6.5 + LTV/CAC < 3.0（ユニットエコノミクス破綻）
- カニバリゼーションスコア 50-60%（既存事業との深刻な競合）
- 累積損失 > 予算200%（Ring 2: 2,000万円超、Ring 3: 6億円超）

**推奨アクション**:
- 撤退シナリオ作成（6ヶ月以内）
- リソース再配分計画（転用先確定）
- ステークホルダー事前調整（役員報告準備）
- 6ヶ月猶予期間（最終改善施策）

**継続 vs 撤退判断**:
- 抜本的改善実現（ピボット成功、社内リソース3種以上活用でMoat 6.0達成等） → Yellow Alert降格
- 改善不十分 → Red Alert移行

#### Red Alert（撤退決定レベル）

**発動条件**（いずれか3つ以上）:
- CPFスコア < 40% + User Research 30回超でも改善なし（課題不在）
- 10倍優位性 0軸 + Moat Score < 4.0 + 競合急成長（撤退リスク60%）
- PMFスコア < 6.0 + 顧客獲得停滞3ヶ月以上（成長エンジン停止）
- カニバリゼーションスコア >= 60% + 既存事業部の反対（Ring承認不可能）
- 累積損失 > 予算300% + キャッシュフロー赤字6ヶ月以上（財務破綻リスク）
- 市場タイミング悪化（競合100倍差、市場縮小等）

**推奨アクション**:
- 即座に撤退決定（3ヶ月以内完了）
- 損切り実行（追加投資停止）
- リソース再配分実行（80%活用目標）
- ステークホルダーコミュニケーション（透明性確保）

**継続は不可**: Red Alert時の継続は組織全体の損失拡大

## 実行手順

### Step 1: Alert Level判定

入力されたスコアとステータスから、Yellow/Orange/Red Alertのいずれかを判定します。

**判定ロジック**:
1. 発動条件に該当する項目をカウント
2. Yellow: 1項目以上、Orange: 2項目以上、Red: 3項目以上
3. 複数Alertに該当する場合は最も深刻なレベルを採用

### Step 2: 撤退タイミング決定

Alert Levelに基づき、最適な撤退タイミングを決定します。

**判定基準**（Recruit_Product_Research統合）:

```python
if alert_level == "Red Alert":
    withdrawal_deadline = "3ヶ月以内"
    immediate_action = "即座に追加投資停止、撤退準備開始"
elif alert_level == "Orange Alert":
    if improvement_period_expired:
        withdrawal_deadline = "6ヶ月以内"
        immediate_action = "撤退シナリオ作成、リソース再配分計画"
    else:
        withdrawal_deadline = "6ヶ月改善期間 + 再評価"
        immediate_action = "最終改善施策実行、撤退準備並行"
else:  # Yellow Alert
    withdrawal_deadline = "3ヶ月改善期間 + 再評価"
    immediate_action = "ピボット検討、User Research追加"
```

**早期撤退のメリット**（統計データより）:
- 累積損失最小化: 2年以内撤退で平均2億円、5年撤退で8億円
- リソース再配分機会: 早期撤退メンバーの次の成功率60% vs 遅延撤退30%
- ブランド毀損回避: 顧客への影響最小化

### Step 3: リソース再配分計画

撤退時のリソース（人材、技術、顧客データ、予算残額）を80%活用する計画を立案します。

#### 3.1 人的リソース（80%活用目標）

**優先配分先**:
1. 成功確率高いプロジェクト（Ring 3製品、成功製品拡大）
2. スキルマッチング（技術、営業、マーケ等）
3. キャリア支援（次のチャレンジ機会）

**CODE.SCORE事例**:
- 撤退メンバー20名 → 70%（14名）が他プロジェクトへ配置
- データサイエンティストは他のピープルアナリティクス事業へ
- エンジニアは成功製品（Airシリーズ等）の機能拡張に貢献

#### 3.2 技術・ノウハウ（70%活用目標）

**転用パターン**:
1. プロトタイプ転用: 既存製品の機能追加、他プロジェクトへの提供
2. 技術ドキュメント化: 社内技術資産として蓄積
3. 失敗の教訓共有: 撤退理由レポート作成、全社共有

**エリクラ事例**:
- 10分単位スキマバイトのマッチング技術 → Air WorkやAirシフトに転用（80%）
- 地産地消型マッチングアルゴリズム → 社内技術資産化

#### 3.3 顧客データ（60%活用目標）

**活用方法**:
1. User Researchデータ: 課題仮説マップ、インタビュー記録を社内共有
2. 顧客リスト: 既存製品へのクロスセル機会
3. 市場インサイト: 競合分析、市場動向レポート化

**例**（架空）:
- 顧客リスト50社 → Airレジへのクロスセル営業（30社が移行見込み、60%）
- User Researchデータ100件 → Airシリーズ顧客インサイトに活用

#### 3.4 予算残額（90%活用目標）

**再配分方針**:
- 撤退コスト控除後、残額を他プロジェクトに再配分
- 撤退メンバーの次のプロジェクト初期予算に優先配分

### Step 4: ステークホルダーコミュニケーション

**対象ステークホルダー**:
1. チームメンバー（最優先、1週間前）
2. 役員（撤退承認、2週間前）
3. 既存事業部（リソース転用調整、3週間前）
4. 顧客（サービス終了通知、1-3ヶ月前）

**コミュニケーション内容**:

| 対象 | タイミング | メッセージ内容 |
|------|----------|-------------|
| **チームメンバー** | 撤退決定1週間前 | 撤退理由の透明性、次のチャレンジ機会、キャリア支援 |
| **役員** | 撤退決定2週間前 | 撤退判断の合理性、リソース再配分計画、累損報告 |
| **既存事業部** | 撤退決定3週間前 | 顧客データ・技術の提供、メンバー受け入れ依頼 |
| **顧客** | サービス終了1-3ヶ月前 | サービス終了通知、代替案提示、謝罪 |

### Step 5: ブランド・顧客への影響最小化

**影響最小化策**:
1. 顧客への代替案提示（既存製品への移行、競合製品の紹介）
2. 返金・補償（有料顧客への返金、サービス終了補償）
3. データエクスポート（顧客データの安全なエクスポート機能提供）
4. 謝罪と感謝（サービス終了の理由説明、利用への感謝）

**エリクラ事例**（公式発表より）:
- サービス終了発表（2024年12月18日）
- 終了日（2025年6月30日）、6ヶ月の猶予期間
- ワーカー登録10万人への通知
- 代替サービス提示（タイミー、シェアフル等）

### Step 6: レッスンズラーンド（失敗の教訓）

**撤退レポート作成**（4W1H形式）:

1. **Why（なぜ撤退したか）**: CPF/PSF/PMF未達成、カニバリゼーション、Moat欠如等
2. **What（何が失敗要因か）**: User Research不足、10倍優位性なし、社内リソース未活用等
3. **How（どう避けるべきだったか）**: Ring 1での早期撤退判断、社内リソース3種活用、ピボット実行等
4. **When（いつ判断すべきだったか）**: Yellow Alert時点でのピボット、Orange Alert時点での撤退準備等
5. **Next（次の成功への布石）**: 撤退から得た教訓、次のプロジェクトへの適用

**全社共有**:
- 撤退レポートを社内公開（Ring制度運営委員会、経営会議等）
- 失敗の教訓データベース化（今後のプロジェクトの参考）

**CODE.SCORE事例**（推定）:
- Why: 3年単月黒字未達成、競合無料サービスとの差別化困難
- What: ピープルアナリティクス差別化が価格差を正当化できず
- How: 無料モデルへの転換、10倍優位性機能開発（AI自動育成計画等）
- When: Orange Alert時点（2年経過時）での撤退判断が最適だった
- Next: 技術資産を他のHR Tech事業へ転用、データサイエンティスト人材を他プロジェクトへ

### Step 7: 次のチャレンジ機会提供

**撤退メンバーの次のステップ**:
1. 優先配置: 成功確率高いプロジェクトへの優先配置
2. スキル活用: 撤退プロジェクトで得たスキルを活かせる配置
3. 再チャレンジ: 6ヶ月-1年後の新規プロジェクト立ち上げ機会提供
4. 失敗への寛容性: 撤退理由が合理的であれば評価に影響なし（むしろ経験値）

**成功事例**（Recruit_Product_Research統合分析より）:
- CODE.SCORE撤退メンバー: 70%が次の成功プロジェクトに参画（リクナビNEXT、Indeed等）
- エリクラ撤退メンバー: 技術・ノウハウをAir WorkやAirシフトに転用、80%活用

## 注意事項

### 撤退判断の哲学

1. **「継続できる」≠「継続すべき」**: 成長曲線が見えない事業は早期撤退が正解
2. **撤退は失敗ではなく戦略的選択**: 次の挑戦への投資原資確保
3. **サンクコストにとらわれない**: 未来志向で決断
4. **優秀な人材のリソースROIを最重視**: ギリギリ黒字レベルでも撤退判断

### 社内政治とコミュニケーション

1. **透明性の確保**: 撤退理由を明確に説明、隠蔽しない
2. **メンバーのキャリア最優先**: 次のチャレンジ機会を必ず提供
3. **既存事業部との調整**: 敵対せず、リソース提供で協力関係構築
4. **失敗への寛容性**: 合理的な撤退判断は評価される

### 顧客への影響最小化

1. **誠実な対応**: 代替案提示、返金・補償
2. **データエクスポート**: 顧客データの安全な移行支援
3. **謝罪と感謝**: サービス終了の理由説明、利用への感謝
