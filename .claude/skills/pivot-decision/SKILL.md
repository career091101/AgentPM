# Pivot Decision

## 目的

CPF/PSF未達成シグナル検知時に、**Pivot 10類型から最適な代替案を提示**し、改善方向を構造化する。データドリブンな判断により、ステージゲート停止時のHuman介入を最小化し、自動Pivot推奨を実現する。

**重要な原則**（起業の科学）:
> 「Pivot（方向転換）は失敗ではなく、学習の証明である」
> — Eric Ries

**統計データ**:
```
Pivot経験スタートアップ:
- 0回: 成功率 10%
- 1-2回: 成功率 40%
- 3-5回: 成功率 60%
```

## 実行ステップ

### STEP 1: 停止原因の特定（5分）

#### 1.1 CPF停止の場合

**対象ファイル**: `documents/3_planning/cpf_diagnosis.md`

**抽出項目**: CPF 4指標

| 指標 | 目標 | 実績 | 達成判定 |
|-----|------|------|---------|
| **インタビュー数** | 20人以上 | [実績値] | ✅/❌ |
| **課題共通率** | 70%以上 | [実績値] | ✅/❌ |
| **支払い意思(WTP)** | 50%以上 | [実績値] | ✅/❌ |
| **緊急性** | 7/10以上 | [実績値] | ✅/❌ |

**Critical Failure判定**:
- ❌が2つ以上 → Pivot推奨
- ❌が1つ → Persevere（改善継続）可能性あり
- すべて✅ → CPF達成（このスキルは実行されない）

**Gap分析**:
各❌指標について、目標値との差分を定量化

```
例:
インタビュー数: 12人（目標20人、-8人、40%未達）
課題共通率: 58%（目標70%、-12%、ペルソナ定義が広すぎる）
```

#### 1.2 PSF停止の場合

**対象ファイル**: `documents/3_planning/psf_diagnosis.md`

**抽出項目**: PSF 3条件

| 条件 | 達成判定 | 詳細 |
|-----|---------|------|
| **10倍検証2軸以上** | ✅/❌ | [達成軸数]/5軸 |
| **MVP完成** | ✅/❌ | [MVP種別] |
| **UVP明確化** | ✅/❌ | [UVPスコア]/40点 |

**Critical Failure判定**:
- 10倍検証0軸 → Solution Pivot推奨
- 10倍検証1軸 → Solution強化またはPivot
- MVP未完成 → MVP Type変更
- UVP < 24/40 → Value Proposition見直し

---

### STEP 2: Pivot vs Persevere判断（3-5分）

**Pivot 5シグナル**（起業の科学定義）:

#### シグナル1: 顧客が存在しない
- インタビュー数 < 10人（3ヶ月努力後）
- 課題共通率 < 30%
→ **推奨Pivot**: Customer Segment Pivot

#### シグナル2: 課題が深刻でない
- 緊急性 < 5/10
- 支払い意思 < 20%
→ **推奨Pivot**: Customer Need Pivot

#### シグナル3: Solutionが10倍でない
- 10倍検証0軸（3ヶ月改善後も）
→ **推奨Pivot**: Technology Pivot または Zoom-in Pivot

#### シグナル4: Unit Economics不成立
- LTV/CAC < 1.0（6ヶ月改善後も）
- Payback Period > 24ヶ月
→ **推奨Pivot**: Business Architecture Pivot または Value Capture Pivot

#### シグナル5: 成長が停滞
- トラクション横ばい3ヶ月以上
- Viral係数 < 0.5（バイラル不成立）
→ **推奨Pivot**: Engine of Growth Pivot または Channel Pivot

**判断ロジック**:
```
該当シグナル数:
- 3つ以上 → ✅ Pivot実行
- 2つ → ⚠️ Pivot検討（3ヶ月改善後に再判定）
- 1つ以下 → 🔄 Persevere（改善継続）
```

**該当シグナル集計**:
各シグナルを✅/❌で評価し、合計数をカウント

---

### STEP 3: Pivot 10類型からの選択（10-15分）

**停止原因に応じた類型マッピング**

各Pivot類型の適用条件、実行内容、期待効果、成功事例を評価し、現状に最適な類型を選択する。

#### 類型1: Zoom-in Pivot（機能絞込）

**適用条件**:
- 10倍検証1-2軸のみ達成（他軸は評価低）
- 特定機能のみ顧客反応が良い
- 複数機能で混乱している

**実行内容**:
1. 機能別使用率分析（データ収集）
2. 評価高い1機能に絞込
3. 他機能を全削除
4. UI/UX最適化

**期待効果**:
- 開発期間: -50%
- UVP明確化: 24/40 → 35/40
- 10倍検証: 1軸 → 3軸達成
- MVP完成度向上

**成功事例**: Instagram（Burbn 11機能 → 写真シェア特化）
```
Before: Burbn（位置情報SNS、11機能）
データ: 写真シェアのみ使用率80%
After: Instagram（写真シェア特化）
結果: 2ヶ月で100万ユーザー
```

**実装期間**: 2-4週間
**リスク**: 低（削除した機能を求める声があるリスク）

---

#### 類型2: Zoom-out Pivot（機能拡張）

**適用条件**:
- 既存機能が「部分解決」のみ
- 顧客が「もっと包括的に」と要望
- 単機能では収益化困難

**実行内容**:
1. 周辺ニーズ調査（インタビュー20件）
2. 関連機能を追加しプラットフォーム化
3. 統合UX設計

**期待効果**:
- ARPU: +50-100%
- Churn率削減: -30%
- LTV向上: +80%

**成功事例**: Slack（ゲーム内チャット → ビジネスチャットプラットフォーム）

**実装期間**: 2-3ヶ月
**リスク**: 中（機能追加でUX複雑化リスク）

---

#### 類型3: Customer Segment Pivot（顧客層変更）

**適用条件**:
- インタビュー数 < 10人（ターゲット市場が小さい）
- 課題共通率 < 40%（ペルソナ定義が広すぎる）
- 想定外の顧客層から強い反応

**実行内容**:
1. ペルソナ再定義（例: CTO/IT部長/経営層 → IT部長のみ）
2. `/create-persona` 再実行
3. `/simulate-interview` 再実行（新ペルソナ3人）
4. `/validate-cpf` 再実行

**期待効果**:
- 課題共通率: +30-50%（例: 58% → 85%）
- 支払い意思: +20-40%（例: 42% → 65%）
- 緊急性: +2.0/10（例: 6.5/10 → 9/10）
- CPFスコア: +25-35%（例: 45% → 78%）

**成功事例**: Slack（ゲーム開発チーム → 全企業向け）

**実装期間**: 1週間
**リスク**: 低（市場規模縮小の可能性、ただし絞込で成功率向上）

---

#### 類型4: Customer Need Pivot（課題変更）

**適用条件**:
- 緊急性 < 5/10（当初想定の課題が軽微）
- インタビューで別の深刻課題を発見
- 支払い意思はあるが別用途

**実行内容**:
1. Problem再定義（例: 「AI導入失敗」→「AI定着率30%問題」）
2. `/research-problem` 再実行（新キーワードで検索）
3. `/simulate-interview` 再実行（新Problem前提でUVP再検証）
4. `/validate-cpf` 再実行

**期待効果**:
- 緊急性: 5/10 → 8/10以上
- 支払い意思: +15-30%
- CPFスコア: +20-30%

**成功事例**: Instagram（位置情報SNS → 写真共有）、YouTube（動画デート → 動画共有）

**実装期間**: 1.5-2週間
**リスク**: 中（既存Solutionとの不整合リスク）

---

#### 類型5: Platform Pivot（プラットフォーム化）

**適用条件**:
- 単一アプリでの成長限界
- サードパーティ需要の顕在化
- API公開要望が多い

**実行内容**:
1. API設計・公開
2. マーケットプレイス開設
3. デベロッパーコミュニティ構築

**期待効果**:
- ネットワーク効果発現
- バイラル係数: 0.5 → 2.0以上
- LTV: +100-300%

**成功事例**: Salesforce（CRM → AppExchange）

**実装期間**: 3-6ヶ月
**リスク**: 高（技術投資大、成功まで時間）

---

#### 類型6: Business Architecture Pivot（B2C⇄B2B）

**適用条件**:
- LTV/CAC < 1.0（Unit Economics不成立）
- B2C市場で競合優位性なし
- B2B市場でニーズ発見

**実行内容**:
1. ターゲット変更（個人 → 企業、またはその逆）
2. 収益モデル変更（サブスク → エンタープライズ契約）
3. 営業体制構築

**期待効果**:
- LTV: +100-300%（B2C→B2Bの場合）
- CAC: +50-100%（営業コスト増）
- Churn率: 高→低（B2Bの場合）
- LTV/CAC比率: 0.6 → 3.0以上

**成功事例**: Yammer（B2C SNS → B2B社内SNS、Microsoft $1.2B買収）

**実装期間**: 2-4ヶ月
**リスク**: 高（ビジネスモデル全面変更）

---

#### 類型7: Value Capture Pivot（価格戦略変更）

**適用条件**:
- 支払い意思はあるが価格感度高い（WTP 30-50%）
- 競合が低価格攻勢
- 現在の収益モデルで支払意思 < 30%

**実行内容**:
1. 収益モデル変更:
   - 有料 → Freemium
   - 買い切り → サブスクリプション
   - サブスク → 従量課金
2. `/apply-lean-canvas` 更新（収益モデル欄変更）
3. `/simulate-interview` 再実行（新価格での支払意思検証）

**期待効果**:
- 顧客獲得数: +200-500%（Freemium導入時）
- ARPU: +50-100%（Enterprise Plan追加時）
- LTV: +80-150%

**成功事例**: Dropbox（有料のみ → Freemium）、Adobe（買い切り → サブスク）

**実装期間**: 1-2週間
**リスク**: 中〜高（ARPU低下リスク、Freemium→有料転換率次第）

---

#### 類型8: Engine of Growth Pivot（成長エンジン変更）

**適用条件**:
- トラクション横ばい3ヶ月以上
- 既存成長戦略が機能停止
- バイラル係数 < 0.5（バイラル不成立）

**Eric Riesの3つの成長エンジン**:
1. **Sticky**: 高Retention、低Churn
2. **Viral**: 紹介・招待ループ
3. **Paid**: 広告・営業

**実行内容**:
1. 現在の成長エンジン特定
2. 別エンジンへの転換:
   - Paid → Viral（紹介プログラム追加）
   - Viral → Sticky（Retention施策強化）
3. KPI変更（CAC → K-factor等）

**期待効果**:
- CAC: -50-70%（Viral化時）
- 成長率: +100-300%
- LTV/CAC比率改善

**成功事例**: Hotmail（広告 → "PS: I love you" 署名バイラル）

**実装期間**: 2-4週間
**リスク**: 中（バイラル成功は不確実）

---

#### 類型9: Channel Pivot（販売チャネル変更）

**適用条件**:
- CAC高騰（現チャネルの限界）
- CAC/LTV比率 > 0.33（不健全）
- 別チャネルでCAC 1/10の実績

**実行内容**:
1. チャネル別CAC分析
2. チャネル変更:
   - 直販 → パートナー販売
   - オンライン → オフライン
   - 展示会 → コンテンツマーケティング
3. GTM戦略変更

**期待効果**:
- CAC: -40-60%
- リーチ: +200-500%
- LTV/CAC比率改善

**成功事例**: Zoom（直販 → OEM/リセラー追加）

**実装期間**: 2-3ヶ月
**リスク**: 中（新チャネル開拓の不確実性）

---

#### 類型10: Technology Pivot（技術スタック変更）

**適用条件**:
- 10倍検証0軸（技術的制約が原因）
- 技術的負債が成長阻害
- 新技術でコスト1/10、速度10倍実現可能

**実行内容**:
1. 技術基盤の再構築（例: デスクトップ → Web、オンプレ → クラウド）
2. MVP再実装
3. 段階的移行

**期待効果**:
- 10倍検証: 0軸 → 2軸以上達成
- 開発速度: +50-100%
- インフラコスト: -50-80%

**成功事例**: Netflix（DVD郵送 → ストリーミング）

**実装期間**: 2-6ヶ月
**リスク**: 高（技術リスク、大規模投資）

---

### STEP 4: Top 3代替案の提示（5-10分）

**選択ロジック**:
1. 停止原因に最も直結する類型を特定（STEP 1, 2の分析結果から）
2. 実装工数・リスクを考慮
3. 期待CPF/PSFスコア改善値を試算
4. 成功事例との類似性評価

**出力形式**:
各代替案につき以下を提示:
- Pivot類型名
- 実行内容（3-5ステップ）
- 期待CPF/PSFスコア改善（定量値）
- 実装期間
- リスク評価（低/中/高）
- 推奨度（⭐⭐⭐ / ⭐⭐ / ⭐）

**優先順位付け基準**:
1. CPF/PSFスコア改善インパクト（最重視）
2. 実装期間（短いほど優先）
3. リスク（低いほど優先）
4. 成功事例との類似性（高いほど優先）

---

### STEP 5: 推奨案の選定（2-5分）

**最終選定基準**:

| 基準 | 重み | 評価方法 |
|------|------|---------|
| **CPF/PSFスコア改善** | 50% | 改善率（%） |
| **実装期間** | 20% | 短いほど高評価 |
| **リスク** | 20% | 低いほど高評価 |
| **成功事例類似性** | 10% | 類似度評価 |

**スコアリング**:
```
総合スコア =
  CPF改善率(%) × 0.5 +
  (1 - 実装期間/6ヶ月) × 20 +
  リスク評価(低10/中5/高0) × 2 +
  成功事例類似性(0-10) × 1

最高スコアの案に ⭐⭐⭐（推奨）マーク付与
```

**推奨案の明示**:
- 選定理由（CPF/PSF改善インパクト、実装容易性等）を3-5行で説明
- 実行後の期待状態を定量的に提示

## 3. Pivot 10類型からの代替案

### 代替案1: Customer Segment Pivot（顧客層変更）⭐⭐⭐ 推奨

**適用理由**:
- 課題共通率58%（ペルソナ定義が広すぎる）
- インタビューでIT部長層の反応が突出（緊急性9/10）
- 3層ターゲット（CTO/IT部長/経営層）→ 1層に絞込で精度向上

**実行内容**:
1. **ペルソナ再定義**:
   - 変更前: CTO/IT部長/経営層の3層
   - 変更後: IT部長層のみに絞込（最も切実な層）

2. **`/create-persona` 再実行**:
   - IT部長の課題（AI導入後の定着率30%問題）に特化
   - 3Uスコア: 各8/10以上目標

3. **`/simulate-interview` 再実行**:
   - IT部長ペルソナ3人で検証
   - UVP刺さり度: 32/40以上目標

4. **`/validate-cpf` 再実行**:
   - CPFスコア60%以上達成確認

**期待効果**:

| 指標 | 改善前 | 改善後 | 変化 |
|-----|--------|--------|------|
| **インタビュー数** | 12人 | 20人 | +8人（絞込でリーチ容易化） |
| **課題共通率** | 58% | 85% | +27%（IT部長は高共通性） |
| **支払い意思** | 42% | 65% | +23%（緊急性と連動） |
| **緊急性** | 6.5/10 | 9.0/10 | +2.5（IT部長は今期必達） |
| **CPFスコア** | 45% | 78% | **+33% ✅** |

**実装期間**: 1週間（45-60分 × 3スキル再実行）
