# CPFパターン P4: マッチング効率の革命

## パターン概要

求職者と企業、顧客と店舗など「需要と供給のマッチング」を、紙媒体や電話から、Web検索・条件絞り込みに転換するパターン。

## 指標

| 指標 | 値 |
|------|-----|
| **課題共通率** | 80% |
| **緊急性** | 8/10 |
| **適用ティア** | TIER2/TIER5 両面市場型 |
| **代表事例** | リクナビNEXT、ホットペッパーグルメ、ゼクシィ相談カウンター |

## 課題の本質

求職者が求人情報を探すのが難しい、または顧客が飲食店を探すのに時間がかかるなど、需要と供給の「マッチング探索コスト」が高い問題を解決。

### 具体的な課題

**求人・転職の場合：**
- 転職希望者が求人情報に簡単にアクセスできない
- 企業と求職者の距離が遠い
- マッチング機会が制限される

**飲食店予約の場合：**
- 電話予約しか受け付けていない店舗が多い
- 営業時間内に予約する必要がある
- 複数店舗を比較するのに時間がかかる

**結婚式場の場合：**
- 情報が多すぎて、どの式場を選べばいいかわからない
- 式場の選択肢が限定される

## 代表事例: リクナビNEXT（CORP_S011）

### 課題共通率: 80%

転職希望者が求人情報にアクセスしづらい。

### 緊急性: 8/10

転職は期限があり、機会損失が大きい。

### WTP（支払意思）確認

企業側が求人掲載料を支払う（1件数万円～数十万円）。

### 検証手法

- 既存の「B-ing」（紙媒体）からの移行
- リクナビ（新卒）の成功モデルを転用

## 他の適用事例

### ホットペッパーグルメ（CORP_S008）

予約効率化：
- 従来: 電話予約のみ
- ホットペッパー: ネット即予約
- **時間削減: 10倍以上**

クーポン配布：
- 従来: 紙クーポン（配布が限定的）
- ホットペッパー: スマホクーポン（全国配布可能）

### ゼクシィ相談カウンター（CORP_S025）

マッチング支援：
- 従来: 雑誌だけでは決められない
- ゼクシィ相談カウンター: 専門アドバイザーが1-2時間で提案
- **意思決定時間: 10倍削減**

## orchestrate-phase1への示唆

### ✅ アクション

需要と供給の「探索コスト」を定量測定し、10倍以上の削減を実現。

**測定方法：**
- 現在、店舗・企業を探すのに何時間かかるか
- 複数の選択肢を比較するのに何時間かかるか
- その過程で「やめてしまった」経験は何度か

### ✅ 両面市場の設計

片側（求職者/顧客）は無料、もう片側（企業/店舗）が支払う。

**ビジネスモデル：**
- 求職者側: 無料で求人閲覧・応募
- 企業側: 掲載料を支払う（月額5万円～数十万円）

**重要な指標：**
- 求職者数が多いほど、企業の掲載意欲が上がる
- 企業の求人が多いほど、求職者の利用継続意欲が上がる
- ネットワーク効果が最大の競争優位性

### ✅ ネットワーク効果

マッチング数が増えるほど価値が向上するフライホイール設計。

**フライホイール例（求人の場合）：**
1. 求職者が増える
2. → 企業の求人掲載意欲が上がる
3. → 求人が増える
4. → 求職者が増える（ループ）

## 実装のポイント

### 1. ターゲット市場の選択

初期段階で課題共通率80%以上を達成する市場を選択：

- 求人市場（リクナビNEXT）
- 飲食店予約（ホットペッパー）
- 結婚式場選択（ゼクシィ相談カウンター）

### 2. 初期マッチング数の確保

**立ち上げの鶏と卵問題を解決：**

**デマンド側（求職者）優先：**
- 既存メディア（紙誌）のユーザーを誘導
- SNS・ワードオブマウスで拡大

**サプライ側（企業）優先：**
- 営業チームで直接掲載を促進
- 初期は成功事例を集中的に作成

### 3. 両面市場の初期成長戦略

**フェーズ1（0～3ヶ月）：**
- サプライ側（企業）の掲載数を確保（営業チーム10名以上）
- デマンド側（求職者）を5万人以上集める

**フェーズ2（3～12ヶ月）：**
- 月次マッチング数が指数関数的に増加
- 企業側のWTP（掲載継続意欲）が高まる
- 有料プランへのアップグレードが進む

**フェーズ3（12ヶ月以上）：**
- 業界標準プラットフォームとしての地位確立
- ネットワーク効果で競合の参入が困難に

### 4. 検索・マッチング機能の実装

基本機能：
- 条件絞り込み検索（地域、職種、給与等）
- リアルタイムランキング（人気求人）
- スカウト機能（企業→求職者への逆マッチング）

高度な機能：
- AI推薦（機械学習による個別最適化マッチング）
- スキルマッチング（技術スキルと要件の自動マッチング）

## PSF（Problem-Solution Fit）での10倍優位性軸

このパターンは以下の10倍優位性を実現：

| 軸 | 倍率 | 説明 |
|-----|------|------|
| **探索効率** | 10倍 | 複数訪問 → 1サイトで検索 |
| **情報量** | 10倍 | 限定的な情報 → 全国の情報 |
| **更新速度** | 50倍 | 紙年1回 → Web週1回（即時） |
| **検索性** | 100倍 | 手動検索 → 条件絞り込み検索 |
| **マッチング精度** | 15倍 | 手作業 → AIアルゴリズム |

## 失敗を避けるポイント

### 警告1: 初期段階での片側だけの成長

- 求職者数は多いが、企業の掲載が少ない
- 企業の掲載は多いが、求職者が集まらない

**対策：**
- デマンド側とサプライ側のバランスを意識的に管理
- KPI: 「求職者100人当たり、企業掲載数5～10」を目安に

### 警告2: ネットワーク効果の過小評価

プラットフォームビジネスでは、ネットワーク効果が最重要。

- 10倍優位性があっても、会員数で劣位なら敗北
- ゼクシィ縁結び（225万人）vs Pairs（2,000万人）の事例

**対策：**
- 初期から会員数獲得に全力投球
- ネットワーク効果が働き始める「臨界点」（100万人～）を目指す
