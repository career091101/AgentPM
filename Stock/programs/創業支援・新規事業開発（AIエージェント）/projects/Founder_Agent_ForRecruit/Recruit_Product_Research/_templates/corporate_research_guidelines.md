# Corporate_Product_Research 調査ガイドライン

最終更新: 2025-12-30
バージョン: 1.0

## 目的

このガイドラインは、Corporate_Product_Researchプロジェクトにおけるケーススタディ調査の品質を担保するための指針です。特に以下の3つのフィールドの記載基準を明確化します：

- **user_research_count**: ユーザー調査数（大企業版のinterview_count）
- **market_need_percentage**: 市場ニーズ率（大企業版のproblem_commonality）
- **competitive_advantage_axes**: 競合優位性（大企業版のten_x_axes、2-5倍でも可）

---

## 1. 90分調査プロセス

### Phase 1: 情報収集（25分）

#### 1.1 公式ソース収集（10分）

**優先度1（Tier 1）: リクルート公式ソース**
- リクルートホールディングス公式サイト
- 製品公式サイト（過去サイトも含む）
- プレスリリース（リクルート公式、PR TIMES等）
- 利用規約・サービス規約

**優先度2（Tier 2）: 企業公式ドキュメント**
- IR資料（決算説明会資料、アニュアルレポート）
- 有価証券報告書
- 事業報告書

検索クエリ例:
```
"[製品名]" site:recruit.co.jp
"[製品名]" リクルート プレスリリース
"[製品名]" リクルート IR
```

#### 1.2 第三者ソース収集（10分）

**優先度3（Tier 3）: 信頼性の高いメディア**
- 日本経済新聞
- 東洋経済オンライン
- ITmedia
- TechCrunch Japan
- ダイヤモンド・オンライン

**優先度4（Tier 4）: 市場調査レポート**
- 矢野経済研究所
- MM総研
- Gartner（グローバル製品時）
- Forrester（グローバル製品時）

検索クエリ例:
```
"[製品名]" リクルート 日経
"[製品名]" 市場調査 矢野経済
"[製品名]" 業界レポート
```

#### 1.3 Archive収集（5分）- 撤退製品時のみ

**優先度5（Tier 5）: Wayback Machine等**
- Wayback Machine（撤退製品の公式サイト）
- ユーザーレビューサイト（価格.com、みん評等）
- 個人ブログ（当時のユーザー体験）

検索クエリ例:
```
site:web.archive.org "[製品名]"
"[製品名]" 使ってみた ブログ
"[製品名]" レビュー site:kakaku.com
```

**ソースTier階層の目標配分**:
- Tier 1+2: 全体の50%以上
- Tier 3+4: 全体の30%以上
- Tier 5: 補完的に10-20%

### Phase 2: データ分析（20分）

#### 2.1 ビジネスモデル分析（7分）

**収益構造の解明**:
- マネタイゼーション: 広告/手数料/サブスク/成果報酬
- 価格戦略: 無料/フリーミアム/有料のみ
- Unit Economics: LTV/CAC比（推定可）

調査項目:
- [ ] 課金モデルの特定
- [ ] 価格帯の確認（公式サイト、プレスリリース）
- [ ] 競合との価格比較

**記載例**:
```yaml
# 例: Airレジ
revenue_model: "無料SaaS + 周辺機器販売 + 決済手数料"
pricing: "基本無料、Airペイ連携で決済手数料3.24%"
unit_economics: "推定LTV 120万円、CAC 15万円、LTV/CAC = 8倍"
```

#### 2.2 市場分析（7分）

**TAM/SAM/SOM推定**:
- TAM（Total Addressable Market）: 市場全体規模
- SAM（Serviceable Available Market）: 実際に到達可能な市場
- SOM（Serviceable Obtainable Market）: 初期で獲得可能な市場

調査項目:
- [ ] 業界市場規模（矢野経済、MM総研等）
- [ ] ターゲットセグメント（事業者数、ユーザー数）
- [ ] 市場シェア推定（公開情報ベース）

**記載例**:
```yaml
# 例: スタディサプリ
market_size:
  TAM: "日本の学習塾市場 9,600億円（矢野経済研究所 2018）"
  SAM: "オンライン学習市場 800億円（推定: TAM 8.3%）"
  SOM: "中高生向けサブスク 200億円（推定: SAM 25%）"
market_share: "2019年時点で約10%（推定: 売上200億円/SOM 200億円）"
```

#### 2.3 Timeline構築（6分）

**主要マイルストーンの特定**:
- ローンチ年月
- 主要アップデート
- ユーザー数マイルストーン
- M&A/撤退時期（該当時）

調査項目:
- [ ] プレスリリースから年表作成
- [ ] IR資料から業績変化点の特定
- [ ] メディア報道から重要イベントの抽出

**記載例**:
```yaml
timeline:
  - date: "2012-03"
    event: "サービス開始（iOS版のみ）"
  - date: "2013-04"
    event: "Android版リリース"
  - date: "2015-09"
    event: "100万ダウンロード達成"
  - date: "2019-12"
    event: "累計会員数200万人突破"
```

### Phase 3: CPF/PMF検証（20分）

#### 3.1 User研究調査（5分）

**user_research_countの推定・確定**

大企業製品は創業者インタビューが少ないため、推定が中心となります。

**ケース1: 公式発表がある場合**
```yaml
user_research_count: 50
# ソース: "100人の教員にヒアリング実施" - プレスリリース 2018-XX-XX
```

**ケース2: 推定可能な場合**

| 製品タイプ | 推定値 | 根拠 |
|----------|--------|------|
| 大企業B2B SaaS | 20-30社 | 企業向け製品は少数深掘り型 |
| 大企業B2C Web/App | 50-100人 | 消費者向けは量的調査が中心 |
| 既存事業派生製品 | 10-20件 | 既存顧客ベースで仮説検証 |
| M&A製品（創業時） | 創業時の値を採用 | M&A前の創業フェーズを参照 |

```yaml
# 例: Airレジ（B2B SaaS）
user_research_count: 25
# 推定: B2B SaaS業界標準、リクルートライフスタイル既存顧客からのヒアリング

# 例: ゼクシィ縁結び（B2C App）
user_research_count: 60
# 推定: ゼクシィ既存ユーザーへのアンケート + インタビュー実施の記載あり
```

**ケース3: データなし（情報源が全くない場合）**
```yaml
user_research_count: 0
# 情報源なし、M&A後の統合製品のためリクルート独自調査は未実施と推測
```

**重要**: `null`のまま残さない！必ず `0` または推定値を記載する。

#### 3.2 市場ニーズ調査（7分）

**market_need_percentageの算出**

大企業製品は市場調査レポートを活用しやすいため、積極的に引用します。

**定義**: ターゲット市場において、その課題を抱える人/企業の割合（%値）

**ケース1: 市場調査データがある場合**
```yaml
market_need_percentage: 75
# ソース: "中小飲食店の75%が予約管理に課題" - 矢野経済研究所 2016年調査
```

**ケース2: IR資料・プレスリリースから逆算**
```yaml
# 例: スタディサプリ
# ターゲット: 中高生 630万人（総務省統計）
# 課題: 学習塾に通えない地方・経済的制約のある層
# 推定: 学習塾普及率 48%（都市部70%、地方30%）→ 未普及層 52%

market_need_percentage: 52
# 推定: 総務省統計より学習塾非通学者率を算出
```

**ケース3: 業界ベンチマークから推定**

| 業界 | 典型的な市場ニーズ率 | 根拠 |
|------|-------------------|------|
| B2B生産性ツール | 60-70% | 大半の企業が生産性課題を認識 |
| B2Cマッチングサービス | 30-50% | 特定ニーズを持つセグメント |
| ニッチB2B | 20-30% | 特定業界・職種に限定 |
| 大企業内派生製品 | 50-60% | 既存顧客基盤の課題から着想 |

```yaml
# 例: Airシフト（B2B生産性ツール）
market_need_percentage: 65
# 推定: B2B生産性ツールの業界標準、シフト管理課題は飲食・小売業界で普遍的

# 例: ゼクシィ縁結びカウンター（ニッチB2C）
market_need_percentage: 35
# 推定: 婚活意向者のうち対面カウンセリング希望層（業界調査より推定）
```

**ケース4: データ不足時の保守的推定**
```yaml
market_need_percentage: 50
# 保守的推定: 定性情報のみ、業界標準を適用
# 根拠: "多くの事業者が課題を感じている"との記載のみ
```

**重要**: `null`のまま残さない！必ず推定値を記載する。

#### 3.3 競合優位性分析（8分）

**competitive_advantage_axesの特定（2-5倍でOK）**

大企業製品は「10倍優位性」ではなく「2-5倍の競合優位性」で判定します。

**必須条件**:
- 最低2軸
- 定量化必須
- リクルート資産活用を重視

**軸の種類と定量化例**:

| 軸 | 定量化方法 | 例 |
|---|-----------|---|
| コスト | 価格比較 | "競合月額5,000円 → 自社0円（広告モデル）" = 無限倍 |
| 導入時間 | セットアップ時間 | "競合1週間 → 自社1日" = 7倍 |
| 使いやすさ | クリック数 | "競合10クリック → 自社2クリック" = 5倍 |
| 営業リーチ | 営業拠点数 | "競合10拠点 → リクルート全国47都道府県" = 4.7倍 |
| データ量 | 掲載件数 | "競合1万件 → 自社5万件" = 5倍 |
| ブランド認知 | 認知率 | "競合認知率10% → リクルート既存ブランド60%" = 6倍 |

**記載例**:
```yaml
# 例: Airレジ
competitive_advantage_axes:
  - axis: "コスト"
    multiplier: 5
    detail: "競合POSレジ月額5,000円 → Airレジ基本無料（5倍）"
    source: "価格.com POSレジ比較 2014"
  - axis: "導入時間"
    multiplier: 10
    detail: "競合1週間（設置工事必要） → Airレジ即日（iPad + アプリ）"
    source: "公式サイト導入事例"
  - axis: "リクルート営業網"
    multiplier: 3
    detail: "競合オンライン販売のみ → リクルート全国営業拠点でサポート"
    source: "リクルートライフスタイル事業説明会 2014"

# 例: スタディサプリ
competitive_advantage_axes:
  - axis: "コスト"
    multiplier: 10
    detail: "学習塾月額3万円 → スタサプ月額980円（約30倍安い）"
    source: "文部科学省 学習塾費用調査 2018"
  - axis: "地理的制約"
    multiplier: 無限
    detail: "学習塾は通学必要 → スタサプは全国どこでも受講可能"
    source: "公式サイト"
```

**リクルート資産活用の重視**:

大企業製品の競合優位性は「リクルートの既存資産をどう活用したか」が重要です。

| 資産タイプ | 競合優位性への転換 | 倍率目安 |
|----------|----------------|---------|
| 営業網 | 全国47都道府県 vs 競合10拠点 | 4-5倍 |
| ブランド | 認知率60% vs 競合10% | 6倍 |
| データベース | 掲載件数5万 vs 競合1万 | 5倍 |
| 既存顧客基盤 | クロスセル可能 vs 競合新規獲得のみ | 3倍（CAC削減） |

**達成目標**:
- 2軸以上: ✅ 達成
- 1軸のみ: ⚠️ 要改善
- 0軸: ❌ 未達成

### Phase 4: リクルート資産分析（10分）

#### 4.1 活用資産の特定（5分）

**調査項目**:
- [ ] 営業網: 全国営業拠点、既存営業チームの活用
- [ ] ブランド: 既存ブランドの拡張（ホットペッパー、リクナビ等）
- [ ] データベース: 既存会員DB、掲載事業者DBの活用
- [ ] プラットフォーム: 既存Webサイト、アプリへの統合
- [ ] 顧客基盤: 既存顧客へのクロスセル

**記載例**:
```yaml
leveraged_assets:
  - asset_type: "営業網"
    description: "ホットペッパーグルメの全国営業チーム（約2,000名）を活用し、Airレジを直販"
    impact: "CAC 1/3に削減（推定）"
  - asset_type: "ブランド"
    description: "ゼクシィブランドを拡張し、ゼクシィ縁結びとしてローンチ"
    impact: "認知率60%からスタート（競合は0%）"
  - asset_type: "データベース"
    description: "リクナビ既存会員400万人にリクナビNEXTを告知"
    impact: "初月10万人獲得（通常は1万人）"
```

#### 4.2 シナジー効果の定量化（5分）

**シナジータイプ**:
- データ連携: 既存サービスとのデータ共有
- クロスセル: 既存顧客への追加販売
- コスト削減: 共通インフラ、営業チーム共有
- スケール効果: プラットフォーム統合による規模拡大

**記載例**:
```yaml
synergy_with_existing:
  - business: "ホットペッパーグルメ"
    synergy_type: "クロスセル"
    quantification: "掲載店舗8万店のうち1万店がAirレジ導入（12.5%転換）"
  - business: "ゼクシィ"
    synergy_type: "ブランド共鳴"
    quantification: "ゼクシィ認知率85% → 縁結び初期認知率60%（通常5-10%）"
  - business: "リクルートライフスタイル既存事業"
    synergy_type: "データ連携"
    quantification: "Airシリーズ5製品でデータ統合、クロスセル率25%"
```

## 2. 業界標準値の定義

### 2.1 user_research_count（ユーザー調査数）

大企業製品は創業時の詳細が不明なことが多いため、業界標準から推定します。

| 製品タイプ | 標準値 | 根拠 |
|----------|--------|------|
| 大企業B2B SaaS | 20-30社 | 企業向けは少数深掘り型、リクルートは既存顧客基盤活用 |
| 大企業B2C Web/App | 50-100人 | 消費者向けは量的調査、アンケート中心 |
| 既存事業派生 | 10-20件 | 既存顧客ヒアリングで仮説検証済み |
| M&A製品（創業時） | 創業時の値 | M&A前のFounder_Researchを参照 |
| データなし | 0 | 理由明記（M&A後統合、プロダクト主導等） |

**記載例**:
```yaml
# 例1: Airレジ（既存事業派生）
user_research_count: 15
# 推定: ホットペッパーグルメ掲載店舗へのヒアリング実施（業界標準10-20件）

# 例2: Indeed（M&A製品）
user_research_count: 100
# ソース: 創業者Paul Forster氏のインタビュー（100社以上の人事担当者に調査）

# 例3: ゼクシィ（データなし、1993年創業）
user_research_count: 0
# 情報源なし、当時はリサーチ手法が確立されていない時代
```

### 2.2 market_need_percentage（市場ニーズ率）

大企業製品は市場調査レポートを活用しやすいため、積極的に引用します。

| 業界 | 標準値 | 根拠 |
|------|--------|------|
| B2B生産性ツール | 60-70% | 大半の企業が生産性課題を認識（パーソル総研等） |
| B2Cマッチングサービス | 30-50% | 特定ニーズを持つセグメント |
| ニッチB2B | 20-30% | 特定業界・職種に限定 |
| 大企業内派生製品 | 50-60% | 既存顧客基盤の課題から着想 |
| データ不足時 | 50% | 保守的推定 |

**推定の優先順位**:
1. 市場調査レポート（矢野経済、MM総研等）
2. IR資料・プレスリリースの記載
3. 業界標準値
4. 保守的推定50%

**記載例**:
```yaml
# 例1: スタディサプリ（市場調査データあり）
market_need_percentage: 52
# ソース: 総務省統計「学習塾非通学者率52%」（2018）

# 例2: Airレジ（業界標準）
market_need_percentage: 65
# 推定: B2B生産性ツールの業界標準60-70%、POSレジ課題は飲食・小売で普遍的

# 例3: ゼクシィ縁結びカウンター（保守的推定）
market_need_percentage: 35
# 推定: 婚活意向者のうち対面カウンセリング希望層（業界調査より推定）
```

### 2.3 competitive_advantage_axes（競合優位性）

**大企業製品は「2-5倍の競合優位性」で判定**

10倍優位性（創業者型）ではなく、大企業の現実的な優位性を評価します。

**必須条件**:
- 最低2軸
- 定量化必須（"コスト 1/5" "導入時間 1/10" など）
- リクルート資産活用を重視

**軸の優先順位**:
1. リクルート資産活用（営業網、ブランド、DB）
2. コスト優位性
3. 導入・利用の容易さ
4. データ量・品質

**倍率の現実的範囲**:
- 2-3倍: 標準的な競合優位性
- 4-5倍: 強い競合優位性
- 6-10倍: 非常に強い競合優位性（稀）
- 10倍以上: 創業者型製品のみ（大企業では稀）

**記載例**:
```yaml
# 例1: Airレジ（3軸達成）
competitive_advantage_axes:
```

## 3. ソースTier階層の定義

信頼性に基づく5段階階層を定義します。

### Tier 1（最優先）: 公式ソース
- リクルート公式サイト
- 製品公式サイト
- プレスリリース（リクルート公式、PR TIMES）
- 利用規約・サービス規約

**信頼性**: 最高
**活用例**: ローンチ年、価格、機能、公式発表数値

### Tier 2（高信頼）: 企業公式ドキュメント
- IR資料（決算説明会資料、アニュアルレポート）
- 有価証券報告書
- 事業報告書
- SECフィリング（グローバル製品）

**信頼性**: 高
**活用例**: 売上、ユーザー数、M&A金額、事業戦略

### Tier 3（信頼）: 信頼性の高いメディア
- 日本経済新聞
- 東洋経済オンライン
- ITmedia
- TechCrunch Japan
- ダイヤモンド・オンライン
- Business Insider Japan

**信頼性**: 中〜高
**活用例**: 製品背景、市場動向、競合比較、専門家コメント

### Tier 4（参考）: 市場調査レポート
- 矢野経済研究所
- MM総研
- IDC Japan
- Gartner（グローバル製品）
- Forrester（グローバル製品）

**信頼性**: 中
**活用例**: 市場規模、業界標準値、トレンド分析

### Tier 5（補完）: Archive・UGC
- Wayback Machine（撤退製品の過去サイト）
- ユーザーレビュー（価格.com、みん評、App Store等）
- 個人ブログ（当時のユーザー体験）
- SNS投稿（Twitter、はてなブックマーク等）

**信頼性**: 低〜中
**活用例**: 撤退製品の情報、ユーザー体験、定性的な評価

## 4. データ不足時の推定ルール

大企業製品は創業時の詳細が不明なことが多いため、推定ルールを明確化します。

### 4.1 推定の基本原則

1. **保守的に推定**: 過大評価を避ける
2. **根拠を明記**: 推定方法を必ず記載
3. **複数ソースから類推**: 1つの情報源のみで推定しない
4. **業界標準を活用**: 類似製品の標準値を参照
5. **完全に不明な場合**: "不明（要追加調査）" と明記

### 4.2 フィールド別推定ルール

#### user_research_count（ユーザー調査数）

**推定可能な場合**:
```yaml
# パターン1: 製品タイプから推定
user_research_count: 25
# 推定: B2B SaaS業界標準20-30社

# パターン2: 既存事業から推定
user_research_count: 15
# 推定: ホットペッパー既存顧客ヒアリング（業界標準10-20件）

# パターン3: プレスリリースから推定
user_research_count: 50
# 推定: "多数の教員にヒアリング"との記載あり、教育業界標準50-100人
```

**推定不可の場合**:
```yaml
user_research_count: 0
# 情報源なし、1990年代創業でリサーチ手法未確立の時代
```

#### market_need_percentage（市場ニーズ率）

**推定可能な場合**:
```yaml
# パターン1: 統計データから推定
market_need_percentage: 52
# 推定: 総務省統計「学習塾非通学者率52%」を適用

# パターン2: TAM/SAM/SOMから逆算
market_need_percentage: 35
# 推定: TAM 1,000万人、SAM 350万人 → 35%

# パターン3: 業界標準を適用
market_need_percentage: 65
# 推定: B2B生産性ツールの業界標準60-70%
```

**推定不可の場合**:
```yaml
market_need_percentage: 50
# 保守的推定: 定性情報のみ、業界標準を適用
```

#### competitive_advantage_axes（競合優位性）

**推定可能な場合**:
```yaml
# パターン1: 価格比較から算出
competitive_advantage_axes:
  - axis: "コスト"
    multiplier: 5
    detail: "競合月額5,000円 → 自社1,000円"
    source: "価格.com比較データ 2015"

# パターン2: リクルート資産から推定
competitive_advantage_axes:
  - axis: "営業網"
    multiplier: 4
    detail: "競合10拠点 → リクルート全国47都道府県"
    source: "リクルート会社概要（営業拠点数）"

# パターン3: ブランド認知率から算出
competitive_advantage_axes:
  - axis: "ブランド認知"
    multiplier: 8
    detail: "競合認知率10% → ゼクシィ認知率85%"
    source: "電通ブランド調査 2000"
```

**推定不可の場合**:
```yaml
competitive_advantage_axes: []
# 不明（要追加調査）: 競合データ不足、優位性の定量化困難
```

**YAMLコメント形式**:

```yaml
# 根拠: [根拠となる情報]
```

## 6. よくある質問（FAQ）

### Q1: user_research_countが全く見つからない場合は？

A: `0` を記載し、コメントで理由を明記してください：
```yaml
user_research_count: 0
# 情報源なし、M&A後の統合製品のためリクルート独自調査は未実施と推測
```

### Q2: market_need_percentageが推定困難な場合は？

A: 業界標準の50%を使用し、保守的推定である旨を記載：
```yaml
market_need_percentage: 50
# 保守的推定: 定性情報のみ、業界標準を適用
```

### Q3: competitive_advantage_axesが2軸未満の場合は？

A: リクルート資産活用を追加軸として検討してください：
```yaml
competitive_advantage_axes:
  - axis: "コスト"
    multiplier: 5
    detail: "競合月額5,000円 → 自社1,000円"
  - axis: "営業網（リクルート資産）"
    multiplier: 4
    detail: "競合オンライン販売のみ → 全国営業拠点サポート"
```

どうしても2軸未満の場合は、コメントで理由を明記：
```yaml
competitive_advantage_axes:
  - axis: "コスト"
    multiplier: 3
    detail: "競合月額3,000円 → 自社1,000円"
# 注記: 他軸の優位性は確認できず、要追加調査
```

### Q4: 複数のソースで数値が異なる場合は？

A: 最も信頼性の高いソース（Tier 1-2）を採用し、他のソースもprimary_sourcesに記載：
```yaml
users: 1000000
# ソース: IR資料「100万人突破」2019-12（Tier 2）
# 他のソースでは"90万人"との記載もあり（日経 2019-11）
```

### Q5: 日本語ソースしかない場合は？

A: 日本語ソースでも可。IRやメディアインタビューを活用：
```yaml
primary_sources:
  - "[Tier 1] Airレジ公式サイト https://airregi.jp/"
  - "[Tier 2] リクルートライフスタイル IR資料 2014年度"
  - "[Tier 3] 日経新聞「リクルート、無料POSレジ参入」2013-11-20"
```

### Q6: M&A製品の場合、創業時とM&A後のどちらを記載？

A: 両方記載してください：
```yaml
# M&A前（創業時）
validation_data:
  cpf:
    user_research_count: 100
    # ソース: 創業者Paul Forster氏のインタビュー

# M&A後（リクルート統合後）
recruit_specific:
  leveraged_assets:
    - asset_type: "グローバル展開"
      description: "リクルートの日本・アジア展開ノウハウを活用"
```

### Q7: 撤退製品の情報がほとんど見つからない場合は？

A: Wayback Machineを活用し、最低限の情報を記録：
```yaml
# 例: CORP_F010_unknown_service
product:
  name: "UnknownService"
  launched_year: 2005
  # ソース: Wayback Machine（公式サイトキャプチャ 2005-06）
  current_status: "shutdown"
  # ソース: リクルートIR資料「事業整理」2008-03

withdrawal:
  occurred: true
  withdrawal_year: 2008
  duration_months: 36
  reason: "不明（IR資料に「事業整理」のみ記載）"

user_research_count: 0
# 情報源なし、Wayback Machineにも詳細なし

market_need_percentage: 50
# 保守的推定: 詳細不明、業界標準を適用

competitive_advantage_axes: []
# 不明（要追加調査）: 情報不足
```

### Q8: orchestrate-phase1との対応関係は？

A: 以下のマッピングで対応：

| Corporate_Product_Research | Founder_Research | orchestrate-phase1 |
|---------------------------|------------------|-------------------|
| user_research_count | interview_count | /validate-cpf インタビュー数 |
| market_need_percentage | problem_commonality | /discover-demand 市場ニーズ |
| competitive_advantage_axes | ten_x_axes | /validate-10x 10倍優位性 |

大企業製品は2-5倍でもOKですが、創業者型は10倍を目指します。

### 7.2 調査順序の最適化

**効率的な調査順序**:

1. **成功事例 → 失敗事例**

   - 失敗事例で撤退基準を検証

2. **大規模製品 → 小規模製品**
   - 大規模製品で情報源を収集
   - 小規模製品で転用

3. **同一事業部内の製品をまとめて調査**
   - リクルートライフスタイル製品（Air、ホットペッパー）
   - リクルートジョブズ製品（タウンワーク、フロム・エー）
   - リクルートマーケティングパートナーズ製品（スタディサプリ、ゼクシィ）

**調査時間の目安**:
- 1件目: 90分（フルリサーチ）
- 2件目: 60分（共通ソース活用）
- 3件目以降: 45分（テンプレート再利用）

## 9. 撤退製品の特別ルール

### 9.1 撤退基準の検証（リクルート3年5年ルール）

**3年単月黒字ルール**:
- 創業から3年以内に単月黒字達成が必須
- 未達成の場合、撤退検討開始

**5年累損解消ルール**:
- 創業から5年以内に累積損失の解消が必須
- 未達成の場合、撤退確定

**検証項目**:
- [ ] 創業年月の特定
- [ ] 3年経過時点の財務状態（IR資料から推定）
- [ ] 5年経過時点の財務状態（IR資料から推定）
- [ ] 撤退年月の特定
- [ ] 撤退理由の分析

**記載例**:
```yaml
withdrawal:
  occurred: true
  withdrawal_year: 2015
  duration_months: 48  # 2011-01 〜 2015-01 = 48ヶ月
  reason: "3年単月黒字未達成 + 5年累損解消未達成"
  three_year_profitability: false  # 2014-01時点で赤字継続
  five_year_cumulative_loss: false  # 2016-01到達前に撤退判断
  final_status: "完全撤退"
```

### 9.2 撤退製品の情報収集戦略

**Wayback Machineの活用**:
1. 公式サイトのキャプチャを時系列で確認
2. サービス内容、価格、機能の変遷を記録
3. 最終更新日を特定（撤退時期の推定）

**IR資料の活用**:
1. 決算説明会資料で「事業整理」「構造改革」を検索
2. セグメント別業績で赤字事業を特定
3. 質疑応答で撤退理由のコメントを確認

**メディア報道の活用**:
1. 日経テレコンで撤退ニュースを検索
2. TechCrunchで製品レビュー・批評を確認
3. 個人ブログで実際のユーザー体験を収集

### 9.3 撤退製品の品質基準

撤退製品は情報が少ないため、最低基準を明確化：
