## 3. ピボット/失敗経験

### 3.1 初期の失敗
- 金券/前払いは、返金・キャンセル・不正対応が重く、運用コストが増えやすい
- 店舗オペ（消込/照合/レジ連携）が現場負担となり、定着が難しい
- 予約プラットフォームの強み（送客）と、決済の強み（必ず消込）が同時に必要

### 3.2 ピボット（該当する場合）
- 元のアイデア: 『予約』を全て前払いへ寄せる
- ピボット後: 主要シーン（宴会/幹事）に限定し、消込と返金を標準化
- きっかけ: CS負荷の増加、加盟/運用の定着率の頭打ち

### 3.3 リクルート撤退基準の検証（失敗事例のみ）

**撤退判断のタイムライン（公開情報＋ケーススタディ再構成）**:
- 提供期間: 2010年代〜2025年（推定）
- 3年経過時点: 単月黒字達成は限定的（決済/CSコストが重い）
- 5年経過時点: 累損解消は難航（運用定着が課題）
- 最終判断: 2025年に機能提供終了（告知あり）

**撤退理由の分析**:
| 要因 | 詳細 |
|------|------|
| 市場要因 | 予約/決済/ポイントが複数サービスに分散し、標準化が進む |
| 競合要因 | 大手決済・予約サービスがセットで提供し、差別化が難化 |
| 内部要因 | 返金/不正/照合などのオペ負荷が、広告モデルの採算構造と衝突 |
| リクルート基準 | 3年単月黒字未達成 / 5年累損解消未達成（推定） |

**撤退後の処理**:
- 機能提供終了（告知）
- ユーザー/店舗向けにFAQと移行導線を提示

## 4. 成長戦略

### 4.1 初期トラクション
- 予約に強いホットペッパー上でチケット導線を露出し、獲得コストを抑える
- 幹事ユースケースに限定して、価値を明確化（集金/当日会計の削減）

### 4.2 フライホイール
- 予約→購入→来店が増えるほど、店舗側の運用が慣れて消込が安定
- 返金/不正の学習が進むほど、ルールが改善されクレームが減る

### 4.3 スケール戦略
- ルール（返金/キャンセル/消込）をテンプレ化し、加盟店運用を標準化
- 決済/台帳連携を強化し、現場オペを減らす

### 4.4 リクルート資産の活用

| 資産タイプ | 具体的な活用方法 | 効果 |
|----------|---------------|------|
| プラットフォーム | ホットペッパーグルメの予約導線に『チケット』を追加し、来店前の購入を設計 | 集客・継続を下支え |
| ポイント | リクルートポイント（のちのPonta等）で購入/還元を促進 | 集客・継続を下支え |
| データベース | 予約・来店・決済・キャンセルを統合し、不正/返金ルールを設計 | 集客・継続を下支え |
| 営業網 | 飲食店営業/CSで加盟・運用を支援（ただし運用負荷が高止まり） | 集客・継続を下支え |

| 既存事業 | シナジータイプ | 具体的な連携 |
|---------|-------------|------------|
| ホットペッパーグルメ | 導線統合 | 検索→予約→チケット購入→来店の一気通貫 |
| Restaurant Board | 運用連携 | 店舗側の運用（予約/台帳/消込）と連携 |
| リクルートポイント | リテンション | ポイントで再来店・再購入を促す |

## 7. 成功/失敗要因分析

### 7.1 主要失敗要因（撤退要因）
1. 金券・決済はオペ/法務/CSの負荷が高く、薄利になりやすい
2. 店舗側の消込運用がボトルネックになり、スケールの再現性が落ちる
3. 決済/予約の統合競争が進むと、単機能での差別化が難化

### 7.2 失敗要因（失敗時）
| フェーズ | 失敗要因 | 具体的内容 |
|---------|---------|----------|
| CPF | 店舗オペの重さ | 消込/照合が現場負担、加盟の伸びが鈍化 |
| PSF | 維持できない優位性 | 10x体験があっても継続運用が難しい |
| PMF | 単価/頻度の限界 | 購入頻度が低く、LTVが伸びにくい |
| 戦略 | 統合競争 | 決済・予約がセット化し、単体機能の価値が薄れる |

### 付録: 返金・キャンセル・不正のケース分類（例）
- CASE-01: 予約前にチケット購入（導線逸脱）→ 返金/再購入導線が必要
- CASE-02: 来店前日に人数変更 → 追加購入/一部返金のUIが必要
- CASE-03: 当日遅刻でコース変更 → 店舗側の消込額調整が必要
- CASE-04: 無断キャンセル → 返金不可/手数料控除などの規約設計が必要
- CASE-05: 期限切れ → ポリシー（延長可否）と告知が必要
- CASE-06: 代理購入（家族/同僚）→ 受取/利用者名寄せが必要
- CASE-07: 二重購入 → 片方を自動返金できるか
- CASE-08: 決済失敗（与信のみ）→ 二重計上防止の整合性
- CASE-09: 店舗側で消込漏れ → CS負荷増、自己申告フローが必要
- CASE-10: 不正利用（スクショ/転売）→ 表示/利用方法の制御が必要
- CASE-11: 店舗が閉店 → 返金と代替店舗案内が必要
- CASE-12: 予約が重複 → 予約台帳との整合が必要

- CASE-16: 法人/領収書 → インボイス/明細の整備
- CASE-17: 端末故障（店舗）→ 代替消込手段
- CASE-18: ネットワーク障害 → オフライン運用
- CASE-19: コース内容変更（価格改定）→ 既購入分の扱い
