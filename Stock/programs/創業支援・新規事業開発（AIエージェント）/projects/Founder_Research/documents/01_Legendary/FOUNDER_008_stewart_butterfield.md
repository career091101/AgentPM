# Stewart Butterfield - Flickr/Slack

## 1. 基本情報

| 項目 | 内容 |

| 創業者 | Stewart Butterfield（出生名: Dharma Jeremy Butterfield） |
| 生年 | 1973年3月21日 |

| 創業前経験 | Ludicorp創業（Flickr）、Yahoo勤務 |
| 企業名 | Tiny Speck → Slack Technologies |

## 2. 創業ストーリー

### 2.1 生い立ちと背景

Stewart Butterfieldは1973年、カナダのブリティッシュコロンビア州ランドで生まれた。両親はベトナム戦争の徴兵を逃れてカナダに移住したアメリカ人ヒッピーだった。

**幼少期の特異な環境**:
- 電気・水道・電話のない19世紀に建てられたログキャビンで生活
- ヒッピーコミューンでの自給自足生活
- 4歳でビクトリアに移住し、現代的な生活へ
- 12歳で「Dharma」から「Stewart」に改名

**教育**:
- St. Michaels University School（ビクトリア）
- ビクトリア大学で哲学学士号取得（1996年）
- ケンブリッジ大学クレア・カレッジで哲学修士号取得（1998年）
- 大学時代にウェブサイト制作で収入を得る

### 2.2 第1回ピボット: Game Neverending → Flickr（2002-2005）

**Ludicorp創業（2002年夏）**:
- 共同創業者: Caterina Fake、Jason Classon
- 場所: カナダ・バンクーバー
- 目標: Game Neverendingというマルチプレイヤーオンラインゲームの開発

**Game Neverendingの特徴**:
- ソーシャルインタラクション重視のロールプレイングゲーム
- プレイヤー同士がオンラインで協力して世界を構築
- Minecraftの先駆け的なコンセプト

**ピボットの経緯**:
1. 資金調達に苦戦（VCからの投資を得られず）
2. 自己資金・友人家族からの資金も枯渇
3. ゲーム内の写真共有機能が予想外に人気
4. Butterfieldがインフルエンザで寝込んでいる際に決断
5. 社内投票は最初同数で膠着 → 1人のコーダーを説得して決着

**Flickrの成功**:
- 2004年2月: Flickr正式ローンチ
- 3ヶ月で成長の兆候が明確に
- 2005年3月: Yahooが買収（推定$35M）
- ローンチから買収まで約13ヶ月

**学び**:
- 副産物が本業を超える可能性
- ユーザーが実際に使っている機能に注目
- 困難な決断でも前に進む勇気

### 2.3 第2回ピボット: Glitch → Slack（2009-2013）

**Tiny Speck創業（2009年）**:
- 共同創業者: Cal Henderson、Eric Costello、Serguei Mourachov
- 全員がFlickr時代のチームメイト
- Butterfieldは再びゲーム開発に挑戦

**Glitchの開発と失敗**:
- 2011年9月27日: Glitch正式ローンチ
- 特徴: ブラウザベースのMMOゲーム、非暴力的、協力型
- 問題: 「ほとんどの人にとって奇妙すぎた」
- データ: 登録者の97%が5分以内に離脱
- 残り3%は熱狂的（週20時間プレイ）だが市場として小さすぎ
- 2012年12月9日: ゲーム終了を発表

**Slackの誕生**:
- Glitch開発中にチーム内コミュニケーションツールを自作
- 既存ツール（メール、MSN Messenger等）では非効率だったため
- 3.5年間、自分たちの「市場1社」のために最適化
- ゲーム終了後、このツールにポテンシャルを見出す

**ピボットの決断**:
- 2012年末〜2013年初頭: 投資家にピボットを報告
- ゲーム会社からエンタープライズソフトウェア会社へ
- Tiny SpeckをSlackに社名変更

## 3. CPF検証（Customer Problem Fit）

### 3.1 課題の発見

**自分たちが直面した課題**:
- メールは最も非効率なメッセージ手段で、認知負荷が高い
- 新入社員が空のメールボックスから始める非効率
- チーム間のコミュニケーションが分断
- 情報の検索性が低い

**3U検証**:
- **Unworkable**: メールではチームコラボレーションに限界
- **Unavoidable**: 組織が成長するほどコミュニケーション問題は深刻化
- **Urgent**: リモートワーク・分散チームの増加

### 3.2 検証方法

**自社利用による検証（2009-2012）**:
- 3.5年間、Glitch開発チームで日常的に使用
- 「1社市場」でプロダクト・マーケット・フィットを達成
- チームが他の会社に移動すると、その非効率さに気づく

**ベータテスト（2013年）**:
- 公開プレビュー前に友人の会社に依頼
- Rdio、TGC、Medium、Cozy等が初期ベータユーザー
- フィードバック収集と改善の反復

**「プレビュー」という名称戦略**:
- 「ベータ」ではなく「プレビュー」と命名
- ベータ: テスターとして協力
- プレビュー: 特権的な早期アクセス者として体験
- 心理的ポジショニングの違いを活用

### 3.3 支払い意思（WTP）の確認

- ローンチ11日で$1M収益達成
- 30%という高いフリーミアム→有料転換率
- 企業が自発的にアップグレード

## 4. PSF検証（Problem Solution Fit）

### 4.1 10倍優位性

| 軸 | 従来の解決策（メール） | Slackソリューション | 倍率 |
|---|---------------------|-------------------|------|
| 検索性 | メール検索は煩雑 | 全会話・知識の検索可能ログ | 10倍 |
| 透明性 | 個人の受信箱に閉じ込められる | チャンネルで組織知識を共有 | 5倍 |
| オンボーディング | 新人は空のメールボックスから | 全組織知識にアクセス可能 | 10倍 |
| リアルタイム性 | 非同期、遅延 | 即座のコミュニケーション | 5倍 |
| 統合性 | ツールが分断 | 100+のアプリ統合 | 10倍 |

### 4.2 MVP戦略

**内部ツールからの進化**:
- 最初から製品として設計したわけではない
- 3.5年間の実使用でプロダクトを磨く
- ゲーム開発者による高いUX基準

**Slackの名前の由来**:
- Searchable Log of All Conversation and Knowledge
- 「すべての会話と知識の検索可能なログ」

### 4.3 UVP（独自の価値提案）

**Butterfieldの製品哲学**:
- 「企業は生産性向上ソフトを買わない」
- 「人々が愛するプロダクトを作れば、生産性は後からついてくる」
- ゲーム開発者の視点でUXを重視

**差別化要因**:
1. 消費者向けアプリのような使いやすさ
2. 楽しさ・遊び心のあるデザイン
3. フリーミアムによる低い導入障壁
4. ボトムアップ導入（IT部門の許可不要）

## 5. 成長戦略

### 5.1 初期トラクション

**ローンチ戦略**:
- 2013年8月: プレビュー版発表
- 24時間で8,000人登録
- 2週間で15,000人
- 2014年2月: 正式公開ローンチ
- 週5-10%の成長率

**驚異的な成長**:
| 時期 | DAU | 備考 |
|------|-----|------|
| 2013年8月 | 8,000 | プレビュー開始 |
| 2014年2月 | 15,000 | 正式ローンチ |
| 2014年8月 | 120,000+ | ローンチ6ヶ月 |
| 2014年10月 | 285,000 | ユニコーン達成 |
| 2015年6月 | 1,100,000 | 100万DAU突破 |

### 5.2 ワード・オブ・マウス戦略

**マーケティング費用ゼロの成長**:
- 「成長は完全に狂気的で、ほぼ完全にワード・オブ・マウス」
- 最初のマーケティング担当者は$1B評価額直前に採用
- CMO不在で$1.1B評価額達成
- アウトバウンド営業チームは2016年まで不在

**ボトムアップ導入**:
1. 1人のチームメンバーが試用
2. 同僚を招待
3. 部署全体に広がる
4. 会社全体が採用
5. 外部コラボレーター（クライアント等）にも拡大

### 5.3 フリーミアム戦略

**モデル設計**:
- 無料プラン: 無制限ユーザー、制限付き機能
- 有料プラン: 高度な機能、無制限履歴
- 転換率: 30%（フリーミアム製品として非常に高い）

**ネットワーク効果**:
- 各ユーザー追加がプラットフォーム価値を向上
- 社内バイラル: 同僚招待
- 社外バイラル: クライアント・パートナー招待
- パワーユーザーが意思決定者となり有料化を推進

### 5.4 フライホイール

```
優れたUX → ユーザー満足 → ワード・オブ・マウス → 新規ユーザー獲得
    ↑                                                    ↓
コスト効率                                          フリーミアム導入
    ↑                                                    ↓
収益増加 ← 有料転換 ← チーム全体での利用 ← 社内バイラル
```

## 8. 成功要因分析

### 8.1 主要成功要因

1. **2度のピボット経験**: 失敗から学び、副産物を本業に転換する能力
2. **自分たちの課題を解決**: 3.5年間の自社利用でプロダクトを磨く
3. **ゲーム開発者のUX感覚**: エンタープライズソフトでも消費者レベルのUX
4. **ワード・オブ・マウス重視**: マーケティング費用なしでの成長
5. **ボトムアップ導入**: IT部門を迂回した草の根的普及

### 8.2 タイミング要因

- クラウドコンピューティングの普及
- モバイルデバイスの浸透
- リモートワークの増加傾向
- メールへの不満の蓄積
- SaaS・サブスクリプションモデルの受容

### 8.3 ピボットの教訓

**Butterfieldの2回のピボットパターン**:
1. ゲームを作ろうとする
2. ゲームのために内部ツールを作る
3. ゲームは失敗する
4. 内部ツールが大成功する

**ピボットを成功させる条件**:
- 既存資産（チーム、技術、ツール）を活用
- 実際に使われている部分に注目
- 迅速な決断と実行
- チームの説得と巻き込み
