## 2. 創業ストーリー

### 2.1 課題発見（需要発見）

**開始：個人のペイン（2011年）**

GitLab の起源は、ウクライナの開発者 Dmitriy Zaporozhets の個人的なペインに遡る。当時、Dmitriy は以下の課題に直面していた：

- GitHub は優れたプラットフォームだが、**自社サーバーでホストできない**（GitHub は SaaS オンリー）
- 企業によっては、コードを外部（GitHub）に置くことができない（セキュリティ要件）
- 既存の Git ホスティング solution は操作性が悪い

**問題：最初は、実は深い市場課題ではなかった**

ただし、Dmitriy は自分の勤務先 Sphere Software で、「実は多くの開発チームが GitHub の代替を欲している」ことを発見した。とくに：

- 欧州の企業（特にドイツ、フランス）は データコンプライアンス理由で GitHub を使えない
- 大手企業では、内部ホスティング Git が必須
- しかし既存の内部ホスティング option（GitLite, Gitorious）は操作性・機能が不十分

**初期仮説**:
"I want to build a Git hosting tool I would use myself"

### 2.2 CPF検証（Customer Problem Fit）

**オープンソースプロジェクトとしての検証（2011-2012）**

Dmitriy は、GitLab を **オープンソースプロジェクト**として 2011年に開始した。自分が使いたいツール = 他の開発者も欲しいはずという想定。

**インタビュー・顧客検証**:

- **実施方法**: GitHub issue, pull request, community feedback によるオープンソース方式の検証
- **実施数**: 推定50社以上（コミュニティ規模）
- **手法**:
  - オープンソース利用者からの feature request
  - エンタープライズユーザーからのセルフホスティング要望
  - European compliance market からのニーズ
- **発見した課題の共通点**:
  - GitHub への依存を避けたい（Compliance, Sovereignty 理由）
  - 内部ホスティング Git の操作性向上
  - CI/CD integration が必要（単なる Git repository tool では不十分）

**3U検証**:

- **Unworkable（現状では解決不可能）**: GitHub は SaaS only、内部ホスティング option は存在しない or 劣悪
- **Unavoidable（避けられない）**: 企業の Git version control は必須、compliance requirement により GitHub は使用不可
- **Urgent（緊急性が高い）**: DevOps automation 波が加速中、Git + CI/CD integration は業界急速成長領域

**支払い意思（WTP）**:

- **確認方法**: GitLab.com（SaaS版）の launch により、アップセルテスト（オープンソース → paid tier）
- **結果**:
  - GitLab.com でのプリペイド conversion rate は positive（paid tier への upgrade）
  - 早期顧客は Premium, Ultimate tier への upgrade意欲を示す
  - WTP confirmed（複数通貨での支払い実績）

### 2.3 PSF検証（Problem Solution Fit）

**ピボット：オープンソース → SaaS + オープンコア戦略（2012-2013）**

当初、GitLab はホスト版（GitLab.com）を計画していなかった。ただし、18ヶ月の観察期間を経て、Dmitriy と Sid は以下を学んだ：

- オープンソースで「十分な」ユーザー獲得が可能
- SaaS版（GitLab.com）を launch すれば、ホストされたバージョンに対する WTP が存在
- **Open Core Model** が DevOps 業界で viable であることを発見

**10倍優位性**

| 軸 | 従来の解決策 | GitLab | 倍率 |
|---|----------|--------|------|
| **統合性** | Jenkins + GitHub + Slack + Jira + Docker Registry = 6+ tools | GitLab single app（SCM + CI/CD + CD + Registry + Security）| **10倍** |
| **デプロイ時間** | Manual integration, tool switching | GitLab CI/CD automation | **5倍** |
| **操作性** | 複雑なセットアップ | Unified UI, one-click deployment | **4倍** |
| **ホスティング選択肢** | SaaS only (GitHub) or deprecated self-hosted | SaaS (GitLab.com) + Self-hosted + Open Source | **3倍** |
| **コスト** | Jenkins (free) + GitHub Enterprise ($100+/month) + other tools | GitLab (free open source + premium tiers) | **2倍** |

**MVP & Key Decisions**

**Initial MVP（2011-2012）**:
- Type: **Open Source Foundation**
- 機能: Git repository hosting + basic project management
- ホスティング: Self-hosted only (no SaaS)

**Critical Pivot Decision（2012-2013）**:

GitLab の重要な決定は、**SCM（Source Control Management）と CI（Continuous Integration）の統合**である。

背景：
- Kamil Trzciński というコミュニティコントリビューターが、"You should merge SCM and CI into one application" と強く主張
- 当初、Sid と Dmitriy は反対（責任範囲の拡大、開発複雑性）
- しかし、市場の demand と Kamil の logic が説得的だったため、統合を決定

結果：
- GitLab が「first single application for the entire DevOps lifecycle」になった
- Jenkins + GitHub の separate tools の時代を終わらせる innovative positioning

**UVP（独自の価値提案）**

"The single application that brings together development, operations, IT, security, and business teams to deliver desired business outcomes."

**競合との差別化**

1. **統合の完全性**: GitHub（SCM only） vs GitLab（SCM + CI/CD + CD + Registry + Security + Analytics）
2. **オープンコア戦略**: GitHub は closed-source のみ；GitLab は open source + commercial tiers
3. **ホスティング選択肢**: SaaS + Self-hosted + Open source の3選択肢
4. **All-Remote Culture**: グローバル採用、時差を活かした開発効率（実は大きな競合優位）

## 3. ピボット/失敗経験

### 3.1 初期の限界：単なるGit Repository Tool から DevOps Platform へ

**元のアイデア**: Self-hosted Git repository hosting tool（簡易版GitHub）
- Target: 企業の internal Git server requirement
- Business Model: Free open source

**限界（2012-2013）**:
- Git repository tool だけでは、競合優位が弱い（GitLite, Gitorious が存在）
- DevOps 市場は、「Git だけ」を欲しているのではなく、「Build, Test, Deploy integration」を欲していた
- ARR model を構築するには、より広範な機能セットが必要

**ピボット後**: DevOps Platform（SCM + CI/CD + Deployment + Monitoring統合）
- Target: DevOps teams wanting single-pane-of-glass solution
- Business Model: Open source (free) + SaaS (GitLab.com) + Premium tier

**学び**:
- 市場ニーズは、"individual tool" ではなく、"integrated workflow"
- Kamil Trzciński の提案（SCM + CI統合）を受け入れた判断が、GitLab の trajectory を大きく変えた

### 3.2 オープンコア vs 完全オープンソース

**議論背景**:
- 初期段階では、GitLab は完全オープンソース（free）として発展
- SaaS版（GitLab.com）の launch を検討する際、monetization method が課題

**決定**:
- **Open Core Strategy** を採用：Core features は open source（永遠に無料）、Premium features は commercial tiers
- This approach は trust を build（オープンソースコミュニティとの関係維持）

**結果**:
- オープンソースユーザーから SaaS/Enterprise customer への natural funnel
- GitHub と異なる、Community-friendly な positioning

## 6. 成功要因分析

### 6.1 主要成功要因

1. **統合プラットフォーム戦略**
   - Jenkins（CI） + GitHub（SCM）の分立体制を単一プラットフォームに統合
   - DevOps workflow の複雑性の削減
   - スイッチングコスト上昇により顧客ロック・イン強化

2. **Open Core Model の採用**
   - オープンソース（無料）core + 有料Premium/Ultimate tiers
   - Developer community との trust構築
   - Organic funnel：free → trial → paid customer

3. **オープンソース採用による市場渗透**
   - GitHub star 40,000+ を achieved（by 2020）
   - Developer mindshare での GitHub に次ぐ positioning
   - 低 CAC（Customer Acquisition Cost）：大部分はオーガニック

4. **All-Remote Culture が生み出す競合優位**
   - Timezone constraint がない global team → 24/7 development velocity
   - Cost efficiency → growth investment への集中
   - Handbook-first documentation → async-first communication
   - DevOps 市場での cultural alignment（リモートワークを活用する team向け）

5. **デジタル化の加速波（COVID-19）**
   - 2020-2021年の DevOps / cloud-native adoption 加速
   - GitLab の all-remote model が「post-pandemic work」の象徴に
   - IPO timing の excellence

6. **Investor Confidence & Narrative**
   - "From Ukraine with all-remote" = compelling founder story
   - European compliance market で natural advantage
   - DevOps market growth（80%+ CAGR）との alignment

### 6.2 タイミング要因

1. **DevOps Movement の急速普及（2012-2020）**
   - Infrastructure-as-code, containerization revolution
   - Cloud adoption acceleration
   - GitOps methodology の emergence

2. **GitHub（Microsoft）の acquihire化**
   - GitHub がMicrosoft傘下に入った後、alternative として GitLab が浮上
   - Compliance-conscious enterprises での adoption加速

3. **COVID-19 Pandemic（2020-2021）**
   - Remote work への forced transition
   - DevOps tool への demand surge
   - IPO市場での SaaS 企業高評価

4. **IPO Market Conditions（2021年10月）**
   - SaaS企業への高い valuation
   - Institutional investor からの tech sector への関心
   - GitLab IPO timing の excellence

### 6.3 差別化要因

1. **統合の深さ**: 業界初の "single application for entire DevOps lifecycle"
2. **コミュニティ力**: 40,000+ GitHub stars でのオープンソース地位
3. **カルチャー**: All-remote が「future of work」と alignment
4. **Compliance**: European GDPR considerations との natural fit
5. **Pricing Transparency**: オープンな pricing model（GitHub との contrast）
