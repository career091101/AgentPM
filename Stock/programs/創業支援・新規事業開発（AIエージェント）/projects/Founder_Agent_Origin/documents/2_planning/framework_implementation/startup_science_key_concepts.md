### 1. アイデア検証（3フレームワーク）

#### 1.1 3U+1の法則

**目的**: アイデアが解決する課題の質を評価

**実践手法**:
1. Unworkable（切実さ）チェック: 「今この課題を解決しないと何が起こるか?」質問
2. Unavoidable（不可避性）チェック: 「ターゲットの何%がこの課題に直面するか?」調査
3. Urgent（緊急性）チェック: 「今日解決したいか、来年でいいか?」確認
4. Underserved（代替なし）チェック: 「既存代替品の問題点は何か?」分析

**メトリクス**:
- 課題遭遇率: ターゲットの80%以上が課題に直面
- 緊急度スコア: 「すぐ解決したい」が70%以上
- 代替品満足度: 既存代替品のNPS < 30

**テンプレート**:
```markdown
## 3U+1検証結果

| 要素 | チェック内容 | 結果 | スコア |
|-----|------------|------|-------|
| Unworkable | [切実さの確認] | [結果] | ○/△/× |
| Unavoidable | [不可避性の確認] | [結果] | ○/△/× |
| Urgent | [緊急性の確認] | [結果] | ○/△/× |
| Underserved | [代替なしの確認] | [結果] | ○/△/× |

**総合判定**: [4要素すべて○なら合格]
```

#### 1.2 市場規模検証

**目的**: 市場が十分大きいか確認（最低$1B）

**実践手法**:
1. TAM（Total Addressable Market）計算: 理論上の最大市場規模
2. SAM（Serviceable Available Market）計算: 実際にサービス可能な市場
3. SOM（Serviceable Obtainable Market）計算: 現実的に獲得できる市場
4. トップダウン分析: 既存市場データから推計
5. ボトムアップ分析: 顧客数×単価で推計

**メトリクス**:
- TAM: $1B以上（最低基準）
- SAM: TAMの10-30%
- SOM: SAMの1-5%（初期）
- 市場成長率: 年10%以上

**テンプレート**:
```markdown
## 市場規模推定

| 指標 | 計算式 | 推定値 | 判定 |
|-----|-------|-------|------|
| TAM | [理論上の最大市場] | $XX億 | ○/× |
| SAM | TAM × [到達可能率] | $XX億 | - |
| SOM | SAM × [獲得可能率] | $XX億 | - |
| 市場成長率 | [CAGR] | XX% | ○/× |
```

#### 1.3 Founder-Problem Fit

**目的**: 創業者自身が課題を深く理解しているか確認

**実践手法**:
1. 創業者の原体験ヒアリング: なぜこの課題に取り組むか
2. ドメイン知識チェック: 業界経験年数、専門性
3. 課題への情熱測定: 10年続けられるか
4. 独自インサイト確認: 他の人が気づいていないことを知っているか

**メトリクス**:
- ドメイン経験: 3年以上
- 課題との接触頻度: 週1回以上
- 情熱スコア: 10年続ける意思（Yes/No）

**テンプレート**:
```markdown
## Founder-Problem Fit評価

| 項目 | 内容 | スコア |
|-----|------|-------|
| 原体験 | [なぜこの課題に取り組むか] | ○/△/× |
| ドメイン知識 | [業界経験: XX年] | ○/△/× |
| 課題への情熱 | [10年続けられるか] | Yes/No |
| 独自インサイト | [他の人が知らないこと] | ○/△/× |
```

### 2. CPF（Customer Problem Fit）（5フレームワーク）

#### 2.1 課題仮説キャンバス

**目的**: 課題・顧客・代替品を構造化

**実践手法**:
1. ターゲット顧客の具体化: ペルソナ作成
2. 課題の明確化: 顧客が抱える主要課題
3. 既存代替品の洗い出し: 顧客が現在使っている解決策
4. 既存代替品の問題点分析: なぜ不満があるか
5. 検証方法の設計: インタビュー計画

**メトリクス**:
- ペルソナ具体性: 名前・年齢・職業・日常が記載されているか
- 代替品リスト: 3個以上
- 問題点の深刻度: スコア1-10（7以上が目標）

**テンプレート**:
```markdown
## 課題仮説キャンバス

| 項目 | 内容 |
|-----|------|
| ターゲット顧客 | [具体的なペルソナ] |
| 課題 | [顧客が抱える主要な課題] |
| 既存代替品 | [顧客が現在使っている解決策1, 2, 3...] |
| 既存代替品の問題点 | [なぜ不満があるか] |
| 検証方法 | [インタビュー計画] |
```

#### 2.2 顧客インタビュー手法

**目的**: 顧客の本音を引き出す

**実践手法**:
1. インタビュー設計: 質問リスト作成（良い質問/悪い質問の区別）
2. リクルーティング: ターゲット顧客の募集
3. インタビュー実施: 1人30-45分、対面/オンライン
4. 記録: ノート・録音（許可を得て）
5. 分析: パターン認識、70%一致ルール

**メトリクス**:
- インタビュー実施数: 最低15-20人、理想30人以上
- パターン一致率: 同じ回答が70%以上
- インタビュー時間: 30-45分/人

**良い質問の例**:
- 「〇〇について、普段どうされていますか?」（現状把握）
- 「一番困っていることは何ですか?」（課題発見）
- 「もしこの課題が解決されたら、いくら払いますか?」（解決意欲）
- 「今はどうやって解決していますか?」（既存代替）
- 「この課題にどのくらいの頻度で直面しますか?」（頻度確認）

**悪い質問の例（避けるべき）**:
- ❌ 「こういうプロダクトがあったら使いますか?」（誘導的）
- ❌ 「いくらなら買いますか?」（架空の状況）
- ❌ 「便利だと思いますか?」（抽象的）

**テンプレート**:
```markdown
## インタビュー計画

| 項目 | 内容 |
|-----|------|
| ターゲット | [ペルソナ] |
| 募集方法 | [リクルーティング手段] |
| 実施予定数 | [XX人（最低20人）] |
| 質問リスト | [5-10個の質問] |
| 実施期間 | [YYYY-MM-DD ~ YYYY-MM-DD] |
```

#### 2.3 ペルソナ作成

**目的**: ターゲット顧客を具体化

**実践手法**:
1. デモグラフィック情報: 年齢・性別・職業・収入
2. サイコグラフィック情報: 価値観・ライフスタイル
3. 行動パターン: 1日の流れ、ツール使用状況
4. 課題・ペイン: 最も困っていること
5. ゴール: 達成したいこと
6. 現在の解決策: 今使っているツール・方法
7. 不満点: 既存解決策への不満

**メトリクス**:
- ペルソナ数: 1-3個（多すぎない）
- 具体性スコア: 名前・年齢・職業・日常がすべて記載されているか

**テンプレート**:
```markdown
## ペルソナ: [名前]

| 項目 | 内容 |
|-----|------|
| 年齢・性別 | [例: 35歳男性] |
| 職業・役職 | [例: IT企業マネージャー] |
| 日常 | [1日の行動パターン] |
| 課題・ペイン | [最も困っていること] |
| ゴール | [達成したいこと] |
| 現在の解決策 | [今使っているツール・方法] |
| 不満点 | [既存解決策への不満] |
```

#### 2.4 インタビュー数の目安

**目的**: 統計的に有意なサンプル数を確保

**実践手法**:
- 最低: 15-20人
- 理想: 30人以上
- パターンが見えるまで: 同じ回答が70%以上

**メトリクス**:
- 実施数: 20人以上
- パターン一致率: 70%以上

#### 2.5 CPF達成の判断基準

**目的**: CPFが達成されたか客観的に判断

**実践手法**:
以下の条件を「すべて」満たすこと:
- [ ] 20人以上の顧客インタビューを実施した
- [ ] 70%以上の顧客が同じ課題を挙げた
- [ ] 顧客が「お金を払ってでも解決したい」と言っている
- [ ] 課題の切実さ（Urgency）が確認できた
- [ ] 既存代替品とその問題点が明確になった

**メトリクス**:
- インタビュー数: ≥20人
- パターン一致率: ≥70%
- 支払い意思: 「お金を払う」と言った人の割合 ≥50%
- Urgencyスコア: 「すぐ解決したい」の割合 ≥70%

2. 解決したい課題の特定

**メトリクス**:

### 4. PMF（Product Market Fit）（7フレームワーク）

#### 4.1 Sean Ellisテスト

**目的**: PMF達成を定量的に測定

**実践手法**:
1. アンケート実施: 「このプロダクトが使えなくなったらどう思いますか?」
2. 選択肢: 「非常に残念」「少し残念」「残念ではない」
3. 分析: 「非常に残念」の割合を計算

**メトリクス**:
- 「非常に残念」の割合: 40%以上でPMF達成

**テンプレート**:
```markdown
## Sean Ellisテスト結果

| 回答 | 人数 | 割合 |
|-----|------|------|
| 非常に残念 | XX人 | XX% |
| 少し残念 | XX人 | XX% |
| 残念ではない | XX人 | XX% |

**判定**: [40%以上なら✅ PMF達成]
```

---

#### 4.2 月次成長率

**目的**: プロダクトが成長しているか確認

**メトリクス**:
- 月次成長率: 10%+/月（PMF達成基準）

**計算式**:
```
月次成長率 = ((今月のユーザー数 - 先月のユーザー数) / 先月のユーザー数) × 100
```

---

#### 4.3 NPS（Net Promoter Score）

**目的**: 顧客満足度を測定

**実践手法**:
1. アンケート: 「このプロダクトを友人に薦める可能性は?」（0-10点）
2. 分類:
   - 9-10点: Promoter（推奨者）
   - 7-8点: Passive（中立者）
   - 0-6点: Detractor（批判者）
3. 計算: NPS = Promoter% - Detractor%

**メトリクス**:
- NPS: 50以上（PMF達成基準）

---

#### 4.4 Churn Rate（解約率）

**目的**: 顧客が離脱していないか確認

**計算式**:
```
Churn Rate = (解約顧客数 / 期初顧客数) × 100
```

**メトリクス**:
- Churn Rate: 5%以下/月（PMF達成基準）

---

#### 4.5 LTV/CAC比率

**目的**: 顧客獲得が経済的に健全か確認

**計算式**:
```
LTV = ARPU × 粗利率 × 平均継続月数
CAC = (マーケティング費用 + 営業費用) / 新規顧客数
LTV/CAC = LTV / CAC
```

**メトリクス**:
- LTV/CAC: 3倍以上（PMF達成基準）

---

#### 4.6 ピボット判断基準

**目的**: ピボットすべきタイミングを見極める

**実践手法**:
以下のサインが出たらピボット検討:
- 顧客インタビューで「いいね」と言われるが購入されない
- 成長率が3ヶ月連続で下がっている
- チームのモチベーションが下がっている
- 資金が6ヶ月以内に尽きる見込み

**メトリクス**:
- 成長率下降期間: 3ヶ月連続で下がったらアラート
- ランウェイ: 6ヶ月以下で危機的

---

#### 4.7 ピボットタイプ分類

**目的**: ピボットの方向性を決定

**実践手法**:
5種類のピボットから選択:

| ピボット種類 | 説明 | 例 |
|------------|------|-----|
| **ズームイン** | 機能の一部に特化 | Instagram（写真共有アプリ→フィルター特化） |
| **ズームアウト** | より大きな課題へ拡大 | YouTube（動画デート→動画共有） |
| **顧客セグメント変更** | 別のターゲットへ | Slack（ゲーム→B2Bチャット） |
| **課題ピボット** | 別の課題を解決 | Twitter（ポッドキャスト→マイクロブログ） |
| **チャネルピボット** | 販売経路を変更 | Airbnb（写真販売→民泊仲介） |

**テンプレート**:
```markdown
## ピボット計画

| 項目 | 内容 |
|-----|------|
| ピボット種類 | [ズームイン/ズームアウト/セグメント/課題/チャネル] |
| 理由 | [なぜピボットするか] |
| 新しいターゲット/課題 | [変更後の内容] |

```

### 7. 戦略（3フレームワーク）

#### 7.1 戦略キャンバス

#### 9.3 ICEスコア

**目的**: 機能の優先順位付け

**計算式**:
```
ICE = Impact × Confidence × Ease
```

| 項目 | 説明 | スケール |
|-----|------|---------|
| Impact | 効果の大きさ | 1-10 |
| Confidence | 確信度 | 1-10 |
| Ease | 実装の容易さ | 1-10 |

**テンプレート**:
```markdown
## ICEスコア評価

| 機能 | Impact | Confidence | Ease | ICEスコア | 優先順位 |
|-----|--------|------------|------|----------|---------|
| [機能A] | 8 | 7 | 5 | 280 | 1 |
| [機能B] | 6 | 9 | 8 | 432 | 2 |
| [機能C] | 9 | 4 | 3 | 108 | 3 |
```

---

#### 9.4 RICEスコア

**計算式**:
```
RICE = Reach × Impact × Confidence / Effort
```
