# プロジェクト憲章: Founder_Agent_ForStartup

## 1. プロジェクト概要

### 1.1 プロジェクト名
Founder_Agent_ForStartup（スタートアップ向け創業エージェント）

### 1.2 プロジェクトの目的
VC資金調達とスケールアップを前提とした起業家支援AIエージェントシステムの構築。Founder Agent Originベースに、急成長（3倍/年）を目指すスタートアップ向けの厳格な検証基準とVC対応機能を追加した特化版を提供する。

### 1.3 ビジネス価値
- スタートアップ創業者のVC調達成功率向上（35点以上のスコアカード達成）
- ピッチデッキ作成時間の90%削減（従来10-20時間 → 1-2時間）
- 創業初期の失敗リスク低減（597件の実証データに基づく失敗パターン回避）
- 資金調達ラウンド設計の最適化（Pre-Seed → Seed → Series A-B）

## 2. プロジェクトの背景

### 2.1 背景
起業の科学フレームワークに基づき、PMF（プロダクトマーケットフィット）達成前の創業者を支援するAIエージェントとして開発された。Founder Agent Originの成果を基に、VC資金調達を前提とするスタートアップ向けに厳格化・特化させる必要性が認識された。

### 2.2 ビジネスニーズ
- **高い失敗率**: 日本のスタートアップの90%以上が5年以内に失敗
- **VC資金調達の困難性**: Seed調達成功率は約5-10%
- **体系的検証手法の不足**: CPF/PSF検証を実施している創業者は10%未満
- **ピッチング準備の負担**: VC対応に月40-80時間を費やすが体系化されていない

### 2.3 Founder_Researchとの関係性

#### プロジェクト構造の現状
現在、以下の2つの構造が存在：

1. **独立プロジェクト**: `/projects/Founder_Research/` （671ファイル、635 Markdown）
2. **サブディレクトリ**: `/projects/Founder_Agent_ForStartup/Founder_Research/` （同一内容）

#### 関係性の整理

| 項目 | Founder_Research（独立） | Founder_Agent_ForStartup |
|------|------------------------|-------------------------|
| **目的** | 世界的起業家597人の実証データ収集 | VC調達特化のエージェントシステム |
| **成果物** | ケーススタディ597件、失敗パターン29種 | AI対話システム、ピッチデッキ自動生成 |
| **役割** | ナレッジベース構築、検証基準の裏付け | エンドユーザー向けサービス提供 |
| **ファイル数** | 671ファイル（md: 635, py: 12, yaml: 1） | 660ファイル（Research含む） |
| **依存関係** | 独立実行可能 | Founder_Researchを参照データとして利用 |

#### 重複と統合検討

**重複の実態**:
- Founder_Agent_ForStartup内のFounder_Researchはシンボリックリンクまたはデータ参照用コピー
- domain_config.yamlに参照パスとして明記: `reference_paths: ["../../Founder_Research/documents/"]`

**統合の可能性**:
- **統合不要（現状維持推奨）**: Founder_Researchは独立したリサーチプロジェクトとして継続し、ForStartupは参照のみ行う
- **理由**:
  - Researchは他のバリアント（ForSolo, ForStartup, ForGenAI）でも共通参照される
  - データ更新と実装の分離により保守性が向上
  - 物理的な重複ではなく論理的な参照関係

**推奨アクション**:
- Founder_Agent_ForStartup内のFounder_Researchがシンボリックリンクであることを確認
- 実体が重複している場合はシンボリックリンクに変更
- domain_config.yamlの参照パスを相対パスで維持

## 3. スコープ

### 3.1 含む（In-Scope）

**Phase 1: コア機能実装（2025-12-30時点で完了済み）**
- Founder Agent Origin基準の実装（CPF/PSF/PMF検証）
- ForStartup特化設定（domain_config.yaml）
- スコアカード評価基準の厳格化（35点/40点以上）

**Phase 2: VC対応機能（実装予定）**
- `/build-pitch-deck`: VCピッチデッキ自動生成（10-15スライド）
- `/prepare-vc-meeting`: VC対応Q&A準備
- `/create-fundraising-plan`: 資金調達ラウンド計画作成

**Phase 3: データ駆動検証**
- Founder_Researchデータ（597件）の参照システム
- 失敗パターン29種の自動警告
- 成功事例からのベストプラクティス抽出

**Phase 4: ユニットエコノミクス強化**
- LTV/CAC比率5.0以上の検証
- Payback期間12ヶ月以内のチェック
- 月次成長率20%目標の設定

### 3.2 含まない（Out-of-Scope）

- 法務・会計サービス

## 9. プロジェクト統合検討

### 9.1 Founder_Researchとの統合可能性

**結論**: **統合不要（独立プロジェクト維持推奨）**

**分析結果**:

1. **目的の違い**
   - Founder_Research: 実証データ収集・分析（リサーチプロジェクト）
   - Founder_Agent_ForStartup: AIエージェント実装（プロダクト開発プロジェクト）

2. **ライフサイクルの違い**
   - Founder_Research: 継続的データ収集（現在597件、目標550件達成済み、今後も拡張）
   - Founder_Agent_ForStartup: 開発→リリース→保守のサイクル

3. **再利用性**
   - Founder_Researchは他のバリアントでも共通利用される
   - 独立プロジェクトとして管理することで保守性・再利用性が向上

### 9.2 統合のメリット・デメリット

**統合した場合のメリット**:
- ディレクトリ構造のシンプル化
- 一元管理による更新の簡略化

**統合した場合のデメリット**:
- リサーチとプロダクト開発の境界が曖昧化
- 他バリアントからの参照が複雑化
- 独立した進捗管理が困難
- データ更新とコード更新の混在による混乱

**現状維持のメリット**:
- 関心の分離（Separation of Concerns）の原則に準拠
- 独立したバージョン管理・リリースサイクル
- 他バリアントとの共有が容易
- データチームとプロダクトチームの並行作業が可能

### 9.3 推奨アクション

**即時対応**:
1. Founder_Agent_ForStartup内のFounder_Researchディレクトリがシンボリックリンクか実体かを確認
2. 実体の場合はシンボリックリンクに変更し、独立プロジェクトを参照する形に統一
3. domain_config.yamlのreference_pathsが正しく設定されていることを確認

**中長期対応**:
1. プログラムレベルのREADME.mdで、プロジェクト間の依存関係を明記
2. Founder_Researchの更新通知システムを実装（webhook等）
3. 定期的な整合性チェックスクリプトの実行（週次）
