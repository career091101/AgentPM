## 2. 創業ストーリー

### 2.1 課題発見(需要発見)

**着想源**:
- Alexandrは高校時代からAI・機械学習に強い興味を持ち、17歳でQuoraにソフトウェアエンジニアとして入社
- QuoraでのML開発経験から「アルゴリズムよりもデータ品質がAIの性能を決める」ことを実感
- MIT入学後、わずか1年で中退し、YC S16に応募
- 当初は「医師向けチャットボット」を構想していたが、チャットボットには大量のラベル付きデータが必要と気づく
- ビッグテック(Google、Facebook等)は社内に巨大なデータラベリングチームを持つが、スタートアップや中堅企業にはそのリソースがない

**需要検証方法**:
- YCバッチ中に「医師向けチャットボット」のアイデアでスタートしたが、顧客ヒアリングで課題の本質に気づく
- 「チャットボット自体」ではなく「チャットボットを作るためのデータ」が真の課題
- 自動運転業界を中心に、AIモデル開発企業がデータラベリングに苦戦していることを発見
- CVPR (Computer Vision and Pattern Recognition) 会議に参加し、ブース営業を実施
  - ラップトップを持ってブースを回り、デモを見せながら直接営業
  - この「泥臭いアプローチ」が初期顧客獲得につながった

### 2.2 CPF検証(Customer Problem Fit)

**インタビュー/顧客検証**:
- 実施数: 約50社以上(自動運転企業、ロボティクス企業、AR/VR企業)
- 手法: CVPR会議でのブース営業、直接訪問、プロトタイプデモ
- 発見した課題の共通点:
  - データラベリングは手作業で時間とコストがかかる(80%の企業が課題と認識)
  - 品質のばらつきが大きく、モデル性能に直結
  - 社内リソースでは規模拡大が困難
  - 既存のクラウドソーシング(Amazon MTurk等)では品質・スピードが不足

**3U検証**:
- Unworkable(現状では解決不可能):
  - Amazon MTurkなどの既存ツールでは精度が低い(70-80%)
  - 社内チームでは規模拡大が不可能(数百万枚の画像をラベリング)
  - 外注では品質管理が困難

- Unavoidable(避けられない):
  - AIモデルは学習データの品質で性能が決まる(ガベージイン・ガベージアウト)
  - 自動運転の安全性はラベリング精度に直結(1%のミスが重大事故につながる)
  - 画像認識、NLP、音声認識など、すべてのAI領域でデータラベリングが必要

- Urgent(緊急性が高い):
  - 自動運転業界は2020年までの商用化を目指し、データラベリングがボトルネックに
  - 競合他社との開発スピード競争が激化
  - データ不足がモデル開発の遅延に直結

**支払い意思(WTP)**:
- 確認方法: プロトタイプデモ後、即座にパイロットプロジェクトを提案
- 結果: GM Cruise、Zoox、Uber、Alphabet等が初期顧客として契約
  - 40%の企業がデモ後すぐに有料パイロットに移行(業界平均10-15%を大幅に上回る)
  - 従来のMTurkと比較して10倍のコストでも「安い」と評価される(品質・スピードを考慮)

### 2.3 PSF検証(Problem Solution Fit)

**10倍優位性**:

| 軸 | 従来の解決策 | Scale AIソリューション | 倍率 |
|---|------------|-----------------|------|
| スピード | MTurk: 1週間/1万枚 | Scale: 数時間/1万枚 | 20x |
| コスト | 社内チーム: $50/時間 | Scale: $5/時間(外注) | 10x |
| 品質(精度) | MTurk: 70-80% | Scale: 99%+ | 5x |
| スケーラビリティ | 社内: 100枚/日限界 | Scale: 100万枚/日可能 | 100x |
| 導入障壁 | 外注: 契約・管理が複雑 | Scale: API 1行で統合 | 10x |

**MVP**:
- タイプ: Wizard of Oz (初期は裏側で手動作業)
- 初期アーキテクチャ:
  - フロントエンド: シンプルなAPIエンドポイント("画像を送るとラベル付きで返ってくる")
  - バックエンド: 初期は手動でクラウドワーカーに依頼し、品質チェック後に返却
  - 徐々にAI支援ツールを導入し、半自動化→自動化を進める
- 初期反応:
  - 自動運転企業から「これを待っていた」という反応
  - パイロットプロジェクトが次々と本格契約に移行
- CVR: 約40%(CVPR会議での商談から契約への転換率)

**UVP(独自の価値提案)**:
「API × ヒューマン・イン・ザ・ループ × AI支援 = 99%精度 × 無限スケール × 低コスト」

Scale AIは、以下の3つを同時に実現する唯一のソリューション:
1. **API-First**: 開発者が1行のコードで統合できるシンプルさ
2. **ヒューマン・イン・ザ・ループ**: AIだけでなく、人間の判断を組み合わせて99%精度を実現
3. **無限スケール**: グローバルなワーカーネットワーク(Remotasks, Outlier)で数百万件/日の処理能力

**競合との差別化**:
- **vs. Amazon MTurk(クラウドソーシング)**:
  - 精度: 70-80% → 99%+(品質管理プロセス)
  - スピード: 数日 → 数時間(専門ワーカーチーム)
  - API統合: 複雑 → 1行のコード

- **vs. 社内データチーム**:
  - コスト: $50/時間 → $5/時間(外部ワーカー活用)
  - スケール: 100枚/日 → 100万枚/日(グローバルネットワーク)
  - 専門性: 汎用 → 業界特化(自動運転、医療画像等)

- **vs. Labelbox, Appen(競合データラベリング)**:
  - エンドツーエンド: ツール提供のみ → データラベリング代行まで一気通貫
  - AI支援: 手動中心 → AI支援で効率化
  - プラットフォーム: 単一用途 → マルチモーダル(画像、テキスト、音声、動画)

## 3. ピボット/失敗経験

### 3.1 初期の失敗

- **YCバッチ中の迷走**:
  - Alexandrは「YCバッチの中盤で非常に迷っていた(very lost mid-batch)」と告白
  - 最初のアイデア「医師向けチャットボット」はトラクションが得られなかった
  - YCパートナーからのフィードバックで「本質的な課題」を探すよう促された

- **情報不足とリサーチ不足**:
  - 医療業界への知見が不足しており、規制や承認プロセスの複雑さを理解していなかった
  - チャットボット市場は既に競合が多く、差別化が困難だった

### 3.2 ピボット(該当する場合)

**ピボット詳細**:

- **元のアイデア**: 医師向けチャットボット
  - 仮説: 医師は患者対応に時間を取られており、チャットボットで効率化できる
  - ターゲット市場: 医療業界(病院、クリニック)
  - ビジネスモデル: SaaS(月額課金)

- **ピボット後**: 自動運転向けデータラベリングAPI
  - 仮説: AIの進化はアルゴリズムではなく、データ品質で決まる
  - ターゲット市場: 自動運転企業、ロボティクス、AR/VR
  - ビジネスモデル: 従量課金(API使用量ベース)

- **きっかけ**:
  - チャットボット開発中に「大量のラベル付きデータが必要」と気づく
  - 既存のデータラベリングツールが不十分であることを発見
  - 自動運転業界がデータラベリングに苦戦しているというインサイトを得る

- **学び**:
  - 「顧客が欲しがるもの」ではなく「顧客が本当に必要とするもの」を見つける
  - インフラ層(データラベリング)は、アプリケーション層(チャットボット)よりも大きな市場
  - YCの教え「Talk to users」を徹底的に実践

- **ピボットの実行スピード**:
  - YCバッチ中盤(約1ヶ月)で意思決定
  - 2週間で新しい仮説を検証
  - 4週間でMVPを開発し、CVPR会議でデモ
  - ピボット後わずか2ヶ月で初期顧客(GM Cruise、Zoox)を獲得

## 6. 成功要因分析

### 6.1 主要成功要因

1. **タイミング: AI革命の黎明期**:
   - 2016年はディープラーニングが実用化され始めた時期
   - 自動運転ブーム(2016-2020)とタイミングが完全一致
   - データが「新しい石油」と認識され始めた時期

2. **ピボットの決断力**:
   - YCバッチ中盤で「医師向けチャットボット」から「データラベリング」へピボット
   - わずか1ヶ月で意思決定し、2ヶ月で初期顧客獲得
   - 「顧客が本当に必要とするもの」を見つける洞察力

3. **インフラ層への着目**:
   - アプリケーション層(チャットボット)ではなく、インフラ層(データ)に着目
   - Peter Thielの「垂直統合」戦略を実践(ツール提供ではなくデータ提供)

4. **品質へのこだわり**:
   - 99%精度という業界最高水準を維持
   - ヒューマン・イン・ザ・ループで品質を担保
   - 自動運転など「命に関わる」領域での信頼獲得

5. **スケーラビリティの設計**:
   - API-Firstで開発者が簡単に統合できる設計
   - グローバルワーカーネットワークで無限スケール
   - AI支援ツールで効率化しながらコスト削減

6. **戦略的市場拡大**:
   - 自動運転 → ロボティクス → LLM/RLHF → 国防
   - 各フェーズで業界No.1ポジションを確立してから次へ拡大

### 6.2 タイミング要因

- **自動運転ブーム(2016-2020)**:
  - GM、Uber、Alphabet、Teslaなどが自動運転に巨額投資
  - データラベリングがボトルネックに

- **ディープラーニング革命(2016-)**:
  - ImageNet、AlphaGoなどでディープラーニングが実証
  - 「データ量がモデル性能を決める」という認識の広がり

- **LLMブーム(2020-)**:
  - GPT-3(2020)、ChatGPT(2022)でLLMが実用化
  - RLHF(人間のフィードバックによる強化学習)の需要急増
  - Scale AIはOpenAI、Anthropicの主要データプロバイダーに

- **AI国家戦略(2022-)**:
  - 米中AI覇権競争の激化
  - U.S. DoDがAIを国防の中核に位置づけ
  - Scale AIは国防総省の主要AIパートナーに

### 6.3 差別化要因

- **99%精度**: 業界最高水準の品質管理プロセス
- **無限スケール**: グローバルワーカーネットワーク(数万人)
- **API-First**: 1行のコードで統合可能
- **マルチモーダル**: 画像、テキスト、音声、動画すべてに対応
- **ドメインエクスパティーズ**: 自動運転、医療画像、LLMなど業界特化知見
- **戦略的パートナーシップ**: OpenAI、Meta、U.S. DoDとの深い連携

## 8. orchestrate-phase1への示唆

### 8.1 需要発見(/discover-demand)

- **ピボットの柔軟性**:
  - Alexandrは「医師向けチャットボット」から「データラベリング」へ素早くピボット
  - 表面的なニーズではなく、根本的な課題を見つける洞察力
  - YCの「Talk to users」を徹底実践

- **CVPR会議での泥臭い営業**:
  - ラップトップを持ってブースを回る「非スケーラブル」なアプローチ
  - しかし、これが初期顧客獲得とドメイン理解の鍵に

- **インフラ層への着目**:
  - アプリケーション層ではなく、インフラ層の課題を見つける
  - 「誰もが必要とするが、誰も解決していない」問題を探す

### 8.2 CPF検証(/validate-cpf)

- **3U検証の徹底**:
  - Unworkable: Amazon MTurkでは精度70-80%で不十分
  - Unavoidable: AIモデルはデータ品質で性能が決まる
  - Urgent: 自動運転の商用化スケジュールがボトルネック

- **高い支払い意思**:
  - 40%の企業がデモ後すぐに有料パイロットに移行
  - 従来の10倍のコストでも「安い」と評価される(ROIが明確)

- **課題の共通性**:
  - 80%の自動運転企業がデータラベリングを課題と認識
  - 業界全体の共通課題を解決することで急成長

### 8.3 PSF検証(/validate-10x)

- **複数軸での10倍優位性**:
  - スピード: 20倍
  - コスト: 10倍
  - 品質: 5倍
  - スケーラビリティ: 100倍
  - 単一軸ではなく、複数軸で圧倒的優位性を確保

- **Wizard of Oz MVP**:
  - 初期はAPIの裏側で手動作業
  - 徐々にAI支援ツールを導入し、自動化を進める
  - 顧客には「API」として提供し、裏側の実装は柔軟に変える

- **UVPの明確化**:
  - 「99%精度 × 無限スケール × 低コスト」という3要素の組み合わせ
  - 既存ソリューション(MTurk、社内チーム)では実現不可能

## 9. 事業アイデア候補

この事例から着想を得られる日本向けビジネスアイデア:

1. **日本語LLM向けRLHFプラットフォーム**:
   - 対象: 日本のLLM開発企業(Stability AI Japan、Sakana AI等)
   - 課題: 日本語の高品質RLHFデータが不足
   - ソリューション: 日本語ネイティブワーカーによるRLHF評価プラットフォーム
   - 差別化: 日本語の文化的ニュアンス、敬語、ビジネスマナーに精通したワーカー

2. **製造業向けAI品質検査データラベリング**:
   - 対象: 日本の製造業(自動車部品、電子機器等)
   - 課題: 画像ベースの品質検査AI導入時、ラベル付きデータ作成が障壁
   - ソリューション: 製造業特化のデータラベリングサービス
   - 差別化: 日本の製造業の品質基準に精通したワーカー、現場との密な連携

3. **医療画像ラベリングプラットフォーム**:
   - 対象: 日本の医療AI開発企業、病院
   - 課題: 医療画像(X線、CT、MRI)のラベリングには医療知識が必要
   - ソリューション: 医療従事者(放射線技師、医師)によるラベリングプラットフォーム
   - 差別化: 日本の医療制度・規制に準拠、PMDA承認サポート
