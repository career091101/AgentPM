## 1. 基本情報

### 1.1 企業プロフィール

Intercomはアイルランド・ダブリンを本拠地とするSaaS企業で、2011年の創業以来、**カスタマーサービスプラットフォーム市場のリーダー**として位置付けられている。現在、世界中の数千社（Fortune 500企業を含む）が同社のプラットフォームを利用し、月間数百万件のカスタマーインタラクションを処理している。

**企業規模**: 従業員1,000名以上、複数の国際オフィス
**主要な資金調達**: シリーズE（2021年）で$250M達成、企業評価額$2.5B

### 1.2 カスタマーサポート業界の背景

Intercomが直面する業界課題：
- **スケーラビリティ**: 企業成長に伴い、サポートコストが二次関数的に増加
- **言語多様性**: グローバル対応でサポートコストが数倍に膨れ上がる
- **24/7対応**: タイムゾーン対応で人員確保が困難
- **一貫性**: 人によるばらつきで顧客満足度が低下

### 1.3 Intercom内のAIイニシアティブ位置付け

Finは単なるチャットボットではなく、**Intercomの戦略的なプロダクト革新の核**である：
1. **自社プラットフォームの差別化**（AI解決率51%が競争優位性）
2. **新規ビジネスモデル**: $0.99/解決済みクエリという従量課金制の導入
3. **AIエージェント市場への参入**: 将来的に「Customer Agent」へ拡大

## 7. 課題と対応

### 7.1 技術的課題

#### 課題1: 複雑なカスタマーサービスクエリへの対応限界

**問題**:
- 注文キャンセル・返金処理：単なるQ&Aではなく、システム操作が必要
- アカウントトラブル復旧：複数の条件分岐と順序的な手順が必要
- 顧客怒り対応：感情的なエスカレーション判定が不十分

**Fin v1での解決率23%の理由**:
```
ChatGPT GPT-4の能力限界:
├─ 外部システムAPI呼び出しが弱い（ハルシネーション発生）
├─ 複雑な条件分岐の判定が曖昧
├─ 多言語での文化的ニュアンス判定が低精度
└─ リアルタイムデータアクセスが限定的
```

**対応（Fin v2/v3）**:
1. **Claude 3への移行**: 推論能力向上により、複雑な意思決定プロセスを正確に処理
2. **Procedures機能**: ワークフロー定義言語として、自然言語ルール → 実行可能な手順に変換
   ```
   自然言語ルール定義:
   "If customer order <7 days old AND not shipped AND refund eligible,
    then auto-issue full refund AND send confirmation email"

   → Procedures Engine で実行可能なワークフローに変換
   ```

#### 課題2: スケーラビリティ・レイテンシ

**問題**:
- 月間数百万会話（ピーク時に秒あたり数百リクエスト）を低レイテンシで処理
- 45言語の並列処理時の応答時間管理
- コスト効率の維持

**対応**:
1. **Prompt Caching**: 同じKnowledge Documentを繰り返し参照する場合、キャッシュして60-90%のトークン削減
   ```
   1回目: "FAQドキュメント読み込み（5,000トークン）" + "クエリ処理（100トークン）" = 5,100トークン
   2回目: "キャッシュされたFAQ（0トークン）" + "クエリ処理（100トークン）" = 100トークン
   削減率: 98%（複数会話で累積効果）
   ```

2. **ハイブリッド推論戦略**:
   - **定型クエリ** → Claude 3 Sonnet（軽量、高速、低コスト）
   - **複雑なクエリ** → Claude 3 Opus（高精度、高推論能力）

#### 課題3: 精度・信頼性

**問題**:
- 99.9%精度 = 1,000会話中1件の不正確（大規模ではリスク）
- 金銭取引を扱うため、誤った返金処理は重大事故
- 多言語での文化的誤解による顧客怒り

**対応**:
1. **Simulations機能**: デプロイ前にシナリオテストを実施
   ```
   テストシナリオ:
   ├─ Happy Path（期待通りの流れ）
   ├─ Edge Cases（境界条件）
   ├─ Error Cases（エラーシナリオ）
   └─ Cultural Sensitivity（言語・文化別テスト）
   ```

2. **Human-in-the-Loop フォールバック**: 不確実性が高い場合は自動的に人間へエスカレーション

### 7.2 組織・採用面での課題

#### 課題4: サポート代理人の職業変化への対応

**問題**:
- 従来: 定型問い合わせ処理（時給14-18ドル）
- AI導入後: 複雑案件・顧客関係構築に専念（スキル要件大幅アップ）
- リスク: 低スキル層の失業、職業訓練ニーズ

**Intercomの対応**（業界リーダーとしての責任）:
1. **顧客企業向けガイダンス**: サポート代理人の再訓練プログラムを推奨
2. **キャリアパス明示化**: 「AI + 人間」の新しい職業像を定義
   ```
   旧職: 定型案件処理 → 評価基準: 処理件数
   新職: 複雑案件 + 顧客関係 → 評価基準: 顧客満足度、リテンション、UGC
   ```

## 附録: Intercom Fin の技術用語解説

### A.1 主要な技術フレームワーク

**Claude 3**: Anthropic が開発した最新世代LLM。推論能力、長文コンテキスト対応、多言語性能が特徴。Enterprise Plan では SLA、カスタムレート制限、優先サポートを提供。

**Fin AI Engine™**: Intercom の特許取得済みアーキテクチャ。Claude 3 をベースに、Knowledge Hub、行動制御（トーン、応答長）、外部データアクセス、分析機能を統合。

**Knowledge Hub**: 複数の情報源（FAQ, SOP, 顧客データベース、外部API）を一元管理し、意味検索（Semantic Search）により関連ドキュメントを自動抽出。

**AI Category Detection**: クエリを自動分類（キャンセル、返金、バグ報告、顧客フラストレーション等）し、適切なルーティング・優先度付けを実施。

**Procedures**: 自然言語で複雑なワークフロー（条件分岐、外部システム連携、アクション実行）を定義し、AI Agent が実行。Zapier 等の自動化ツール発展版のイメージ。

**Prompt Caching**: 同じドキュメントを繰り返し処理する際、キャッシュを活用してトークン消費を削減。企業のFAQドキュメント（数千ページ）をキャッシュすることで、後続会話のコストを60-90%削減。
