# GUGA Case Study 034: 流通企業O - 発注予測AI導入による在庫最適化

**Case ID**: GUGA_0034
**企業**: 流通企業O（仮称）
**業界**: 小売・流通
**AIツール**: AI予測分析（在庫管理・発注予測システム）
**公表日**: 2025-12-01
**導入期間**: 2024年4月～2025年11月（20ヶ月）

---

## 1. エグゼクティブサマリー

### ビジネスインパクト（定量）

流通企業Oは、過去3年間の売上データを学習した発注予測AIを導入し、以下の成果を実現：

- **在庫保管費用**: 20%削減（年間3,500万円の削減効果）
- **品切防止率**: 98.2% → 99.7%に向上（1.5ポイント改善）
- **過剰在庫削減**: 前年比15%削減
- **発注精度**: RMSE 12.3% → 8.7%に改善
- **ROI**: 初期投資回収期間 14ヶ月

### 戦略的意義

本事例は、**従来の統計予測モデルからディープラーニングベースのAIへの移行**を示す典型的なケースであり、以下の点で重要：

1. **小売流通業での実装成功**: 季節変動・プロモーション・外部要因が複雑な業界でのAI活用
2. **レガシーシステム統合**: 既存ERP/POS システムとの継ぎ目ない統合アーキテクチャ
3. **人間-AI協調**: 完全自動化ではなく、人間の判断を組み込んだハイブリッド意思決定
4. **段階的展開モデル**: パイロット（単一店舗）→ 複数拠点 → 全社展開

## 2. 背景と課題

### 企業プロフィール

**流通企業O**：
- 売上規模: 350億円（2024年度）
- 店舗数: 280店舗（全国）
- 従業員数: 3,200名
- 主要事業: 食品・日用雑貨・衣料品等の総合小売
- 営業形態: スーパーマーケット、ドラッグストア、ディスカウントストア

### 事業上の課題

#### 課題1: 在庫管理の非効率性

**現状**:
- 従来のExcelベースの発注方式により、以下の問題が発生
  - 定量発注法（ROP: Reorder Point）が手作業で計算
  - 店舗ごと・季節ごとの需要変動に対応できない
  - パートタイム店員の属人的判断による発注ばらつき

**影響**:
- 年間を通じた過剰在庫: 約2.2億円相当
- 品切れによる売上機会損失: 推定年間1.8億円
- 在庫保管費用（冷蔵機器・スペース賃借）: 年間1.75億円

#### 課題2: 季節変動への対応不足

**具体例**:
- **夏季**: 冷たい飲料・アイスクリームの需要急増 → 予測困難で品切れ多発
- **冬季**: 鍋もの用食材の需要変動 → 過剰在庫が廃棄
- **GW・盆・正月**: イベント連動の需要予測失敗

**定量的課題**:
- 需要予測の平均誤差率: 25～35%
- イベント期間の品切れ率: 12～18%

#### 課題3: スタッフスキルの属人性

**問題**:
- ベテラン発注担当者の退職による知見喪失
- 新人スタッフの育成期間が6ヶ月～1年と長期
- 発注判断に個人差が最大40%に達する

#### 課題4: システムの老朽化

**既存システム**:
- 導入年: 2015年
- 統計手法: 単純移動平均、指数平滑化法のみ
- スケーラビリティ: 新店舗追加時の対応に3ヶ月要する
- API連携: POS・WMS との遅延同期（日次）

### ビジネス課題の深掘り

#### 在庫保有コストの構成（年間1.75億円）
- 冷蔵・冷凍機器の電気代: 7,500万円（43%）
- 保管スペース賃借: 5,200万円（30%）
- 廃棄・ロス費用: 2,300万円（13%）
- 在庫管理システム保守費: 1,000万円（6%）
- その他（保険・手数料等）: 1,800万円（8%）

#### 過剰在庫のビジネスインパクト
- **フロー観点**: 資金繰りの悪化（流動資産効率 12.3% → 目標14%）
- **P&L観点**: 廃棄ロス年間2.3億円（売上原価の0.66%）
- **競争力観点**: 小売周辺技術のスピード劣化（Amazon・楽天との差）

## 3. AI導入前のアプローチと限界

### 従来の発注システム（Baseline）

#### 方法論：定量発注法（ROP）
```
ROP = (平均日間需要 × リードタイム日数) + 安全在庫
安全在庫 = 安全係数 × 標準偏差
```

**パラメータ設定**:
- 安全係数: 1.5～2.0（手作業で調整）
- 計算周期: 月次見直し（自動化なし）
- 適用対象: 全SKU（約8,000点）

#### 実運用の問題
1. **パラメータの過期化**
   - 季節因子の組み込み: 手作業で4月・7月・10月・12月のみ手動調整
   - 新商品対応: 3ヶ月の販売データ蓄積を待つ → その間は手動発注

2. **外部要因への非対応**
   - 天候（降雨・台風）: 対応なし
   - プロモーション連動: 「予定表にマーク」で対応（自動反映なし）
   - 競合他社キャンペーン: 事後対応のみ

3. **店舗ごとの柔軟性欠如**
   - 同一商品を全店舗で同じパラメータで発注
   - 大型店舗と小型店舗の需要差に未対応
   - 地域特性（北海道の乳製品需要など）に未対応

### 業界の標準的アプローチとの比較

#### 競合他社の先行事例（参考データ）
- **大手スーパーA**: 統計的需要予測API（Demand.ai）を2022年導入 → 発注精度12%改善
- **CVSチェーンB**: 機械学習ベースの発注システム（自社開発）→ 過剰在庫8%削減

**流通企業Oとの差異**:
- 資本規模が小さく、自社開発リソース不足
- 既存システム（ERP）の刷新コストが高い
- 段階的導入ニーズが高い

### 限界を突破する必要性

#### KPI目標設定（AI導入の方向性）
- **短期目標（12ヶ月）**: 在庫保管費用15%削減
- **中期目標（24ヶ月）**: 在庫保管費用20%削減、品切防止率98%以上
- **長期目標（36ヶ月）**: 在庫回転率の業界平均化（25回転 → 28回転）

## 5. AI技術の詳細・アーキテクチャ

### 採用したAI技術スタック

```
┌─────────────────────────────────────────────┐
│         Demand Forecasting Pipeline         │
├─────────────────────────────────────────────┤
│                                             │
│  Data Layer                                 │
│  ├─ POS Data (280店舗)                     │
│  ├─ WMS Inventory                          │
│  ├─ Weather API (気象庁)                    │
│  └─ Campaign Calendar (プロモーション)      │
│                                             │
│  Feature Engineering Layer (自動)          │
│  ├─ Trend: 過去90日移動平均                 │
│  ├─ Seasonality: 年次周期分解              │
│  ├─ Promo Effect: イベント係数              │
│  ├─ External Factor: 天候スコア             │
│  └─ Store Profile: 店舗特性ベクトル        │
│                                             │
│  ML Model Layer                             │
│  ├─ LSTM (Long Short-Term Memory)          │
│  ├─ Gradient Boosting (XGBoost)            │
│  └─ Ensemble Method (加重平均)              │
│                                             │
│  Output Layer                               │
│  ├─ 14日先予測（日次）                      │
│  ├─ 信頼度スコア（95%信頼区間）             │
│  └─ 異常検知フラグ                         │
│                                             │
└─────────────────────────────────────────────┘
```

### 主要技術コンポーネント

#### 1. LSTM（ディープラーニング）

**役割**: 時系列の長期依存性を学習

**パラメータ**:
- Layer数: 3層（隠れ層ユニット: 128 → 64 → 32）
- Dropout率: 0.2（過学習防止）
- Activation関数: ReLU（中間層）、Linear（出力層）
- Optimizer: Adam（学習率 0.001）

**学習データ**:
- 学習期間: 2021年1月～2023年12月（3年間）
- SKU数: 8,000品目
- 店舗数: 280店舗
- データポイント総数: 約6.5億件

**推論速度**: CPU処理で全SKU・全店舗の予測を5分以内に完了

#### 2. Gradient Boosting（XGBoost）

**役割**: カテゴリ変数・複雑な特徴交互作用をキャプチャ

**特徴エンジニアリング** (手動 + 自動):
```
入力特徴（計47個）:
├─ 時系列特徴
│  ├─ 過去7日平均売上
│  ├─ 過去30日平均売上
│  ├─ 過去1年同週売上
│  └─ トレンド係数（90日傾斜）
│
├─ 外部要因特徴
│  ├─ 天候スコア（降雨量・気温）
│  ├─ プロモーション有無フラグ
│  ├─ 競合セール情報
│  └─ 祝日・イベント
│
├─ 店舗特性特徴
│  ├─ 店舗売上規模（百分位）
│  ├─ 地域ジャンル（北海道・関東など）
│  ├─ 店舗タイプ（スーパー・ドラッグなど）
│  └─ 顧客層スコア
│
└─ 商品特性特徴
   ├─ SKU売上規模
   ├─ カテゴリ
   ├─ 季節係数
   └─ 賞味期限影響度
```

**ハイパーパラメータ**:
- Tree数: 300
- Max Depth: 8
- Learning Rate: 0.05
- Subsample比: 0.8

#### 3. Ensemble Method（予測統合）

**アルゴリズム**:
```
最終予測 = w1 × LSTM予測 + w2 × XGBoost予測 + w3 × 従来モデル予測

重み付け:
- w1 (LSTM): 0.50 (中期・季節性に強い)
- w2 (XGBoost): 0.40 (短期・複雑パターンに強い)
- w3 (従来法): 0.10 (外れ値抑制)
```

**重み最適化**: 過去1年の検証データを用いて、誤差最小化（最小二乗法）

### 学習・推論パイプラインの流れ

```
[日次実行スケジュール]

00:00  → POS/WMS データ取得 (自動)
00:30  → 前日売上データの確定・検証
01:00  → 特徴エンジニアリング (並列処理, 280店舗分)
02:30  → LSTM推論実行
03:00  → XGBoost推論実行
03:30  → Ensemble統合・信頼度スコア計算
04:00  → 異常値検知 (閾値外フラグ)
04:30  → 推奨発注数の計算・出力
05:00  → 発注システムへ自動反映
05:30  → レポート生成 (朝礼用)
06:00  → 店舗スタッフが確認・承認
```

**レイテンシ要件**: 完了まで6時間以内（朝6時までに全予測完了）

### データ品質管理

#### データクレンジング
- **外れ値除去**: IQR法（Q1-1.5×IQR, Q3+1.5×IQR）
- **欠損値補完**: 線形補間（短期）/ 季節調整後の平均値（長期）
- **異常値検知**: Z-scoreで|Z| > 3の値を1日外れ値とマーク

#### 特徴スケーリング
- **手法**: Robust Scaler（中央値・四分位範囲で正規化）
- **理由**: 外れ値の影響を低減、勾配降下法の収束安定化

## 6. 実装・運用プロセス

### フェーズ別導入スケジュール

#### Phase 1: パイロット (2024年4月～6月, 3ヶ月)

**対象**: 単一店舗（大型スーパー1店）、3カテゴリ（生鮮・冷蔵・ドライ）

**実施内容**:
1. データ接続の検証（POS → AI システム）
2. 予測精度の検証（従来法との比較）
3. 運用フロー確立（店舗スタッフによる確認・調整）

**結果**:
- 予測精度: RMSE 18.2% → 11.5% (37%改善)
- 過剰在庫削減: 8%
- システム安定性: ✅ OK

**課題抽出**:
- 生鮮食品の短期変動への対応不足
- 夜間配送データとのズレ

#### Phase 2: 複数拠点展開 (2024年7月～8月, 2ヶ月)

**対象**: パイロット店舗+地域代表店舗 8店（異なる売上規模・地域）

**実施内容**:
1. モデル再学習（新データ追加、パラメータ調整）
2. 店舗スタッフ研修（使用方法・判断ルール）
3. 運用体制確立（発注担当者の責任範囲）

**結果**:
- 平均精度: RMSE 12.8%
- スタッフ受容度: 78% (満足度調査)
- 技術サポート負荷: 週15件 → 週8件に低下

**課題**:
- 小型店舗では予測精度が76%（大型店85%）
- プロモーション連動の遅延（カレンダー入力の手作業）

#### Phase 3: 全社展開 (2024年9月～11月, 3ヶ月)

**対象**: 全280店舗

**実施内容**:
1. 一括データマイグレーション
2. 全スタッフ研修（階層別）
3. 発注システム統合（自動反映機能オン）

**実行方法**:
- Week 1-2: 系統1～3 (100店舗) のロールアウト
- Week 3-4: 系統4～6 (90店舗) のロールアウト
- Week 5-6: 系統7～10 (90店舗) のロールアウト
- リバート可能な段階的展開

**結果**:
- ✅ 全店舗の予測配信達成
- ✅ システムエラー: 0件
- ✅ データ連携遅延: 平均2分以内

#### Phase 4: 最適化・継続改善 (2024年12月～2025年11月, 12ヶ月)

**実施内容**:
1. **Q1 (12月～2月)**: 冬季需要パターンの検証・微調整
2. **Q2 (3月～5月)**: 新商品カテゴリの追加学習
3. **Q3 (6月～8月)**: 夏季需要シミュレーション・プロモーション連動強化
4. **Q4 (9月～11月)**: 年間パフォーマンス分析・来年度へのフィードバック

## 9. 技術的課題と解決策

### 実装時に直面した主要課題

#### 課題1: データ品質の問題（Phase 1）

**事象**:
- POS システム の記録ミス（レジ・スキャン誤り）
- WMS の遅延同期（日次バッチ、1日遅延）
- 店舗スタッフの手修正データ（信頼性不明）

**影響**:
- 初期学習時、予測精度が18.2% に低下（目標10%）

**解決策**:
1. **POS データの検証ロジック追加**
   - スキャンなし販売（現金支払いの一部）を補完
   - 返品・値引きデータの分離・正規化

2. **WMS データの実装修正**
   - API統合で準リアルタイム（3時間遅延）に改善
   - 欠損データの線形補間ロジック

3. **信頼度スコアの導入**
   - 各ポイントにメタデータ（修正有無フラグ）を付与
   - 学習時に信頼度ウェイトを反映

**結果**: 精度が11.5% に改善

#### 課題2: 季節性・周期性の複雑さ（Phase 2）

**事象**:
- 単純な年次周期だけでなく、多層的な周期が存在
  - 週次周期（日曜に家族購買ピーク）
  - 月次周期（給与日前後の購買量変動）
  - 年次周期（クリスマス・正月など）
- これらが非線形に相互作用

**影響**:
- XGBoost単体では説明力60%（複雑な交互作用に未対応）

**解決策**:
1. **特徴エンジニアリングの強化**
   - Fourier特徴（周期成分を明示的に組み込み）
   - 複数周期の季節調整（STL分解など）

2. **LSTM の層数・ユニット数調整**
   - 初期: 1層 64ユニット → 最終: 3層 (128→64→32)
   - Dropout を 0.2 に設定（過学習抑制）

3. **Ensemble 重み付けの動的化**
   - 固定重み（0.5, 0.4, 0.1）から、
   - 「短期・中期・長期」別の動的重み付けに変更
   - 予測対象日までの距離に応じて重み調整

**結果**: 複合周期の説明力 85% に改善

#### 課題3: プロモーション・イベント連動（Phase 2-3）

**事象**:
- プロモーション実施時の売上は平時の 2～4倍に急増
- プロモーション情報がAIシステムに手作業で入力されているため、以下の問題
  - 入力遅延（企画立案 → システム入力に2週間）
  - 入力漏れ（地域別・店舗別キャンペーン見落とし）
  - 効果予測の不正確さ

**影響**:
- プロモーション期間の予測誤差が 22% に悪化

**解決策**:
1. **プロモーションカレンダーの自動取得**
   - 社内キャンペーン管理システムとAPI連携
   - リアルタイムに最新情報を AI に反映

2. **プロモーション効果の定量化モデル**
   - 過去1年のプロモーション実績データから
   - 「チラシ掲載 → +150%」「店頭展示 → +80%」など
   - 効果係数を自動抽出し、予測に組み込み

3. **外部データ（競合セール情報）の統合**
   - 競合チラシ情報を RPA で自動収集
   - 競合セール時に「買い替え需要」を考慮した予測調整

**結果**: プロモーション期間の誤差が 8% に改善

#### 課題4: リアルタイム異常対応（運用段階）

**事象**:
- 天災・外出自粛命令などの突発的外部要因
- 2024年冬のインフルエンザ流行 → マスク・風邪薬の需要爆増
- AI モデルが予測できない新しいパターン

**影響**:
- 品切れ・過剰在庫の多発（一時的にKPI悪化）

**解決策**:
1. **異常検知アルゴリズムの導入**
   - Isolation Forest を用いて「異常な需要パターン」を自動検知
   - 閾値超過時はスタッフに通知・手動判断

2. **フィードバックループの構築**
   - スタッフが実際の売上を「確認」する際に
   - 「予測との乖離理由」をコメント入力
   - その情報が自動的に「異常パターン」として記録

3. **外部データソースの拡充**
   - ニュース API で「感染症流行」「天災」の自動検出
   - Google Trends で「検索ボリューム変化」を検出
   - これらを新データとして学習に反映

**結果**: 異常発生時の対応時間を 24時間 → 2時間に短縮

### 技術債務と今後の改善課題

| 項目 | 現状 | 改善予定 | 優先度 |
|------|------|---------|--------|
| **リアルタイム学習** | 日次バッチ | 時間単位学習への移行 | 中 |
| **マルチモーダル学習** | テキスト/数値のみ | 画像データ（店舗POPなど）の追加 | 低 |

   - 在庫保管費用 20% 削減（目標達成）
   - 品切防止率 98.2% → 99.7%（目標超過）
   - ROI 153%（初期見積もりの 108%超過）

2. **技術的な実装レベルの高さ**
   - LSTM + XGBoost Ensemble による高精度
   - 複雑な季節性・外部要因の適切な組み込み
   - 異常検知・リアルタイム対応の実装

3. **組織への波及効果**
   - スタッフスキルの高度化（定型 → 例外対応へのシフト）
   - 経営層のデータドリブン思考の浸透
   - 他部門への DX 投資への触発

4. **業界への示唆の大きさ**
   - 中堅企業でも導入可能なモデルを示証
   - 技術選択・段階的展開の成功パターン確立

### 今後の展開予定（2026年以降）

#### Short-term（6ヶ月以内）
1. **説明可能性の強化**
   - SHAP値による予測根拠の詳細化
   - スタッフ向けダッシュボードの改善

2. **動的プライシング機能の追加**
   - 在庫 + 需要予測 + 競合価格から最適価格を提案
   - パイロット: 5～10カテゴリで実験

#### Mid-term（6～18ヶ月）
1. **サプライチェーン上流への拡張**
   - 仕入先への発注提案
   - 配送計画の最適化

   - POSデータ + 会員DB から顧客セグメント別需要予測
   - 個別顧客への推奨商品提案

#### Long-term（18ヶ月以上）
1. **完全自動運用への移行**
   - 現在: AI提案 + 人間承認
   - 目標: 異常フラグ以外は完全自動発注

2. **業界標準化への寄与**
   - 本モデルのオープン化・提供
   - 中堅企業向けテンプレート化

### AI活用成熟度のステップ

```
ステップ1（現在）: 予測
  └─ AI が需要を予測、人間が最終判断

ステップ2（6ヶ月以内）: 推奨
  └─ AI が予測 + 「この価格で売り切れます」と提案

ステップ3（12～18ヶ月）: 最適化
  └─ AI が「在庫・売上・利益」を統合最適化

ステップ4（24ヶ月以上）: 自律
  └─ AI が完全自動意思決定（例外のみ人間介入）
```

### 最終結論

**流通企業O の AI 発注予測システム導入は、以下の点で成功事例として高く評価される：**

1. **技術的難題（季節性・外部要因）の適切な解決**
   - LSTM + XGBoost Ensemble により、従来予測 25% → 8.7% の誤差まで改善

2. **組織導入の成功パターン確立**
   - パイロット → 複数拠点 → 全社展開の段階的アプローチが奏功
   - スタッフの変更管理も適切に実施、99% の稼働達成

3. **ビジネスインパクトの確実性**
   - 初期投資 9,100万円に対して初年度 ROI 153%
   - 在庫保管費用 20% 削減、品切防止率向上など定量効果確実

4. **今後の拡張性の高さ**
   - 他部門（営業・HR）への応用検討開始
   - 次世代機能（動的プライシング・サプライチェーン最適化）への基盤確立
