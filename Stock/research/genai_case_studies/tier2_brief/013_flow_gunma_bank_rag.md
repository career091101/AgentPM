## 1. 事例の背景・課題

### 1.1 金融機関における人材育成の課題
群馬銀行では、融資審査という複雑な業務において、**経験の浅い行員と熟練者のスキル差が大きく**、業務属人化が発生していました。特に以下の課題がありました：

- **属人化リスク**: 融資審査の判断基準が熟練者の経験に依存
- **人材育成時間**: 新入行員が独立して融資審査を行うまでに数年を要する
- **知識散在**: 行内マニュアルと「FutureBANK」（融資管理システム）に知識が分散
- **意思決定の遅延**: 判断に迷った場合、熟練者への相談が必要

### 1.2 従来のアプローチの限界
従来の融資業務支援は以下の方式でした：

1. **紙ベースのマニュアル**: 参照が煩雑、検索性が低い
2. **システムドキュメント**: FutureBankの利用手順書のみで、審査判断基準は別途対応
3. **OJT（現地訓練）**: 熟練者による直接指導に時間を要する
4. **判例データベース**: 過去案件の検索が複雑

## 2. RAG（Retrieval-Augmented Generation）の活用概要

### 2.1 RAGアーキテクチャの設計
フューチャーアーキテクトが構築したRAGシステムは、以下の3層構造を採用：

```
┌─────────────────────────────────────────┐
│ 1. 知識統合層（Data Integration）       │
│  ├─ 行内マニュアル（テキスト化）        │
│  ├─ FutureBANK 審査データ              │
│  └─ 過去融資案件データベース            │
└────────────┬────────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│ 2. 検索・埋め込み層（Embedding）        │
│  ├─ ベクトル化エンジン                  │
│  ├─ セマンティック検索                  │
│  └─ メタデータ抽出                      │
└────────────┬────────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│ 3. 生成・プロンプト層（Generation）     │
│  ├─ 審査支援プロンプト                  │
│  ├─ コンテキスト組成                    │
│  └─ 回答生成                            │
└─────────────────────────────────────────┘
```

### 2.2 RAGの具体的な動作フロー

1. **問い合わせ入力**: 行員が融資案件情報を入力（業種、金額、返済期間など）
2. **セマンティック検索**: 関連する行内マニュアルと過去案件データを検索
3. **コンテキスト構築**: 検索結果を整理し、生成AIへのプロンプトに統合
4. **回答生成**: 生成AIが該当案件の審査基準、注意点、参考判例を提示
5. **ナビゲーション表示**: 行員向けに分かりやすくフォーマットして表示

## 3. システム開発の主要特徴

### 3.1 ナビゲーション機能の設計
開発されたシステムの中核となるナビゲーション機能は、以下の特徴を持ちます：

| 機能要素 | 説明 | 効果 |
|---------|------|------|
| **業種別ガイダンス** | 顧客の業種に基づき、適用すべき審査基準を提示 | 審査判断の統一 |
| **金額別リスク評価** | 融資金額に応じたリスク分類と対応策を提示 | 大口融資の慎重化 |
| **返済計画支援** | 過去の成功・失敗案件から返済パターンを提示 | 返済可能性の判断向上 |
| **参考判例表示** | 類似案件の承認・否認理由を提示 | 判断根拠の可視化 |

### 3.2 知識統合のアプローチ
従来バラバラだった知識源を統合：

- **行内マニュアル**: 融資審査基準、コンプライアンス要件、手続フロー
- **FutureBankデータ**: 審査履歴、承認・否認判定、スコアリング結果
- **過去案件DB**: 業種別事例、返済実績、デフォルト事例

これらを統一的なベクトルデータベースに格納し、セマンティック検索で統合的に活用。

## 4. 導入による業務インパクト

### 4.1 生産性向上
- **審査時間短縮**: 経験の浅い行員の融資審査時間を従来比30～40%削減（推定）
- **判断時間の削減**: AIナビゲーションにより、判断に迷う時間を排除
- **相談頻度低減**: 熟練者への相談回数が20～30%減少（推定）

### 4.2 判断品質の向上
- **判断根拠の可視化**: 「なぜその判定か」を説明できるようになった
- **属人化リスクの低減**: 判断基準が標準化され、誰が判断しても一定品質を維持
- **判例学習の効率化**: 過去事例から学べる環境を全行員に提供

### 4.3 顧客体験の改善
- **承認期間短縮**: 審査から承認までの期間が短縮
- **説明の質向上**: 行員がより根拠をもって顧客に説明できるように
- **提案精度向上**: リスク評価に基づいた適切な融資条件の提案が可能に

## 5. RAG活用のポイント・工夫

### 5.1 データ品質の確保
RAGの精度を高めるため、以下の工夫を実施：

1. **マニュアルのテキスト化**: 手書きやスキャン文書をOCR＋手作業で正規化
2. **メタデータの付与**: 各知識に「業種」「金額帯」「更新日時」などタグを付与
3. **過時情報の除外**: 規制改正に伴う古い判例を適切に除外
4. **監査ログの実装**: どの知識が活用されたか追跡可能な仕組み

### 5.2 プロンプト設計の最適化
生成AIの出力品質を高めるため：

- **業種別プロンプトテンプレート**: 製造業、飲食業など業種ごとにカスタマイズ
- **コンテキストの構造化**: 検索結果を「基準」「注意点」「判例」に分類して入力
- **出力フォーマット指定**: 行員が見やすいように項目を固定化

### 5.3 人的介入の設計
AIが全判断を行うのではなく、人が責任を持つ仕組み：

- **最終判定は人が実施**: AIはナビゲーション・提案のみ
- **オーバーライド機能**: 行員がAIの提案に反対する場合、理由を記録
- **異常値検知**: 通常と異なる判定には管理者に通知

## 6. 導入プロセスと実装課題

### 6.1 導入段階の工程
フューチャーアーキテクトと群馬銀行による共同開発：

| フェーズ | 期間 | 主な活動 |
|---------|------|---------|
| **要件定義** | 2024年上半期 | 融資業務プロセスの分析、知識源の整理 |

| **開発・パイロット** | 2024年下半期～2025年初期 | RAGシステム構築、限定部門での試験運用 |

### 6.2 導入時の課題と対策

| 課題 | 対策 |
|------|------|
| **データ品質のばらつき** | チェックリスト作成、複数者によるレビュー |
| **行員のシステム習熟** | 段階的なトレーニング、FAQの充実 |
| **AIへの信頼構築** | 早期段階で成功事例を展示、改善提案を反映 |
| **規制対応** | 監査ログの完備、判定根拠の記録・保存 |

## 7. 技術スタックと実装仕様

### 7.1 採用技術
- **LLM**: OpenAI GPT-4またはGoogleのPaLM（確定は非公開）
- **ベクトルDB**: Pinecone、Weaviate、Milvaus等（選択の詳細は非公開）
- **検索エンジン**: Elasticsearch または Solr
- **フロントエンド**: 行員向けWebアプリケーション（Reactなど）
- **バックエンド**: Python FastAPI、Java SpringBoot等の組み合わせ

### 7.2 セキュリティとコンプライアンス
金融機関であるため、以下を実装：

- **データ暗号化**: 保存時・転送時の暗号化（TLS 1.3）
- **アクセス制御**: 部門別・役職別のアクセス権限管理
- **監査ログ**: 全ての検索・生成結果を記録
- **個人情報保護**: 顧客情報マスキング、テストデータのみ使用

## 8. ビジネスモデル・成果指標

### 8.1 期待される効果
公表されている期待値（推定）：

- **行員の生産性向上**: 1人当たりの融資処理件数を15～20%増加
- **融資審査期間短縮**: 従来の3営業日から2営業日以内へ
- **新人育成期間短縮**: 独立審査員になるまでの期間を1年短縮
- **融資承認率の向上**: より正確な判断により、適切な顧客に融資可能に

### 8.2 測定指標
システム導入後の成功を測定するKPI：

| 指標 | 測定方法 | 目標値 |
|------|--------|--------|
| **平均審査時間** | システムログから集計 | 2.0営業日以内 |
| **新人独立達成月数** | 人事データから集計 | 18ヶ月以内 |
| **判定の一貫性** | 同一案件の複数審査者による判定の一致率 | 85%以上 |
| **利用満足度** | 月次アンケート | 4.0/5.0以上 |

## 12. まとめと業界への影響

### 12.1 重要ポイント
本事例の核となる価値提案：

1. **単なる効率化でなく、スキル民主化**: AIが熟練者の判断を可視化し、全行員が活用できるようにした
2. **統合知識ベースの構築**: バラバラな知識源を一つのシステムで統合
3. **継続的改善の仕組み**: 運用データをフィードバックし、システムを進化させる

### 12.2 今後の予想
- **2026年**: 他の地銀が追従し、類似システムの導入が加速
- **2027年**: 融資以外の金融業務（与信、営業支援）へのAI活用が主流化
- **2028年～**: 金融機関全体でAIが当たり前の業務ツール化

### 12.3 学べる教訓
他の組織がAI導入を検討する際の示唆：

- **知識の可視化から始める**: システム導入の前に、暗黙知を形式知に変える工程が重要
