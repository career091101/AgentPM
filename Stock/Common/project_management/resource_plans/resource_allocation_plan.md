## タスク優先順位ルール

### 日次タスク選定ルール

**毎朝のタスク選定プロセス**（15分）:

1. **Tier 1タスク優先**: 必ず1つ以上のTier 1タスクを含める
2. **緊急度チェック**: 期限が近いタスク（3日以内）を優先
3. **バランス確保**: Tier 1とTier 2のバランス（75:25の比率）
4. **時間配分確認**: 8時間/日を超えないように調整

**タスク選定例（1日8時間）**:
```
✅ Tier 1: 副業 SNS - SNS投稿カレンダー作成（4時間）
✅ Tier 1: Note メディア - ノート戦略策定（3時間）
✅ Tier 2: 資産運用 - 週次分析（1時間）
→ 合計: 8時間
```

## リソース配分の調整ルール

### Tier間リソース再配分

**リソース再配分の条件**:

| 条件 | アクション |
|------|----------|
| **Tier 1プロジェクトが80%達成** | Tier 2に+5時間/週シフト |
| **Tier 1プロジェクトが60%未達** | Tier 2から-5時間/週シフト |
| **Tier 2プロジェクトが90%達成** | Tier 3保留プロジェクトの再開検討 |
| **緊急タスク発生** | 全Tierから-10%ずつリソース確保 |

**再配分プロセス**:
1. 月次レビューで達成率を確認
2. 再配分の必要性を判断
3. 翌月の週次計画に反映
4. `resource_allocation_plan.md` を更新

---

### 緊急タスク対応ルール

**緊急タスクの定義**:
- 期限が3日以内のタスク
- ブロッカーとなるバグ修正
- 外部依存の緊急対応（API障害等）

**緊急タスク対応手順**:
1. 即座に通常タスクを中断
2. 緊急タスク用に4時間/日を確保
3. Tier 1から2時間、Tier 2から2時間をシフト
4. 緊急タスク完了後、通常タスクに復帰

**緊急タスク記録**:
```markdown
## 緊急タスク記録（2026-01-15）

### タスク
- TradingAgentsバグ修正（API接続エラー）

### 優先度
- 最優先（Tier 1以上）

### 所要時間
- 4時間

### 影響
- 副業 SNS: -2時間（SNS投稿カレンダー作成を翌日に延期）
- 資産運用: -2時間（週次分析を週末に延期）
```

## Tier 3プロジェクトの再開条件

### AI未来予測・戦略策定（55点）

**保留理由**:
- 高戦略性だが実行可能性・ROIが低い
- 週40時間のリソースでは優先度が低い

**再開条件**:
1. **Tier 1プロジェクトが安定稼働**
   - 副業 SNS: 月次エンゲージメント率+20%達成
   - Note メディア: 週3本投稿が3ヶ月継続

2. **リソースに余裕ができた場合**
   - 週40時間 → 週50時間に増やせる場合
   - Tier 2プロジェクトの1つが完了した場合

3. **戦略的判断が急務になった場合**
   - 2027年AGI予測が重要な意思決定に必要
   - 資産運用戦略の大幅見直しが必要

**再開時の週次配分**: 5-10時間/週（Tier 2と同等）

---

### システム基盤・インフラ（40点）

**凍結理由**:
- 緊急性なし、他プロジェクト優先
- Phase 2以降が未実行（完成度判定、分類、自動移行）
- 具体的なROIが不明確（効率化の定量評価なし）

**再開条件**:
1. **Flow → Stock移行の手動作業が限界に達した場合**
   - 週5時間以上の手動移行作業が発生
   - ファイル数が500個以上に増加

2. **他の高優先度プロジェクトが完了した場合**
   - Tier 1プロジェクトの1つが完了
   - Tier 2プロジェクトの1つが完了

3. **必要性が明確になるまで延期**
   - 開発効率化の定量評価が可能になった場合
   - 自動化によるROIが明確になった場合

**再開時の週次配分**: 2-3時間/週（最低限のメンテナンス）
